# 全球顶级 AI / Agent 体系差距清单与追赶路线图

**日期：** 2026-03-07  
**版本：** v1.0  
**作者：** 洞察分析师（OpenClaw Analyst）  
**置信度：** 高（基于 2026 年 Q1 最新公开基准数据）

## 执行摘要

本报告从六个核心维度，系统对标全球顶级 AI / Agent 体系（OpenAI ChatGPT Agent / Deep Research / CUA、Anthropic Claude + Computer Use、Google Gemini 3.1 / Antigravity）与我们当前 OpenClaw 多智能体体系之间的能力差距，并给出按优先级排序的追赶行动项。

**核心结论：**
- 我们在「评测体系」和「运行时治理」上差距最大（P0），既缺乏权威量化基准，又缺乏系统性安全分级框架
- 「真实自治能力」有基础但与顶级系统仍有 1–2 个代际差距（P0）
- 「唯一入口」和「言出法随」属于中期差距（P1），架构已有雏形但执行不稳定
- 「闭卷可信度」在当前体系下是结构性弱点（P1），依赖外部搜索兜底

## 二、六维差距逐项对标

### 维度 1：评测体系
**What the best do（顶级玩家在做什么）**

顶级 AI 公司全部建立了多层次评测矩阵，分为：
- **能力基准**：SWE-bench（代码）、HLE（学术推理）、GPQA（科学）、ARC-AGI-2（抽象推理）
- **Agent 专项基准**：APEX-Agents（长程专业任务）、OSWorld（计算机使用）、WebArena（网页操作）、τ-bench（工具使用）
- **内部红队评测**：系统级安全测试，由专门的 Red Team 执行，有公开 System Card
- **持续回归**：每次模型迭代都对照历史分数，形成版本间可追溯的对比

Anthropic 的 RSP（Responsible Scaling Policy）将评测结果直接绑定到部署决策——未通过评测不得上线。

**我们现在的状态**
- 有 E2E dispatch 测试套件（9/12 通过），但没有标准化 Agent 能力评测
- 有 AEO Gate 系统，但判定标准主观，不对标行业量化指标
- 没有任何公认的基准分数，无法对外声称系统处于何种水平
- 缺乏系统性红队测试机制，安全评估依赖 ad hoc 修复

**差距等级：🔴 P0**

**行动项：**
- [ ] A1-1：在 3 个月内建立最小可行评测矩阵（至少覆盖代码任务、工具调用成功率、指令遵循三项）
- [ ] A1-2：挑选至少 1 个行业公认 Agent 基准（如 τ-bench），建立月度测量基线
- [ ] A1-3：建立版本间分数回归报告，每次重大功能上线附带 diff 数据

### 维度 2：真实自治能力
**What the best do**

以 OpenAI CUA（Computer-Using Agent）为例，其自治能力特点：
- **真实环境操作**：用截图感知屏幕，用鼠标键盘交互，无需 API；OSWorld 完整计算机使用评测得分 38.1%，是其上一代的 1.7 倍
- **自我纠错闭环**：任务失败时，模型能通过推理调整策略（Chain-of-Thought + 强化学习）
- **渐进式人工介入**：敏感操作（登录、支付）主动请求用户接管，而非盲目执行
- **长程任务规划**：Deep Research 可独立完成 5–30 分钟的深度研究，分析数百个来源

Gemini 3 通过 Google Antigravity 平台支持「同时执行多步骤任务」，APEX-Agents 长程专业任务得分 33.5%（Gemini 3.1 Pro），Claude Opus 4.6 得 29.8%。

**我们现在的状态**
- 多 Agent 编排架构存在（主 Agent + 分析师/研究员/审计员等子 Agent）
- Dispatch Engine 支持任务路由和并行执行，有灰度切换机制
- 但子 Agent 自治深度有限：任务完成后不主动汇报质量，无自我纠错循环
- CRAS-A 学习引擎设计有自我修改概念，但尚未落地（代码架构存在，真实能力不到位）
- 实际「言出必行」比例（承诺执行的任务真正完成且质量合格）未被量化

**差距等级：🔴 P0**

**行动项：**
- [ ] A2-1：为每个子 Agent 任务增加完成质量自报（而不仅是完成/未完成的布尔状态）
- [ ] A2-2：实现基础自我纠错：Agent 遇到失败时，尝试 1-2 次策略调整后再上报
- [ ] A2-3：引入长程任务拆解能力（类 Deep Research 模式）：1 个复杂目标 → 自动分解为 N 步子任务 → 顺序/并行执行 → 综合结果
- [ ] A2-4：建立「任务成功率」指标，按任务类型分类统计，每周汇报

### 维度 3：运行时治理
**What the best do**

运行时治理是顶级 AI 系统的核心护城河，体现在：

**Anthropic RSP（负责任扩展政策）：**
- 按能力等级划分 ASL-1 到 ASL-4+ 四个安全等级
- 有明确的 Capability Threshold（触发升级的红线），如 CBRN 武器辅助、自主 AI 研究
- 每次模型迭代必须重新评测，评测失败不得部署
- 有外部 AISI（AI 安全研究所）独立验证

**OpenAI Preparedness Framework（CUA 系统卡）：**
- 针对 Agent 特有风险分四类评分：CBRN / 网络安全 / 劝说 / 模型自主
- 每类评分 Low/Medium/High/Critical，只有 Medium 及以下才能部署
- 专门设计了「自主复制评测」（autonomous replication eval）防止失控
- 多层防御：模型层拒绝 + 产品层确认 + 监控层异常检测 + 事后审计

**我们现在的状态**
- 有 `CRITICAL_ENFORCEMENT_RULES.md` 等约束性文档，但依赖文本约定，无量化评测
- 有 ISC（Instruction Safety Compliance）规则检查，但覆盖范围主要是单轮指令
- 缺少 Agent 自主行为边界的明确分级（等效于 ASL 等级的东西）
- 没有「部署门禁」机制：功能写完就上，不需通过任何安全评测
- 审计日志（`logs/config-audit.jsonl`）存在，但无自动分析和告警

**差距等级：🔴 P0**

**行动项：**
- [ ] A3-1：建立最简 Agent 安全分级框架（3 级即可：Observe / Act-Reversible / Act-Irreversible），不同级别需要不同的用户授权
- [ ] A3-2：为所有「不可逆操作」（删除、外发消息、金融操作）添加强制确认拦截
- [ ] A3-3：建立「部署门禁」概念：重大功能上线前需通过 N 条安全检查用例
- [ ] A3-4：审计日志升级为结构化格式，增加自动异常检测（超出预期行为范围时告警）

### 维度 4：唯一入口（Single Entry Point）
**What the best do**

顶级系统都在强调「单一协作界面」：
- **OpenAI ChatGPT Agent**：将 Deep Research + Operator + 代码执行统一入口，用「Agent Mode」一键切换，用户无需感知底层路由
- **Google Antigravity**：新 IDE 平台，将 Gemini 3 的 agent 编码、工具调用、MCP 接入统一在一个 IDE 环境中
- **Anthropic Claude.ai**：工具菜单 + Artifacts + Computer Use 在同一对话界面，用户统一管理所有工具权限

关键特点：
1. 对用户只暴露一个入口，底层多模型/多工具路由对用户透明
2. 上下文在所有工具间共享（任务历史、文件、用户偏好）
3. 单一权限管理，用户在一处控制所有 Agent 的访问权

**我们现在的状态**
- 飞书 IM 作为唯一对话入口已基本建立
- ISC-DTO 路由支持多 Agent 分发，但路由规则需要人工维护
- 不同 Agent workspace（analyst/researcher/reviewer 等）是独立的，上下文不互通
- 用户需要知道不同 Agent 的存在和分工，认知负担高
- 没有「我的所有任务」统一视图（有时完成通知会丢失或在不同 Agent 分散回报）

**差距等级：🟡 P1**

**行动项：**
- [ ] A4-1：完善 ISC-DTO 动态路由：用户无需指定 Agent，由主 Agent 自动选择最优执行者
- [ ] A4-2：建立跨 Agent 共享上下文池（至少做到同一用户的任务历史在所有 Agent 可查）
- [ ] A4-3：建立任务状态统一视图（「我今天派出去的任务，现在都什么状态」一问即知）
- [ ] A4-4：梳理并精简用户可见的入口，让功能层次更清晰（什么时候会触发哪个 Agent）

### 维度 5：闭卷可信度（Closed-book Trustworthiness）
**What the best do**

「闭卷可信度」指：在没有外部检索/搜索的情况下，模型给出的回答有多可靠。

顶级系统在这一维度的特点：
- **HLE（Humanity's Last Exam）无工具模式**：Gemini 3.1 Pro 得 44.4%，Claude Opus 4.6 得 40.0%，GPT-5.2 得 34.5%
- **GPQA Diamond（科学知识，无工具）**：Gemini 3.1 Pro 94.3%，Claude Opus 4.6 91.3%，GPT-5.2 92.4%
- **MMLU（学科知识）**：所有顶级系统均在 89%+

这意味着：即使没有搜索工具，顶级模型在专业领域的知识准确度极高，不靠「不知道就搜索」来兜底。

从 Anthropic 和 OpenAI 的系统设计看，可信度的核心是：
1. 模型自身知识足够扎实（训练质量）
2. 不确定时明确说「不确定」而非伪造答案（校准度 calibration）
3. 当搜索/工具可用时，优先用工具验证，而不是省事靠记忆

**我们现在的状态**
- 我们是在顶级模型（Claude Sonnet 4.6 / GPT-5 系列）之上构建的，基础知识能力由底层模型提供
- 但我们的 Agent 体系存在「过度依赖搜索」的风险：碰到任何不确定的事就调 web_fetch，延迟高、容错差
- 没有建立「模型知识截止日期」的明确声明机制，用户不知道什么信息是可信的
- 子 Agent 产生的内容没有系统性的「可信度标注」（哪些是模型固有知识，哪些是实时抓取）
- closed-book-gate-validation.md 存在，说明有意识，但执行深度不足

**差距等级：🟡 P1**

**行动项：**
- [ ] A5-1：建立「信息来源标注」规范：每条关键结论标明来源（模型知识 / 实时搜索 / 内部文档）
- [ ] A5-2：区分「知识型任务」和「实时信息型任务」，前者优先用模型固有能力回答，减少不必要的 web 调用
- [ ] A5-3：建立 Agent 应答可信度评分（类似：高置信 / 中置信 / 低置信 / 不确定），在汇报时附带
- [ ] A5-4：对 closed-book 场景（无网络、内部敏感任务）增加专项测试用例

### 维度 6：言出法随（Policy-to-Action Fidelity）
**What the best do**

「言出法随」指：我们在规则文档中写的东西，系统实际上是否真的执行了。顶级系统的实践：

**OpenAI Operator 的做法：**
- Policy 转化为三层机制：模型训练层（RLHF 对政策内化）→ 产品层（takeover mode / 确认弹窗）→ 监控层（dedicated monitor model）
- 有独立的「监控模型」专门检测主 Agent 行为是否违反 policy
- 安全策略既有模型内化（软约束），也有产品强制拦截（硬约束），二者互补

**Anthropic 的做法：**
- Constitutional AI：将规则写入模型训练过程，不只是运行时 prompt
- 有独立团队负责「RSP 合规监督」
- 评测结果和部署决策强耦合，形成闭环

**我们现在的状态**
- 有大量规则文档（CRITICAL_ENFORCEMENT_RULES.md、ISC 规则、SOUL.md 等），但都是文本 prompt
- 规则遵守完全依赖 LLM 的 in-context 理解，没有独立的合规检测层
- 已发现过多次「规则写了但执行时没做到」的情况（如：图片 OCR 场景先声称无能力；调度宣称已切换但实际未全量切换）
- 没有自动验证「今天的执行是否符合规则」的机制，只能靠用户发现问题后反馈
- 规则文档分散在多个 workspace，维护不一致

**差距等级：🟡 P1**

**行动项：**
- [ ] A6-1：建立「规则执行回归测试集」：核心规则 → 对应测试用例，每周自动运行
- [ ] A6-2：引入独立的 Policy Checker（可以是轻量 Agent 或规则脚本），监控主 Agent 行为是否越界
- [ ] A6-3：规则集中化管理：将分散在各 workspace 的规则统一收录到主 workspace，形成单一权威来源
- [ ] A6-4：建立「规则偏差事件」日志，每次发现执行偏差时记录根因，形成知识积累

## 四、追赶路线图（按季度）

### Q2 2026（当前季度，1–3 月）：夯实地基

重点：把「不知道自己在哪」变成「知道差距在哪」。

**核心任务：**
1. **评测地基**：选定 2–3 个行业标准 Agent 基准，建立月度测量基线（A1-1/A1-2）
2. **治理地基**：建立 3 级安全分级框架 + 不可逆操作强制拦截（A3-1/A3-2）
3. **自治提升**：子 Agent 完成质量自报 + 基础自我纠错（A2-1/A2-2）

**预期收益：** 3 个月后，能回答「我们系统在代码/工具调用/指令遵循三个维度各处于什么水平」。

---

### Q3 2026（4–6 月）：闭合差距

重点：从「知道差距在哪」变成「关键差距可见缩小」。

**核心任务：**
1. **长程自治**：实现复杂任务自动拆解与执行（类 Deep Research 模式）（A2-3）
2. **统一入口**：完善 ISC-DTO 智能路由，跨 Agent 上下文共享（A4-1/A4-2）
3. **可信度提升**：信息来源标注 + Agent 可信度评分机制（A5-1/A5-3）
4. **言出法随**：规则回归测试集 + Policy Checker 上线（A6-1/A6-2）

**预期收益：** 6 个月后，系统能在 3 个核心 Agent 基准上给出可重复的成绩，安全事件率可测量并持续下降。

---

### Q4 2026（7–9 月）：建立竞争力

重点：从「追赶」到「特定场景领先」。

**核心任务：**
1. 基于评测数据，识别 1–2 个相对优势场景（如：飞书生态集成深度、中文理解、企业内部知识库整合）
2. 在优势场景建立「竞品对比报告」，量化展示差异化价值
3. 对标 Anthropic RSP，形成适合我们体量的「扩展治理框架」

**预期收益：** 能对内对外清晰描述「我们在哪些维度与全球顶级系统接近，在哪些维度有独特价值」。

## 五、风险与不确定性

| 风险 | 概率 | 影响 | 缓解思路 |
|:---|:---:|:---:|:---|
| 底层模型迭代加速，基准数据每月过期 | 高 | 中 | 建立滚动更新机制，关注模型厂商 Release Notes |
| 安全分级框架阻碍快速迭代 | 中 | 中 | 分级要轻量，避免过度工程化；先 Observe 级全覆盖，再逐步收紧 |
| 评测基准与真实用户需求脱节 | 中 | 高 | 同时收集真实用户满意度，不只看基准分数 |
| 竞品 MCP 生态快速扩展（Gemini 的 MCP Atlas 69.2%），我们生态落后 | 高 | 高 | 优先接入高价值 MCP，不求全覆盖 |
| 子 Agent 之间上下文共享带来隐私泄露风险 | 低 | 高 | 先在同一用户粒度内共享，跨用户严格隔离 |
