## Day 2 交付项（按优先级排序）

### P0 — 必须交付（Day 2关门条件）

**D2-01: ISC运行时Enforcement引擎**
- 在benchmark提交、报告生成、代码合并三个关键节点植入gate_check
- 至少3条P0规则有可执行的运行时拦截（不是JSON字段，是真代码）
- 验收：故意提交违规内容，被自动拦截

**D2-02: 真实数据场景Benchmark**
- 从对话历史/decision-log提取真实用户消息作为测试样本
- 至少10条真实数据，覆盖3+不同意图类型
- 必须走LLM路径（至少50%样本）
- 验收：每条样本标注数据来源（session/log/timestamp）

**D2-03: 事件驱动自愈PoC**
- `cron.task.consecutive_errors >= 3` → 自动诊断 → 尝试修复 → 通知
- 至少1个真实故障场景跑通（用今天的cron报错复现）
- 验收：模拟连续报错，系统自动修复不需人工介入

### P1 — 应该交付

**D2-04: 版本号语义化**
- auto-commit前diff分析，按变更类型决定是否递增
- 纯日志/report数字变动不递增版本号
- 验收：连续5次心跳提交，版本号不变

**D2-05: Pipeline E2E独立可运行**
- l3-pipeline-cron.js必须能`node xxx.js`直接跑
- QA能独立复现38/38（或当前真实数字）
- 验收：质量仲裁官从零运行，得到一致结果

**D2-06: 任务自动流转**
- Day N完成 → 自动生成Day N+1计划 → 自动spawn执行
- 验收：Day 2关闭后，Day 3计划自动生成

### P0 — 核心功能（追加）

**D2-07: Cron任务→事件驱动迁移**
- 16个cron任务迁移到事件驱动架构
- cron从"定时执行"变为"定时兜底扫描"，真正触发走事件总线
- 优先迁移核心链路：event-dispatcher(*/5)、ISC变更检测(*/15)、全局自主决策流水线(*/30)、DTO-AEO流水线(每小时)
- 全局自主决策流水线完全重构：文件变更→emit事件→L3 Pipeline路由→按需执行
- 验收：核心4个cron任务改为事件触发+cron兜底双模式

**D2-08: L3 Pipeline从旁路到主路**
- user.message事件真正走L3（IntentScanner→RuleMatcher→Dispatcher v2）
- 灰度：先核心链路走L3，旧路径作fallback
- 验收：至少1条真实事件链路走通L3全流程

### P2 — 延后到Day 3（运营闭环/AI效果/终验）

**D3-01: ISC规则自动回收评测样本**
- eval-sample-auto-collection-001从文档变为可执行
- IntentScanner每次处理真实消息自动入样本池

**D2-08: 凌霄阁裁决自动化触发**
- Day完成 → 自动触发凌霄阁 → 自动收集裁决 → 自动生成修复任务
