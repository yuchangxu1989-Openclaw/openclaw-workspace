# AEO（智能体效果运营）方案分析
## 另一个AI方案 vs 我们系统设计

---

## 1. 另一个AI方案的精华 ✅

### 1.1 核心功能完整
| 功能 | 价值 | 借鉴度 |
|:---|:---|:---:|
| **评测集管理** | 按场景组织评测数据，版本控制 | ⭐⭐⭐⭐⭐ |
| **自动化评测** | 批量并行执行，效率提升 | ⭐⭐⭐⭐⭐ |
| **用户评价沉淀** | 反馈转化为可量化标准 | ⭐⭐⭐⭐⭐ |
| **效果闭环** | ISC↔AEO双向同步 | ⭐⭐⭐⭐⭐ |
| **可视化看板** | 趋势追踪、问题热力图 | ⭐⭐⭐⭐ |

### 1.2 与ISC协作闭环设计优秀
```
ISC下发标准 → AEO执行评测 → 收集评价 → 沉淀原则 → 同步ISC
```
**符合我们系统的ISC-DTO架构**

### 1.3 评测维度详细
- PPT生成：内容质量、格式规范、创意度、完整性
- 报告生成：完整性、准确性、逻辑性、可读性
- 代码生成：正确性、效率、可读性、安全性

## 2. 另一个AI方案的糟粕 ❌

### 2.1 与我们系统的冲突

| 问题 | 另一个AI方案 | 我们系统要求 |
|:---|:---|:---|
| **架构层级** | "ISC下层模块"，硬编码归属 | 应是独立技能，通过DTO订阅ISC |
| **子Agent** | 未提及子Agent独立评测 | 每个子Agent应有独立评测配置 |
| **LEP复用** | 自建评测执行 | 应复用LEP批量执行能力 |
| **配置化** | Python类硬编码 | 应N019/N020配置化 |
| **模型路由** | 未提及 | 评测任务也应MR路由到最优模型 |

### 2.2 缺失的能力
- ❌ 无子Agent个性化评测配置
- ❌ 无MR模型路由集成
- ❌ 无LEP韧性复用
- ❌ 无N022 ISC合规审计
- ❌ 硬编码文件路径

---

## 3. 改进方案：AEO-MR集成设计

### 3.1 核心改进点

| 维度 | 另一个AI | 改进后AEO |
|:---|:---|:---|
| **定位** | ISC下层模块 | **独立技能**，DTO订阅ISC |
| **执行** | 自建评测执行 | **LEP批量执行** |
| **路由** | 无 | **MR路由**（评测→最优模型评估） |
| **配置** | Python类 | **N019/N020 JSON配置** |
| **子Agent** | 未提及 | **独立评测配置** per Agent |

### 3.2 与现有系统集成

```
┌─────────────────────────────────────────────────────────┐
│                    AEO 智能体效果运营                    │
├─────────────────────────────────────────────────────────┤
│  输入: 子Agent执行结果 + 用户反馈                        │
│                    ↓                                    │
│  评测集管理 (N019配置)                                   │
│                    ↓                                    │
│  MR路由 → 选择评测模型 (glm-5评估器)                    │
│                    ↓                                    │
│  LEP批量执行评测任务                                     │
│                    ↓                                    │
│  根因分析 (调用N020)                                     │
│                    ↓                                    │
│  可视化报告 → 飞书卡片                                   │
│                    ↓                                    │
│  本地任务编排 → 同步改进建议到ISC                                 │
└─────────────────────────────────────────────────────────┘
```

### 3.3 子Agent独立评测配置

```json
// skills/agent-ppt-generator/aeo-config.json
{
  "agent_id": "agent-ppt-generator",
  "evaluation_sets": ["ppt-content", "ppt-format", "ppt-creative"],
  "evaluation_model": "{{MODEL_DEEP_THINKING}}",
  "dimensions": {
    "content_quality": { "weight": 0.4, "threshold": 0.8 },
    "format_standard": { "weight": 0.3, "threshold": 0.9 },
    "creativity": { "weight": 0.3, "threshold": 0.7 }
  },
  "auto_evaluation": true,
  "feedback_collection": true
}
```

### 3.4 与MR的集成

评测任务也应路由到最优评估模型：
- 简单评测 → `{{MODEL_GENERAL}}`
- 复杂代码评估 → `{{MODEL_CODE_REVIEW}}`
- 创意评估 → `{{MODEL_DEEP_THINKING}}`

## 4. 建议实施方案

### Phase 1: AEO基础（2天）
- 评测集管理（JSON配置化）
- 基础评测执行（LEP委托）
- MR路由集成

### Phase 2: 闭环（2天）
- 用户评价收集
- N020根因分析调用
- ISC-DTO同步

### Phase 3: 可视化（1天）
- 飞书卡片报告
- 效果趋势图

**总计：5天（vs 另一个AI的10天+）**

## 5. 决策建议

### 立即实施AEO？
**建议：暂缓，理由：**

1. **MR刚上线**，先让MR稳定运行
2. **AEO与N020有重叠**，需先明确边界
3. **当前系统规模**，手动评测足够
4. **资源聚焦**，先完成Elite Memory决策

### 触发AEO的条件
- 子Agent数量 > 10个
- 频繁需要效果对比
- 用户反馈收集成为痛点
