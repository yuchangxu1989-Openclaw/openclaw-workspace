## 1. AEO双轨制：AI效果运营 vs 功能质量运营

### 1.1 明确区分定义

| 维度 | AI效果运营 (AI-Effect) | 功能质量运营 (Function-Quality) |
|:---:|:---|:---|
| **关注点** | 模型智能程度 | 系统功能完整性 |
| **评测对象** | AI模型输出质量 | 功能正确性/稳定性 |
| **核心指标** | 准确率、创造力、相关性 | 可用性、性能、可靠性 |
| **报告类型** | 效果评测报告 | 质量检查报告 |
| **改进方向** | 模型优化、Prompt工程 | 代码修复、架构优化 |

## 2. 深度闭环：用户反馈→AEO→整改

### 2.1 问题收录体系（主动发现）

```
用户反馈来源:
├─ 直接反馈
│  ├─ 飞书消息中的质疑/澄清
│  ├─ 明确的效果投诉
│  └─ 功能使用困惑
│
├─ 间接信号
│  ├─ 反复修改同一问题
│  ├─ 长时间无响应后追问
│  └─ 绕过推荐方案自行处理
│
├─ 系统内部
│  ├─ 执行失败日志
│  ├─ 超时/异常统计
│  └─ 降级触发记录
│
└─ EvoMap网络
   ├─ 其他Agent效果对比
   ├─ 社区评价/排名
   └─ 跨Agent协作反馈
```

### 2.2 用户质疑自动收录机制

**示例场景**: 用户反复提的质疑、澄清

```
用户消息: "你刚才的理解不对，我说的是..."
    ↓
AEO自动收录:
    ├─ 标记为 "理解偏差"
    ├─ 记录上下文 (前后3轮对话)
    ├─ 关联当前技能 (如MR模型路由)
    └─ 触发 N020 根因分析
    ↓
洞察分析:
    ├─ 是否为共性问题? (查询历史记录)
    ├─ 影响范围评估 (多少用户遇到)
    └─ 根因定位 (意图识别? 上下文理解?)
    ↓
整改方案生成:
    ├─ 短期: 补充训练数据
    ├─ 中期: 优化Prompt工程
    └─ 长期: 模型微调或架构调整
    ↓
闭环验证:
    ├─ 更新评测标准 (增加同类边界用例)
    ├─ 重新评测验证
    └─ 用户回访确认
```

## 3. CRAS/DTO/SEEF深度闭环设计

### 3.1 CRAS联动（知识沉淀→评测标准）

```
CRAS学习到的知识
      │
      ├─ 成功案例 → 生成黄金标准
      ├─ 失败模式 → 生成边界测试用例
      └─ 用户偏好 → 调整评测权重
      │
      ▼
AEO评测标准更新
      │
      └─ 自动同步到评测集
```

**具体联动机制**:
```python
# CRAS发现新失败模式时自动触发
def on_cras_failure_pattern_detected(pattern):
    # 1. 生成对应测试用例
    test_case = generate_test_case_from_pattern(pattern)
    
    # 2. 更新AEO评测集
    aeo.add_test_case(skill_name, test_case)
    
    # 3. 调整评测标准（如有必要）
    if pattern.severity > 0.7:
        aeo.adjust_threshold(skill_name, dimension, new_threshold)
    
    # 4. 触发重新评测
    aeo.schedule_reevaluation(skill_name)
```

### 3.2 DTO联动（声明式评测触发）

```
ISC规则变更
      │
      ▼
DTO订阅分发
      │
      ├─ 到AEO: 更新评测标准
      ├─ 到CRAS: 更新知识库
      └─ 到SEEF: 更新技能编排
      │
      ▼
AEO自动调整
      ├─ 评测维度更新
      ├─ 阈值调整
      └─ 触发回归测试
```

**DTO订阅配置**:
```json
{
  "subscription_id": "aeo-evaluation-sync",
  "subscriber": "AEO-System",
  "rules": [
    "N023-aeo-evaluation-standard-generation",
    "N007-v2-skill-quality",
    "N022-architecture-design-isc-compliance"
  ],
  "actions": {
    "on_rule_update": "update_evaluation_standard",
    "on_threshold_change": "adjust_gate_criteria",
    "on_new_skill": "auto_generate_evaluation_set"
  }
}
```

### 3.3 SEEF联动（技能编排→准入准出）

```
SEEF编排新子技能
      │
      ▼
AEO准入检查
      ├─ 评测标准生成 (N023)
      ├─ 准入评测执行
      └─ 效果达标验证
      │
      ├─ 通过 → SEEF发布
      └─ 失败 → SEEF优化 → 重新评测
```

**SEEF-AEO集成**:
```python
# SEEF编排流程中嵌入AEO检查
class SEEFOrchestrator:
    def deploy_subskill(self, skill_config):
        # 1. AEO准入检查
        admission_result = aeo.check_admission(skill_config)
        
        if not admission_result.passed:
            # 2. 失败反馈给SEEF优化
            return SEEFOptimizeRequest(
                issues=admission_result.issues,
                suggestions=admission_result.improvements
            )
        
        # 3. 通过后发布
        return self.publish(skill_config)
```

## 4. 实际案例：用户反复质疑的处理流程

### 场景重现
**用户多次质疑**: "你刚才的理解不对，我说的是..."

### AEO处理流程

```
第1次质疑:
  ├─ AEO收录: 标记为潜在问题
  ├─ 频率统计: 1次（观察中）
  └─ 暂不触发深入分析

第2次质疑（同一类型）:
  ├─ AEO收录: 升级为关注问题
  ├─ 频率统计: 2次（7天内）
  ├─ 自动触发 N020 根因分析
  └─ 初步洞察: 意图分类在模糊输入时偏差

第3次质疑（确认模式）:
  ├─ AEO升级: 确认为系统性问题
  ├─ 生成 Issue: "MR-意图分类-模糊输入偏差"
  ├─ 自动调整: 评测标准增加边界用例
  ├─ 触发整改: 
  │   ├─ 短期: 优化意图分类Prompt
  │   ├─ 中期: 增加模糊输入训练数据
  │   └─ 长期: 评估语义模型升级
  └─ 用户通知: "已针对您反馈的问题进行优化"

闭环验证:
  ├─ 重新评测: 新测试用例通过率验证
  ├─ 灰度发布: 10%流量验证效果
  └─ 用户回访: 确认问题是否解决
```
