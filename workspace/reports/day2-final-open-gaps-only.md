# Day2 仍未收口缺口清单

## 1. 长上下文压缩与记忆淘汰未形成稳定机制
- **问题**：会话一长，Agent 仍容易把无关历史带入当前决策，推高 token、拖慢响应并放大跑偏概率。
- **当前状态**：有零散约束与人工裁剪，但没有稳定、自动、可验证的压缩/淘汰闭环。
- **为什么没收口**：缺少统一的摘要触发条件、记忆分层策略和压缩后质量校验。
- **建议动作**：上线“分层记忆 + 阈值触发摘要 + 摘要后回归校验”，默认只保留任务必需上下文。

## 2. 工具调用成本控制仍偏弱
- **问题**：Agent 在查信息、读文件、跑命令时仍可能出现过读、过查、过调用，效率损耗明显。
- **当前状态**：有“尽量少调工具”的原则，但未沉淀成强约束的调用预算与降级路径。
- **为什么没收口**：缺少按任务类型划分的工具预算、停止条件和失败后最小化重试策略。
- **建议动作**：给常见任务加“工具预算 + 超预算降级 + 命中即停”规则，并把无效调用纳入复盘。

## 3. 失败恢复路径不够标准化
- **问题**：当命令超时、页面异常、权限失败时，Agent 的恢复动作还不够一致，容易反复试错。
- **当前状态**：能恢复部分场景，但不同任务之间的兜底动作不统一。
- **为什么没收口**：没有形成按故障类型分类的标准恢复树，导致恢复质量依赖临场判断。
- **建议动作**：建立 timeout / permission / not-found / flaky 四类标准恢复剧本，限制重复试错次数。

## 4. 结果质量校验仍偏后置
- **问题**：部分任务仍以“做完”为主，而不是“校验完再交付”，存在输出表面完成、实际遗漏的风险。
- **当前状态**：有检查意识，但缺少任务结束前的强制校验清单。
- **为什么没收口**：未把质量门禁嵌入执行流，导致验证动作容易被跳过。
- **建议动作**：对文件产出、数据修改、页面操作三类任务增加强制验收步骤，不通过不结束。

## 5. 子任务拆分粒度仍不稳定
- **问题**：复杂请求下，Agent 仍可能拆分过粗，导致单步跨度大、返工率高。
- **当前状态**：部分复杂任务能拆，但拆分质量不稳定，依赖具体上下文。
- **为什么没收口**：缺少统一的拆分模板与“何时必须拆分”的判定线。
- **建议动作**：为多工具、多阶段、可回滚任务设硬性拆分规则，先计划最小可验证步骤再执行。

## 6. 关键指标未闭环到日常优化
- **问题**：效率与质量问题能被感知，但未稳定映射到少数核心指标持续跟踪。
- **当前状态**：缺少一套日常可读、可比、可追责的运行指标面板。
- **为什么没收口**：没有把 token、工具调用数、首轮成功率、重试率、返工率统一收敛到同一观察口径。
- **建议动作**：先收 5 个核心指标并周更，所有优化动作必须绑定指标变化。
