# ISC规则全链路审计整改需求清单（架构方案）

> 基于《ISC规则全链路审计综合总结报告》形成  
> 目标：将当前"声明—注册—感知—执行"失配体系，整改为可扩展、可审计、可演进的规则治理平台。  
> 范围：优先覆盖已审计问题（含 186 条规则），方案可扩展至 500/1000 条规模。

---

## 0. 执行原则（先根因，后症状）

1. **根因优先**：先修"模型与机制"（schema、词表治理、生成流水线、兼容迁移），再批量修"实例问题"。
2. **最小破坏迁移**：引入双读/别名映射/灰度开关，避免一次性改动冲击运行时。
3. **自动化替代人工逐条补丁**：对 16 条缺失处理器 + N 条骨架处理器采用"分类模板 + 代码生成 + 测试脚手架"策略。
4. **可扩展治理**：所有整改必须支持 500/1000 规则规模，避免后续再次熵增。

---

## 1. 四大系统性缺陷与整改任务清单

---

### 缺陷A：规则声明层异构（触发结构/ID/路径规范不统一）

#### A-1 任务：统一规则元数据 Schema（含意图注册表重构）
- **目标**：建立唯一规则声明规范，消除 `trigger.events[] / trigger.event / trigger:string` 等异构结构；重构意图注册表 schema，支持版本化与可追踪。
- **输入**：
  - 现有规则声明文件全集（186条）
  - 当前意图注册实现与字段定义
  - 审计报告中的异构样例
- **输出产物**：
  1. `rule.schema.v2.json`（强校验）
  2. `intent-registry.schema.v2.json`（重构后）
  3. 迁移映射文档 `schema-migration-map.md`
- **验收标准**：
  - 100% 规则声明可被 v2 schema 校验通过（或被迁移器自动转换后通过）
  - 不允许 `id` 为空、`trigger.events` 为空、`handler.path` 不存在
  - 意图注册至少包含：`intent_id`、`version`、`owner`、`lifecycle`、`trigger_binding`、`capability_binding`、`deprecation`
- **预估工时**：2 人日
- **依赖关系**：无（关键路径起点）

#### A-2 任务：规则身份治理（ID 唯一性、命名规范、追踪索引）
- **目标**：消除空ID、重复ID、命名漂移问题，建立可追溯索引。
- **输入**：
  - 全量规则清单
  - 命名规范草案
- **输出产物**：
  1. `rule-id-index.json`（唯一索引）
  2. `rule-naming-convention.md`
  3. 自动检查器 `lint-rule-id`
- **验收标准**：
  - 空ID = 0，重复ID = 0
  - CI 阻断：命名不合规禁止合入
- **预估工时**：1 人日
- **依赖关系**：依赖 A-1 schema 定义

#### A-3 任务：路径基线统一与解析器收敛
- **目标**：统一处理器路径策略（仓库相对路径），消除绝对路径/技能相对路径混用。
- **输入**：
  - 声明路径与实际文件路径差异清单
- **输出产物**：
  1. `handler-path-policy.md`
  2. 统一解析器 `resolveHandlerPathV2()`
  3. 路径迁移脚本 `migrate-handler-paths.ts`
- **验收标准**：
  - 声明路径可达率 100%
  - 跨环境（dev/ci/prod）解析一致
- **预估工时**：1.5 人日
- **依赖关系**：依赖 A-1

---

### 缺陷B：事件词表漂移（事件注册不一致、长期再漂移）

#### B-1 任务：建立"事件词表治理中心"（Canonical Event Vocabulary）
- **目标**：形成唯一事件主词表 + 别名策略，解决 `updated/modified`、命名风格混用。
- **输入**：
  - 当前规则触发事件全集
  - 总线已注册事件清单
- **输出产物**：
  1. `event-vocabulary.v1.yaml`（主词表）
  2. `event-alias-map.yaml`（别名映射）
  3. `event-lifecycle-policy.md`（新增/废弃流程）
- **验收标准**：
  - 规则事件对主词表映射覆盖率 100%
  - 未注册新事件默认拒绝（需走审批流程）
- **预估工时**：2 人日
- **依赖关系**：可与 A-2 并行；建议在 A-1 后执行

#### B-2 任务：事件注册闸门（防再漂移机制）
- **目标**：把"词表治理"落到 CI/CD 与运行时检查，防止未来漂移。
- **输入**：
  - 词表与别名映射
  - CI pipeline
- **输出产物**：
  1. CI检查器 `lint-event-vocabulary`
  2. 运行时告警器 `event-drift-detector`
- **验收标准**：
  - PR 中新增事件未入词表 -> 必定失败
  - 运行时出现未知事件 -> 1分钟内告警
- **预估工时**：1.5 人日
- **依赖关系**：依赖 B-1

#### B-3 任务：历史规则事件迁移与回写
- **目标**：将现有186条规则触发事件归一到主词表（保留别名兼容）。
- **输入**：
  - 规则文件全集
  - alias map
- **输出产物**：
  1. 自动迁移脚本 `migrate-events-to-canonical.ts`
  2. 迁移报告 `event-migration-report.md`
- **验收标准**：
  - 规则触发事件 canonical 化覆盖率 ≥ 98%
  - 兼容别名命中率可观测，4周后逐步下线低频别名
- **预估工时**：1 人日
- **依赖关系**：依赖 B-1；可与 C 类任务并行

---

### 缺陷C：执行层质量不足（16条缺失处理器 + N条骨架处理器）

#### C-1 任务：处理器分类分层（禁止逐条手工补）
- **目标**：按处理器职责分类（校验型/路由型/写入型/通知型/编排型），为批量生成打基础。
- **输入**：
  - 缺失处理器清单（16条）
  - 骨架处理器清单（N条）
- **输出产物**：
  1. `handler-taxonomy.md`
  2. 每类标准接口契约与错误码规范
- **验收标准**：
  - 100% 缺失/骨架规则可归类
- **预估工时**：1 人日
- **依赖关系**：依赖 A-1（接口契约需与 schema 对齐）

#### C-2 任务：批量生成处理器框架（代码生成器）
- **目标**：通过模板引擎一次性生成 16 条缺失处理器基础实现 + N条骨架补全骨架。
- **输入**：
  - 分类结果
  - 模板定义
- **输出产物**：
  1. `handler-generator` 工具
  2. 生成后的处理器代码与单测骨架
- **验收标准**：
  - 16 条缺失处理器全部落地，可编译可加载
  - N 条骨架处理器补齐"输入校验、失败分支、审计日志"
- **预估工时**：2 人日
- **依赖关系**：依赖 C-1

#### C-3 任务：处理器行为验收（契约测试 + 回归测试）
- **目标**：避免"存在即通过"的伪闭环，建立行为级质量门槛。
- **输入**：
  - 生成处理器
  - 契约样例
- **输出产物**：
  1. `handler-contract-tests/` 测试套件
  2. CI 集成：处理器变更必须通过契约测试
- **验收标准**：
  - 契约测试覆盖率 ≥ 90%
  - 行为回归 0 容忍
- **预估工时**：2 人日
- **依赖关系**：依赖 C-2

---

### 缺陷D：全链路可观测与反熵增机制缺失

#### D-1 任务：规则健康度仪表盘
- **目标**：建立"声明—注册—感知—执行"全链路可观测面板。
- **输入**：
  - 审计脚本产出
  - 运行时日志
- **输出产物**：
  1. `rule-health-dashboard` 定时任务
  2. 健康度指标定义 `rule-health-metrics.md`
- **验收标准**：
  - 可实时查看：规则总数、合规率、触发成功率、处理器执行成功率
  - 异常规则自动标红告警
- **预估工时**：1.5 人日
- **依赖关系**：可在 A/B/C 完成后独立执行

#### D-2 任务：定期自动审计与熵增防护
- **目标**：防止整改后再次退化，建立持续审计机制。
- **输入**：
  - 审计脚本
  - CI pipeline
- **输出产物**：
  1. 每日自动审计 cron `daily-isc-audit`
  2. 审计报告自动推送（飞书/邮件）
  3. 合规率低于阈值自动阻断部署
- **验收标准**：
  - 日审计覆盖 100% 规则
  - 合规率 < 95% 时阻断 + 告警
- **预估工时**：1 人日
- **依赖关系**：依赖 D-1

## 2. 关键路径与并行策略

```
Week 1:
  A-1 (schema) ──→ A-2 (ID治理) ──→ A-3 (路径统一)
       │
       └──→ B-1 (词表中心) ──→ B-2 (闸门) ──→ B-3 (迁移)
       │
       └──→ C-1 (分类) ──→ C-2 (批量生成) ──→ C-3 (契约测试)

Week 2:
  D-1 (仪表盘) ──→ D-2 (自动审计)
```

- **关键路径**：A-1 → C-1 → C-2 → C-3（约 7 人日）
- **最大并行度**：A/B/C 三条线可并行，D 在最后收尾
- **总预估工时**：~19 人日（单人串行）；3 人并行约 9.5 人日

## 3. 风险控制

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Schema 迁移导致运行时规则加载失败 | P0 | 双栈兼容：v1/v2 同时读取，灰度切换 |
| 事件别名映射遗漏导致规则静默失效 | P0 | 迁移前全量 diff + 别名命中计数器 |
| 批量生成处理器质量不足 | P1 | 契约测试 + 人工审查关键路径处理器 |
| 整改完成后再次熵增 | P1 | D-2 自动审计 + 合规率阻断 |

---

## 4. 扩展性与反熵增设计

1. **Schema 版本化**：`rule.schema.v2.json` 支持 `$schema` 声明，未来升级为 v3 时可自动迁移。
2. **事件词表生命周期**：新增事件需 PR 审批 + 词表注册，废弃事件有 4 周过渡期。
3. **处理器模板化**：新增规则只需声明分类 + 填写业务逻辑，框架代码由生成器产出。
4. **持续审计**：每日自动审计 + 合规率门槛，防止"修完即忘"。
