# 任务类型工具预算表

> 目的：为常见任务设定默认工具预算，控制调用次数、执行时长与升级时机，避免低价值反复试探。

## 预算设计原则

- **先轻后重**：优先使用低成本工具，再升级到高成本工具。
- **单轮聚合**：能一次拿到的信息，不拆成多次调用。
- **有信号才升级**：只有出现明确阻塞、信息不足或高不确定性时才提高预算。
- **超预算即汇报**：达到预算上限时，优先输出当前结论、缺口和下一步建议。
- **默认保守**：预算用于默认场景；高价值、高风险、强时效任务可按升级条件放宽。

## 预算字段说明

- **默认预算**：建议的工具调用上限、典型时间上限或资源上限。
- **超预算信号**：说明当前路径可能效率低、重复、信息收益变差。
- **升级条件**：满足后可提高预算、切换更强工具或进入更深入调查。

## 任务类型预算表

| 任务类型 | 常用工具 | 默认预算 | 超预算信号 | 升级条件 |
|---|---|---|---|---|
| 查信息 / 事实检索 | `web_search`、`web_fetch` | 搜索 1–3 次；抓取 1–5 页；总时长 5–8 分钟 | 同义关键词反复搜索仍无新增；来源互相矛盾；开始依赖低质量站点补空白 | 任务要求高可信度；需交叉验证 3 个以上来源；涉及政策、价格、时效性强内容 |

| 构建 / 测试 / 调试 | `exec`、`process` | 主命令 1 次；补充命令 1–3 次；总时长 15–25 分钟 | 失败模式未变化却持续重试；长时间等待无阶段性输出；问题从单点变系统性 | 有明确错误栈可继续缩小范围；用户明确要求修到通过；项目价值足够高 |
| 网页操作 / UI 交互 | `browser` | 1 个标签页；快照 1–3 次；动作 3–10 次；总时长 8–12 分钟 | 连续试点不同元素；页面状态不稳定；需要等待/重载才能推进 | 必须登录、上传、下载、截图取证；任务天然依赖交互闭环 |

| 飞书文档 / 知识库操作 | `feishu_doc`、`feishu_wiki`、`feishu_drive` | 定位对象 1–3 次；内容写入/更新 1–5 次；核对 1 次 | block/节点定位反复失败；权限不足；文档结构复杂导致多轮试插入 | 需要目录迁移、跨空间移动、复杂表格或附件上传 |
| 会话历史 / 上下文追溯 | `sessions_list`、`sessions_history` | 会话定位 1 次；历史读取 1–3 次 | 为补上下文读取过长历史；关键信息仍未收敛；跨多个会话来回切换 | 用户问题明确依赖历史承诺、先前结论或工具输出 |
| 子代理拆分调查 | `sessions_spawn`、`subagents` | 并行子任务 1–3 个；每个目标明确、边界清晰 | 主任务本身还没收敛就盲目拆分；子任务结果重叠；协调成本高于收益 | 明显可并行；存在多来源/多模块独立调查；主代理需要压缩总时长 |
| 多模态取证（截图/相机/录屏） | `browser`、`nodes`、`canvas` | 截图/快照 1–3 次；录屏 1 次短片；总时长 5–10 分钟 | 为同一事实重复采样；画面无新增证据；开始“多拍几次碰碰运气” | 需要保留证据链；远程设备状态必须可视化确认 |

## 通用超预算判定

出现以下任一情况，应视为接近或进入超预算：

1. **重复动作无新增信息**：同类搜索、同类命令、同类点击连续进行，但结论没有推进。  
2. **结果质量下降**：新增结果主要是噪声、重复页面、重复日志或无关文件。  
3. **问题边界扩大**：从单文件变成跨模块，从单网页变成跨系统，从单命令变成环境级故障。  
4. **工具开始互相补洞**：先天信息不足，靠不断换工具弥补，但缺口并未缩小。  
5. **时间消耗超过任务价值**：探索成本明显高于该轮任务应得收益。

## 通用升级策略

### 1. 轻度升级

适用：已获得部分有效线索，但还差关键一跳。

- 搜索次数从 3 次放宽到 5 次。
- 文件读取范围从 3 个放宽到 8–12 个。
- 浏览器动作从 10 次放宽到 20 次。
- 命令执行增加 1–2 个辅助诊断命令。

### 2. 中度升级

适用：任务高价值，且已有明确调查方向。

- 允许跨工具联动：如 `web_search` + `web_fetch` + `browser`，或 `read` + `exec` + `edit`。
- 允许更长超时：如构建/测试命令延长到 5–20 分钟。
- 允许引入子代理并行处理不同子问题。

### 3. 重度升级

适用：用户明确要求“查到底 / 修到通过 / 跑完整流程”，且失败成本高。

- 扩大文件、页面、日志覆盖范围。
- 进行端到端验证、截图取证、批量修改或系统性排障。
- 必须阶段性汇报：当前发现、剩余风险、继续投入的预期收益。

## 执行建议模板

### A. 默认执行口径

- 先用最低成本工具验证方向。
- 达到默认预算前，追求“足够回答”而不是“穷尽一切”。
- 达到默认预算后，优先输出：已知、未知、建议升级路径。

### B. 超预算时的汇报口径

可按以下格式输出：

- **已完成**：列出已查来源 / 已读文件 / 已跑命令 / 已做页面操作。  
- **当前卡点**：说明是信息缺失、权限限制、动态页面、环境错误还是范围扩大。  
- **建议升级**：说明要增加哪类预算，以及为什么值得。  
- **继续投入的预期收益**：例如“再读 5 个相关文件可补齐调用链”或“切换浏览器交互可拿到动态表格内容”。

## 简明默认值速查

| 资源类型 | 默认上限 |
|---|---|
| 搜索次数 | 1–3 次 |
| 页面抓取 | 1–5 页 |
| 文件读取 | 3–8 个文件 |
| 精确修改 | 1–5 处 |
| 命令执行 | 1–3 次 |
| 浏览器快照 | 1–4 次 |
| 浏览器动作 | 3–15 次 |
| 子代理数量 | 1–3 个 |
| 默认调查时长 | 5–15 分钟 |
| 较重任务时长 | 15–25 分钟 |

## 结论

这份预算表适用于大多数常见任务的默认执行策略。核心不是机械限制工具次数，而是用“默认预算—超预算信号—升级条件”三段式控制投入：**先快速拿到足够信息，再根据信号决定是否继续深挖。**
