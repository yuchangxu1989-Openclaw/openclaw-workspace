# Reviewer 与 Analyst 职责边界定义报告

> 日期：2026-03-10
> 作者：researcher subagent

## 1. 现状分析

### 1.1 原始定义

| Agent | 原标题 | 原内容侧重 |
|-------|--------|------------|
| reviewer | 质量仲裁官 🔎 | Review 方法论、三层递进审查、P0-P3 严重度分级、通过/不通过判定 |
| analyst | 系统架构师 📐 | ADR 模板、Anti-entropy 校验、风险优先级矩阵、方案对比 |

### 1.2 SOUL.md 角色模型中的定义

SOUL.md 列出了 6 个角色，其中与本次分析相关的有 3 个：

- **系统架构师**：方案设计、边界定义、技术风险收敛
- **质量仲裁官**：用例设计、回归验证、质量门禁
- **洞察分析师**：技术调研、竞品分析、方案比选

问题：analyst agent 的 AGENTS.md 标题是"系统架构师"，但实际上它同时承担了 SOUL.md 中"系统架构师"和"洞察分析师"两个角色的职责。这本身不是问题（一个 agent 可以覆盖多个角色），但需要明确。

## 2. 重叠地带分析

### 2.1 识别出的模糊场景

| 场景 | 可能给 reviewer | 可能给 analyst | 根因 |
|------|----------------|---------------|------|
| "review 这个架构" | ✓ 因为有"review"关键词 | ✓ 因为涉及"架构" | "review"一词的歧义 |
| "这个方案有风险吗" | ✓ reviewer 检查安全风险 | ✓ analyst 做风险评估 | "风险"的语境不同 |
| "检查这个设计文档" | ✓ reviewer 审查文档质量 | ✓ analyst 评估设计合理性 | "检查"的目的不同 |
| "评估这段代码的质量" | ✓ reviewer 做代码审查 | ✓ analyst 可能被误派 | 缺乏明确路由规则 |

### 2.2 根因

两者的核心区别不在于"看什么"，而在于：

- **时间维度**：reviewer 看已完成的产出物（事后），analyst 看待决策的方案（事前）
- **判定维度**：reviewer 给 pass/fail，analyst 给推荐方案 + 权衡分析
- **问题维度**：reviewer 问"对不对"，analyst 问"行不行"

## 3. 边界设计

### 3.1 核心判定规则

```
已经存在的东西 → reviewer 判定质量
还没做的东西   → analyst 评估方案
```

### 3.2 详细边界矩阵

| 任务类型 | 归属 | 理由 |
|---------|------|------|
| 代码审查（逻辑、安全、规范） | reviewer | 审查已有产出物 |
| 文档审查（完整性、准确性） | reviewer | 审查已有产出物 |
| PR review | reviewer | 审查已有产出物 |
| 部署验证 | reviewer | 验证已完成的部署 |
| Agent 产出评测 | reviewer | 评判已有产出质量 |
| 架构设计 | analyst | 设计未来方案 |
| 技术选型（A vs B） | analyst | 方案比选决策 |
| 可行性论证 | analyst | 评估方案可行性 |
| 技术调研 | analyst | 信息收集与分析 |
| 风险预判（方案层面） | analyst | 评估未实现方案的风险 |

### 3.3 易混淆场景的明确归属

| 场景 | 归属 | 判定依据 |
|------|------|---------|
| "review 这个架构文档" | reviewer | 审查的是文档（已有产出物）的质量 |
| "评估这个架构设计是否合理" | analyst | 评估的是方案（未来决策）的可行性 |
| "这段代码有安全风险吗" | reviewer | 检查已有代码 |
| "这个方案有什么技术风险" | analyst | 评估未实现方案 |
| "检查实现是否符合设计" | reviewer | 验证产出物与规格的一致性 |
| "对比两个技术方案" | analyst | 方案比选 |

## 5. 主Agent路由建议

主 Agent 在分派任务时，可用以下决策树：

```
用户请求
  ├─ 涉及"已有产出物"（代码、文档、部署、PR）？
  │   └─ YES → reviewer
  ├─ 涉及"方案决策"（选型、架构、设计、对比）？
  │   └─ YES → analyst
  ├─ 涉及"可行性/风险"？
  │   ├─ 针对已有实现的风险 → reviewer
  │   └─ 针对未来方案的风险 → analyst
  └─ 仍不确定？
      └─ 看用户期望的输出格式：
          ├─ 期望 pass/fail 判定 → reviewer
          └─ 期望推荐方案 + 分析 → analyst
```
