## 二、失败模式分类（7大根因）

### 根因1：超时 — 24条（最大单一失败原因）

**表现**：`超时自动标记(产出不存在)`，占全部非done任务的27%。

**典型案例**：
| 任务 | 描述 | 耗时 |
|------|------|------|
| eval-full-quality-audit | 622条评测集全量质量审计 | 30min超时 |
| eval-audit-fix-batch-A | 评测集修复A(49条) | 38min超时 |
| eval-audit-fix-batch-D | 评测集修复D(301条) | 38min超时 |
| e2e-eval-l1l2-pipeline | L1L2端到端pipeline | 36min超时 |

**根因**：
- 批次粒度过大：622条全量审计、301条修复等远超subagent单次处理能力
- 虽然ISC规则 `rule.eval-batch-size-limit-001.sh` 限制batch_size≤10，但该规则检查的是config文件，**未拦截任务描述中的实际批次大小**
- 默认超时约30分钟，对大批量eval任务不够用
- 部分任务在超时前已完成大量工作但未及时写入文件，导致"产出不存在"

### 根因2：Agent空转 — 约15条

**表现**：Agent声称要做/已做，但实际未写入任何文件。

**子类型**：
| 子类型 | 数量 | 典型result_summary |
|--------|------|--------------------|
| 文件未写入 | 5 | `空转-文件未写入` |
| 只说计划不执行 | 3 | `空转：只说计划不执行，6秒就结束` |
| 等确认后超时 | 2 | `空转等确认,已推但timeout` / `写了空数组,等确认timeout` |
| 声称完成但无产出 | 3 | `声称写入但文件不存在,boom空转` |
| 探查后退出 | 2 | `reviewer也只做了探查没有实际写脚本就退出了` |

**根因**：
- Subagent自主性不足：遇到不确定情况时选择"等确认"而非自主决策
- 任务描述不够具体：缺少明确的输出文件路径和格式要求
- 部分agent（尤其writer、scout）倾向于输出方案而非直接执行
- boom模型（gpt-5.3-codex）空转率明显高于claude-opus

### 根因3：数据破坏/幻写 — 约8条

**表现**：Agent修改数据文件时造成数据污染，需git恢复。

**典型案例**：
| result_summary | 严重程度 |
|----------------|----------|
| `严重事故:截断45条为3条,已git恢复` | 🔴 严重 |
| `135条膨胀到480条,数据被污染,已git恢复` | 🔴 严重 |
| `误清:原有数据被清成0,boom空转` | 🔴 严重 |
| `45条0标记,幻写,已git恢复` | 🟡 中等 |
| `30条0标记+条数变了,已git恢复` | 🟡 中等 |
| `20→30条膨胀+0标记,已git恢复` | 🟡 中等 |

**根因**：
- 缺少写前校验：Agent直接覆写JSONL文件，无条数/schema校验
- 缺少写后断言：无自动化检查确认写入后数据完整性
- 幻写问题：Agent生成了格式正确但内容为空/0标记的数据
- 级联破坏：eval-clean-batch-04的数据被eval-clean-batch-03清空，前后任务互相踩踏

### 根因4：环境/路径/工作区隔离 — 约8条

**表现**：Agent找不到文件或目录。

**典型案例**：
- `scout工作区是空模板,目标文件都在主workspace,需要用绝对路径重派`
- `scout工作区没有skills/scripts目录,需要指定主workspace路径`
- `文件不存在,boom-scout空转`（2条）
- `环境阻断未完成审计`

**根因**：
- 每个agent有独立workspace（`/root/.openclaw/workspace-{agent}/`），但eval数据在主workspace（`/root/.openclaw/workspace/`）
- 任务描述中使用相对路径，agent在自己的workspace中找不到文件
- 未统一使用绝对路径 `/root/.openclaw/workspace/` 前缀

### 根因5：外部依赖/API失败 — 约6条

**表现**：第三方API调用失败。

| 依赖 | 失败数 | 表现 |
|------|--------|------|
| GLM-5 API | 3 | `GLM-5 network_error` / `写入前断开` |
| boom渠道 | 2 | `boom渠道gpt-5.3-codex不支持` |
| 飞书API | 1 | `超时15分钟，卡在飞书API` |

**根因**：
- GLM-5 API不稳定，网络错误频发
- boom渠道配置问题：部分agent尝试用boom渠道调gpt-5.3-codex但该渠道不支持
- 飞书API无超时保护，卡住后整个任务超时

### 根因6：代码/脚本Bug — 约6条

**表现**：eval脚本本身有bug。

| Bug | 影响 |
|-----|------|
| `providers是object不是array` | eval-codex-pilot和eval-opus-pilot都因此失败 |
| `run-eval.js不支持数组格式JSON` | 评测脚本无法处理新格式数据 |
| `旧代码,待重跑` / `旧代码不支持数组格式` | 5条任务因代码未更新失败 |
| `脚本报错未落盘+又等确认` | 错误处理不当 |

**根因**：
- `eval-codex-pilot.js` 硬编码 `config.models.providers['boom-main']`，但providers是object格式，读取方式有误
- 数据格式从V3迁移到V4时，脚本未同步更新
- 缺少CI/预检：脚本变更后无自动化测试

### 根因7：任务粒度/输入过大 — 约4条

**表现**：输入数据超出agent处理能力。

- `events.jsonl太大(2728行)读取截断，未完成审计`
- `日志单行超50KB无法读取,等确认才继续=空转`
- `粒度太粗撤回，改为1规则1任务`

**根因**：
- 大文件读取被截断，agent无法获取完整数据
- 未预先拆分大文件为可处理的chunk

## 五、改进建议

### P0（立即执行）

1. **强制绝对路径**：所有eval任务描述必须使用 `/root/.openclaw/workspace/` 绝对路径，不允许相对路径
2. **批次上限硬限制**：单个subagent任务处理的eval case数 ≤ 50条（当前ISC规则限制10条但未生效）
3. **写前写后校验**：eval数据文件修改前后必须校验条数一致性，防止截断/膨胀/清空

### P1（本周内）

4. **修复eval脚本**：`eval-codex-pilot.js` 和 `eval-opus-pilot.js` 的providers读取bug
5. **超时分级**：小批次(≤20条)=15min，中批次(20-50条)=30min，禁止>50条单任务
6. **空转检测**：subagent完成时自动检查是否有文件写入，无产出则标记warning

### P2（持续优化）

7. **任务互斥锁**：同一eval数据文件不允许被多个subagent同时修改（防止级联破坏）
8. **API fallback**：GLM-5失败自动切claude，boom渠道失败自动切penguin
9. **大文件预处理**：>1000行的文件自动拆分后再分发给subagent

## 六、结论

Eval pipeline是任务看板最大失败源（89/152=59%的非done任务），核心问题不是eval任务本身有多难，而是**基础设施层面的系统性缺陷**：

- **批次太大** → 超时（27%）
- **Agent行为不可靠** → 空转+幻写（26%）
- **工作区隔离未适配** → 路径找不到（9%）
- **脚本未跟上数据格式迭代** → 代码bug（7%）

这些问题叠加后形成"失败→重试→再失败"的恶性循环，240条eval任务中产生了大量重复劳动。修复P0的3项措施预计可消除约50%的失败。
