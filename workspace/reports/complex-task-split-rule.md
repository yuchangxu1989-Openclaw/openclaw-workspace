# 复杂任务拆分规则

## 目的
当任务具备较高复杂度、风险或不确定性时，必须先拆成可独立验证的子任务，再逐步执行，以降低误操作、便于回滚、提升可审计性。

## 必须拆子任务的判定线
满足以下任一条件，即必须拆分：

### 1. 多工具协同
若完成任务需要使用 **2 种及以上工具**，且工具之间存在依赖关系，必须拆分。

**典型情形：**
- 先检索/读取，再编辑/写入
- 先执行命令，再解析结果，再修改文件
- 浏览器操作与本地文件修改联动

**判定线：**
- 单工具单步可完成：可不拆
- 多工具且前一步输出决定后一步输入：必须拆

### 2. 多阶段执行
任务天然包含 **2 个及以上阶段**，且阶段之间存在“完成—验证—再继续”的合理边界时，必须拆分。

**常见阶段：**
- 现状识别
- 方案生成
- 实施变更
- 验证结果
- 收尾/清理

**判定线：**
- 若可明确分出“调查”和“实施”两个阶段：必须拆
- 若实施后还需额外验证、发布、迁移、同步：必须继续细分

### 3. 可回滚要求明显
只要变更对象具有较强持久性，或错误后恢复成本高，就必须拆分并显式设计回滚点。

**高回滚需求对象：**
- 代码库核心文件
- 配置文件
- 数据库/表结构/批量数据
- 自动化脚本
- 权限、账号、消息发送、外部系统写操作

**判定线：**
- 若修改后不能靠一次简单覆盖恢复：必须拆
- 若影响范围可能超出当前文件/当前命令：必须拆
- 若需在执行前保留基线、备份、diff 或快照：必须拆

### 4. 高风险修改
涉及高风险动作时，必须拆分为“风险识别 → 小步实施 → 验证 → 扩大范围”。

**高风险场景：**
- 删除、覆盖、批量替换
- 生产/线上环境操作
- 权限变更、安全策略变更
- 对外发送消息、通知、API 写入
- 依赖安装/升级/迁移
- 不可逆或近似不可逆操作

**判定线：**
- 命令或操作具有删除、覆盖、批改、批量写入性质：必须拆
- 涉及线上、共享资源、多人协作环境：必须拆
- 出错可能造成服务中断、数据损坏、误通知：必须拆

### 5. 范围或影响面不易一次看清
若任务影响面跨文件、跨模块、跨环境、跨系统，必须拆分。

**判定线：**
- 修改文件数 > 3：建议拆；> 5：必须拆
- 涉及多个目录/模块/服务：必须拆
- 需要先确认引用关系、依赖链、调用链：必须拆

### 6. 不确定性较高
如果需求、上下文、执行路径、工具输出存在明显不确定性，必须拆分为先求证再实施。

**判定线：**
- 关键前提未经验证
- 文件/接口/页面是否存在未知
- 命令副作用不明确
- 预期结果缺少可观测指标

满足任一项：必须先做“最小探测步骤”。

## 可不拆的情形
同时满足以下条件时，可不拆或仅做轻量拆分：
- 单工具即可完成
- 单文件或单对象的小改动
- 影响范围清晰且局部
- 变更可即时验证
- 回滚简单明确
- 无外部副作用

**示例：**
- 读取单个文件并修正一个明显错字
- 在已确认存在的目录下新建一个简单说明文件
- 执行无副作用的信息查询命令

## 强制拆分模板
当判定必须拆分时，至少拆为以下结构：

1. **现状确认**
   - 确认对象存在
   - 确认当前状态
   - 确认影响范围

2. **最小变更设计**
   - 定义只改变什么
   - 定义不改变什么
   - 明确成功标准

3. **小步执行**
   - 先在最小范围实施
   - 保留中间结果
   - 避免一次性大改

4. **结果验证**
   - 验证语法/结构/输出
   - 验证副作用是否符合预期
   - 验证未影响无关对象

5. **回滚准备或执行**
   - 记录回滚入口
   - 必要时恢复基线
   - 清理临时产物

## 最小可验证步骤模板
以下模板适用于多工具、多阶段、可回滚、高风险修改场景。

### 模板 A：通用最小可验证步骤
1. **识别目标**
   - 目标对象是什么
   - 目标边界是什么
   - 成功标准是什么

2. **读取现状**
   - 只读获取当前状态
   - 记录关键事实
   - 标出未知项

3. **建立基线/回滚点**
   - 保存原始内容、快照、diff、备份或命令输出
   - 明确如何恢复

4. **执行单一最小变更**
   - 一次只做一种性质的变更
   - 避免同时改结构、命名、逻辑、配置

5. **立即验证**
   - 验证变更是否生效
   - 验证是否引入错误
   - 验证影响是否局限在预期范围

6. **决定下一步**
   - 验证通过：进入下一子任务
   - 验证失败：停止并回滚或重新评估

### 模板 B：多工具任务
1. 工具 1：采集现状
2. 工具 2：基于现状生成变更
3. 工具 2 或 3：执行最小修改
4. 工具 1 或校验工具：验证结果
5. 必要时：回滚或修正

**要求：**
- 每次工具切换前，必须明确上一工具产出的可验证结果
- 不允许在关键事实未确认前直接写入

### 模板 C：高风险修改
1. 标记风险点
2. 缩小作用范围
3. 先做可逆步骤
4. 验证可逆步骤无误
5. 再执行高风险步骤
6. 立刻做结果与副作用验证
7. 保留回滚记录

## 明确执行规则

### 必须先验证再继续的节点
以下节点必须停下来验证：
- 第一次写入前
- 第一次删除/覆盖前
- 从只读阶段切换到写阶段时
- 从局部修改切换到批量修改时
- 从测试/草稿环境切换到正式对象时

### 必须单独成子任务的动作
以下动作不能与其他动作混成一步：
- 备份/快照
- 批量替换
- 删除
- 迁移
- 权限调整
- 对外发送
- 发布/提交

### 失败停止线
出现以下任一情况，应停止后续子任务：
- 关键前提与预期不符
- 验证结果不通过
- 影响范围超出预估
- 无法明确回滚路径
- 工具输出自相矛盾或不稳定

## 简化判定口诀
可用以下口诀快速判断：

**“多工具、分阶段、难回滚、高风险、范围大、不确定”——命中一条就拆。**

若同时命中 2 条及以上，应采用严格拆分，并在每一步后验证；若命中 3 条及以上，应默认按高风险流程处理。

## 建议输出格式
在真正执行复杂任务前，可先写出如下子任务清单：

- 子任务 1：确认现状与边界
- 子任务 2：建立基线与回滚点
- 子任务 3：执行最小改动
- 子任务 4：验证结果与副作用
- 子任务 5：扩大范围或结束

以上结构应优先于“一次做完”的执行方式。
