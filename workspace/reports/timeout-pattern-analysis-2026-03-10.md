## 二、核心发现：两种完全不同的超时原因

对 66 个超时任务的 `completeTime` 进行聚类分析后，发现超时分为两大类：

### 类型 A：批量超时 — Gateway 重启/清理（32个，48%）

多个任务在**同一秒**被标记为超时，这不是任务本身跑超时，而是 Gateway 进程重启或 session 清理导致所有在飞任务被一刀切。

| 时间点 | 同时超时数 | 典型任务 |
|--------|-----------|---------|
| 03-08 12:38:10 | 5个 | programmatic-completion-hook (177min), lto-core-dirname-rename (119min) |
| 03-09 05:13:16 | 3个 | fix-pdca-eval-stats (63min), context-engine-lifecycle-analysis (45min) |
| 03-10 00:35:51 | 2个 | isc-compliance-fix-remaining-handlers (395min!), voice-chat-process-guard (392min) |
| 03-10 10:22:00 | 6个 | agent-role-prompts-upgrade, daily-report-0310, skill-health-scan 等 |

**特征**: 耗时从 30min 到 395min 不等，但 completeTime 完全一致 → 外部强制终止。

### 类型 B：独立超时 — 真正撞上超时墙（34个，52%）

这些任务是一个一个独立超时的，耗时分布极度集中：

```
耗时分布（独立超时，34个）:
  <15min:   3个  ← 异常早退（可能是其他错误被标记为timeout）
  30-31min: 21个 ← ⚠️ 撞上30分钟硬限制
  31-35min: 7个  ← 略超30min（清理延迟）
  35-45min: 3个  ← 可能有不同的timeout配置
  >45min:   0个
  >200min:  1个  ← glm-5模型异常（205min）
```

**中位数: 31.1 分钟 | 平均: 34.8 分钟**

结论：**系统存在一个 ~30 分钟的默认超时阈值**，绝大多数独立超时都精确撞在这个墙上。

所有 66 个超时任务的 `retry_count` 均为 0。**没有任何超时任务被自动重试**，这意味着每次超时都是纯粹的工作浪费。

### 3个异常早退（<15min）

| 任务 | Agent | 耗时 |
|------|-------|------|
| misplaced-code-audit | analyst | 4.7min |

这 3 个任务远未到 30 分钟就被标记为 timeout，可能是 session 连接断开或其他异常，建议增加 `timeout_reason` 字段区分。

## 七、根因总结

```
超时的 66 个任务
├── 48% 批量超时（Gateway 重启/清理杀死在飞任务）
│   └── 根因: Gateway 不稳定 + 无优雅关闭机制
├── 47% 撞上 30 分钟硬限制
│   └── 根因: 默认 timeout 对复杂任务偏短
├── 3% 异常早退（<15min）
│   └── 根因: session 断开或其他未知错误
└── 2% 模型异常（glm-5 跑了 205min）
    └── 根因: 模型卡死 + 缺少心跳检测
```

## 八、建议：Timeout 配置优化

### 1. 分级超时策略（最高优先级）

当前一刀切 30 分钟不合理。建议按 Agent 角色分级：

| Agent 类型 | 建议 Timeout | 理由 |
|-----------|-------------|------|
| reviewer | 20 分钟 | 任务聚焦，2% 超时率已很低 |
| researcher, scout | 30 分钟 | 维持现状，超时率可接受 |
| writer | 40 分钟 | 长文写作需要更多时间 |
| coder | 45 分钟 | 编码实现是最耗时的任务类型 |
| analyst | 45 分钟 | 深度分析/审计需要充裕时间 |

### 2. 实现自动重试（高优先级）

当前 0% 重试率 = 100% 浪费率。建议：
- 独立超时任务自动重试 1 次（`max_retry: 1`）
- 批量超时（Gateway 重启）任务自动重新入队
- 重试时适当增加 timeout（如 1.5x）

### 3. Gateway 优雅关闭（高优先级）

48% 的超时来自 Gateway 重启。建议：
- Gateway 重启前等待在飞任务完成（或设置 drain 超时）
- 将 Gateway 重启导致的终止标记为 `killed` 而非 `timeout`，便于区分
- 记录 `timeout_reason` 字段（`wall_clock` / `gateway_restart` / `session_lost`）

### 4. 心跳检测（中优先级）

glm-5 任务跑了 205 分钟才被发现，说明缺少运行时健康检查。建议：
- 子Agent 每 5 分钟发送心跳
- 连续 3 次无心跳则标记为 `stale` 并终止
- 避免僵尸任务长期占用资源

### 5. 动态超时（低优先级，长期）

根据任务描述/标签预估复杂度，动态设置 timeout：
- 包含 "全量"、"审计"、"pipeline" 等关键词 → 60 分钟
- 包含 "fix"、"cleanup" 等关键词 → 30 分钟
- 评测类任务（eval-*）→ 按数据量线性估算

### 预期收益

如果实施建议 1-3：
- 批量超时 32 个 → 降至 ~0（优雅关闭 + 重新入队）
- 独立超时 31 个撞墙的 → 预计 60%+ 可在 45min 内完成
- 保守估计超时率从 9.7% 降至 **3-4%**
