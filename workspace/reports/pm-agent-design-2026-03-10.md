# PM Agent（项目经理Agent）设计方案

> 版本: v1.0 | 日期: 2026-03-10 | 作者: researcher (系统架构师)
> 状态: 设计稿，待评审

---

## 摘要

**核心结论：引入PM Agent作为"流程大脑"，让主Agent专注战略决策，PM负责流程编排和质量门禁。**

当前系统的main Agent（战略家）同时承担战略决策和流程管理两种职责，导致认知负荷过重、流程卡点不够刚性、进度追踪依赖人工。PM Agent的引入目标是：

1. **解耦战略与流程**：main只做"做什么"的决策，PM负责"怎么做、做到哪了、质量够不够"
2. **流程刚性化**：借鉴IPD的TR/DCP机制，将关键卡点从"提醒"升级为"程序化门禁"
3. **主动调度**：PM自动处理80%的流程性决策（子任务派发、质量核查触发、超时重试），只有20%的关键决策升级到main或用户

设计原则：**轻量可行 > 体系完备**。不照搬100人团队的IPD全流程，只取5个核心机制。

## 1. 背景与问题分析

### 1.1 现有架构

```
用户 ──→ main（战略家 🎖️）
              │
              ├── researcher（系统架构师 🔍）   × 2 实例
              ├── coder（开发工程师 💻）        × 2 实例
              ├── reviewer（质量仲裁官 🔎）     × 2 实例
              ├── writer（创作大师 ✍️）          × 2 实例
              ├── analyst（洞察分析师 📊）       × 2 实例
              ├── scout（情报专家 🎯）          × 2 实例
              ├── cron-worker（定时任务 ⏰）     × 2 实例
              └── worker-03~06（通用执行者 ⚡）  × 4 实例
```

总计 19 个并发slot，由 multi-agent-dispatch 引擎调度。

### 1.2 现有流程基础设施

| 组件 | 功能 | 现状 |
|------|------|------|
| **ISC（智能标准中心）** | 188条规则，覆盖命名/质量/流程/架构 | 成熟，有事件桥接和handler体系 |
| **PDCA引擎** | 状态机（init→plan→do→check→act→done），8个门禁 | 已实现角色分离、准入准出 |
| **multi-agent-dispatch** | 19 lane调度器，task-board持久化 | 成熟，有批次管理和历史查询 |
| **completion-handler** | 子Agent完成事件的标准处理流程 | 已程序化（ISC-COMPLETION-HANDLER-001） |

### 1.3 痛点诊断

| 痛点 | 根因 | 当前表现 |
|------|------|----------|
| **main认知过载** | 战略决策+流程管理+进度追踪 全压在main上 | main需要频繁做"下一步该派谁"这种流程性判断 |
| **流程卡点不刚性** | 质量核查靠echo提示，main可以跳过 | completion-handler输出"需要质量核查"，但main不一定执行 |
| **进度黑洞** | 用户无法知道整体进展百分比 | 只能通过task-board的running/done状态推断 |
| **缺少阶段交付物管理** | 没有"需求文档→设计文档→代码→测试报告"的交付链 | 各Agent产出物散落在workspace中 |
| **重试/超时处理被动** | completion-handler能检测到空闲slot，但只是提示 | "扩列提醒"依赖main主动响应 |

## 2. IPD精髓提炼

### 2.1 取舍原则

华为IPD是为数百人团队设计的重量级流程，我们需要**极度精简**。取舍标准：

- ✅ **取**：能用代码/规则自动执行的机制
- ✅ **取**：能降低main决策负荷的卡点
- ❌ **弃**：需要人类会议/投票的流程
- ❌ **弃**：需要组织层级的管理机制
- ❌ **弃**：文档模板/签审/版本冻结等重型文档管理

### 2.2 五个核心机制

#### 机制一：轻量TR（技术评审）→ 阶段门禁

**IPD原版**：TR1~TR6，每个技术评审点需要多人评审会议，有正式的评审报告和问题跟踪。

**我们的简化版**：

```
TR = 自动化质量门禁
    ├── 准入条件：上一阶段交付物存在且格式合规
    ├── 检查方式：PM自动执行checklist（不开会）
    ├── 通过标准：所有必选项✅ + 可选项≥80%
    └── 不通过处理：自动打回到上一阶段，附上具体缺陷列表
```

简化要点：
- 不分TR1~TR6，只区分**轻检查**（文档存在性/格式）和**重检查**（派reviewer深度审）
- 轻检查PM自己做（秒级），重检查PM派reviewer/analyst（分钟级）
- 没有"评审会议"，只有"检查+判定+记录"

#### 机制二：轻量DCP（决策检查点）→ 升级触发器

**IPD原版**：DCP0~DCP4，跨部门高管参与的决策会议，决定项目继续/重新规划/终止。

**我们的简化版**：

```
DCP = 升级触发器
    ├── 触发条件：方向性变化、资源超限、用户原始意图可能被偏离
    ├── 升级目标：main Agent（对应IPD的IPMT）或 用户（对应IPD的IRB）
    └── 决策选项：继续 / 调整方向 / 终止
```

简化要点：
- 不按阶段固定设置DCP，改为**条件触发**
- DCP不是"会议"，是PM发给main/用户的一条结构化消息 + 等待回复
- 只有3种触发条件：①方向偏移风险 ②资源/时间超限 ③用户需确认的岔路口

#### 机制三：阶段交付物管理 → 产出物登记表

**IPD原版**：每个阶段有严格定义的交付物清单，需要基线化和配置管理。

**我们的简化版**：

```
阶段产出物 = 结构化文件路径 + 状态标记
    ├── 需求阶段：requirements.md（用户意图 + 验收标准）
    ├── 设计阶段：design.md（方案 + 接口定义 + 风险评估）
    ├── 实现阶段：代码/文档/报告（实际产出物）
    ├── 验证阶段：review-report.md（质量核查结果）
    └── 交付阶段：delivery-summary.md（最终摘要 + 交付清单）
```

简化要点：
- 不做版本冻结和基线化，只做"产出物存在且被PM登记"
- 产出物路径在project-plan中预定义，PM检查时只看"文件存在+非空+格式对"
- 每个项目一个 `project-<id>/` 目录，交付物全在里面

#### 机制四：风险管理 → 自动风险雷达

**IPD原版**：风险登记册，定期风险评审会议，风险等级和缓解计划。

**我们的简化版**：

```
风险雷达 = 自动检测 + 分级响应
    ├── 🟢 低风险：PM自动处理（如子任务轻微超时→自动重试）
    ├── 🟡 中风险：PM处理+通知main（如关键路径延误→调整并行度）
    └── 🔴 高风险：升级到main/用户（如核心方向偏移→暂停等决策）
```

自动检测的风险类型：
| 风险类型 | 检测方式 | 响应等级 |
|----------|----------|----------|
| 子任务超时 | task-board中running超过预期时间 | 🟢→自动重试 / 🟡→通知 |
| 资源阻塞 | 19 slot全占满+queue堆积 | 🟡→优先级调整 |
| 依赖冲突 | 任务B依赖任务A的产出，但A失败 | 🟡→自动重排 / 🔴→升级 |
| 方向偏移 | 子Agent产出与用户原始意图不符 | 🔴→升级到用户 |
| 质量退化 | 连续多次质量检查不通过 | 🟡→换Agent重做 / 🔴→升级 |

#### 机制五：阶段化流程 → 五阶段标准流程

**IPD原版**：概念→计划→开发→验证→发布→生命周期，每个阶段数月。

**我们的简化版**（适配AI Agent的分钟级执行周期）：

```
需求理解 → 方案设计 → 并行实施 → 质量验证 → 交付确认
 (分钟)      (分钟)     (分钟~小时)   (分钟)      (分钟)
```

| 阶段 | PM动作 | 准入条件 | 准出条件 |
|------|--------|----------|----------|
| **需求理解** | 解析用户意图，输出结构化需求 | 用户输入存在 | 需求文档有明确的验收标准 |
| **方案设计** | 拆解任务为里程碑，分配Agent角色 | 需求文档通过审核 | 项目计划有WBS+时间估算 |
| **并行实施** | 调度子Agent执行，维护进度 | 项目计划通过审核 | 所有子任务completed |
| **质量验证** | 触发reviewer/analyst检查 | 所有产出物已登记 | 质量报告全部pass |
| **交付确认** | 汇总结果，确认用户满意 | 质量报告通过 | 用户确认或自动关闭 |

## 3. PM Agent职责定义

### 3.1 角色定位

```
PM Agent = 流程大脑 + 调度中枢 + 质量守门人
```

**类比**：如果main是CEO（决定公司战略），PM就是COO（确保战略落地执行）。

**不做的事**：
- ❌ 不做具体的代码编写、文档撰写、数据分析（那是专业Agent的事）
- ❌ 不做战略层面的"该不该做这个项目"的决策（那是main的事）
- ❌ 不直接与用户沟通（除非main授权）

### 3.2 六大职责

#### 职责一：目标管理（从意图到里程碑）

**触发**：main将用户请求转交给PM时。

**流程**：
1. 接收main传递的用户意图（可能是一句话，也可能是结构化需求）
2. 解析为**结构化需求文档**：目标、范围、验收标准、约束条件
3. 拆解为**里程碑**（milestone）：每个milestone有明确的交付物和完成标准
4. 进一步拆解为**子任务**（task）：每个task指定执行Agent角色、预估时间、依赖关系
5. 输出**项目计划**（project-plan.json）

**产出物**：
```json
{
  "projectId": "proj_20260310_xxx",
  "goal": "用户的原始意图",
  "milestones": [
    {
      "id": "M1",
      "name": "需求分析完成",
      "deliverables": ["requirements.md"],
      "tasks": [
        {
          "id": "T1.1",
          "title": "分析用户意图",
          "agent": "researcher",
          "estimatedMinutes": 10,
          "dependencies": [],
          "acceptanceCriteria": "输出结构化需求文档"
        }
      ]
    }
  ],
  "risks": [],
  "phase": "planning",
  "createdAt": "2026-03-10T01:00:00Z"
}
```

#### 职责二：流程管控（五阶段门禁）

**核心**：维护项目在五阶段流程中的推进，确保每个阶段的准入/准出条件被满足。

**PM在每个阶段的具体动作**：

| 阶段 | PM做什么 |
|------|----------|
| 需求理解 | 自己解析（轻量需求），或派researcher深挖（复杂需求） |
| 方案设计 | 自己拆解任务+做WBS，或派researcher做技术方案 |
| 并行实施 | 调用dispatch引擎派发任务，监控task-board |
| 质量验证 | 自动触发reviewer/analyst检查，收集检查结果 |
| 交付确认 | 汇总结果，通过main传递给用户 |

**阶段转换的门禁检查**：PM在每次阶段推进前自动执行checklist，这里复用PDCA的gate机制思路，但更轻量。

#### 职责三：质量门禁（自动介入检查）

**核心**：将当前的"echo提示需要质量核查"升级为"PM自动派发核查任务"。

**检查层次**：

| 层次 | 内容 | 执行者 | 耗时 |
|------|------|--------|------|
| L1 格式检查 | 文件存在、非空、格式合规 | PM自己 | 秒级 |
| L2 内容检查 | 逻辑完整性、与需求对齐 | PM自己（通过prompt） | 秒级 |
| L3 深度审查 | 代码质量/文档质量/方案合理性 | PM派analyst | 分钟级 |

**规则**：
- coder产出 → 必须L3（派analyst审查）
- writer产出 → L2（PM自查）+ 可选L3
- researcher产出 → L2（PM自查与需求对齐）
- 复用ISC-EVAL-ROLE-SEPARATION-001：审查者≠执行者

#### 职责四：进度追踪（主动汇报）

**核心**：PM是task-board的"解读者"，不只维护数据，还主动解读并汇报。

**追踪内容**：
```
项目进度报告：
├── 整体进展：██████░░░░ 60%（3/5阶段完成）
├── 当前阶段：并行实施
├── 活跃任务：4个running，2个queued
├── 已完成：8/12子任务
├── 阻塞项：T3.2超时（已自动重试1次）
├── 预计完成：~15分钟后
└── 风险：🟡 T3.2如果重试仍失败，可能需要换方案
```

**汇报时机**：
- 阶段转换时（主动汇报）
- 关键里程碑完成时（主动汇报）
- 每N个子任务完成时（batch汇报）
- 用户/main主动查询时（按需汇报）
- 风险触发时（告警式汇报）

#### 职责五：风险预警（自动检测与缓解）

**核心**：PM持续扫描task-board和项目状态，自动识别和响应风险。

**风险检测循环**（PM的"心跳"）：
```
每次收到completion事件时：
  1. 更新task-board状态
  2. 检查：是否有超时任务？
  3. 检查：是否有依赖阻塞？
  4. 检查：是否所有slot都被有效利用？
  5. 检查：项目整体是否偏离预期时间线？
  6. 如果发现风险 → 执行响应策略
```

**自动缓解措施**（PM自主执行，不需审批）：
- 超时重试：标记失败→enqueue重试（最多3次）
- 空闲回填：发现空slot→从queue中取任务派发
- 优先级调整：关键路径任务优先级提升
- Agent替换：某Agent连续失败→换另一个同角色Agent

#### 职责六：决策分流（该找谁决定）

**核心**：PM的最关键能力——判断一个决策是自己做、通知main、还是升级。

**分流模型**：
```
收到一个需要决策的情况：
  ├── 流程性？（如"下一个该派哪个Agent"）→ PM自己决定
  ├── 战术性？（如"方案A和方案B选哪个"）→ PM决定，通知main
  ├── 战略性？（如"这个方向对不对"）→ 升级到main
  └── 关乎用户期望？（如"需求理解对不对"）→ 升级到用户
```

详见[第5章 决策权限矩阵](#5-决策权限矩阵)。

## 4. 与现有系统集成方案

### 4.1 整体集成架构

```
                          ┌──────────────┐
                          │     用户     │
                          └──────┬───────┘
                                 │ 意图/确认
                          ┌──────▼───────┐
                          │   main 🎖️    │
                          │  (战略决策)   │
                          └──────┬───────┘
                                 │ 任务委托 / DCP升级
                          ┌──────▼───────┐
                          │   PM 📋      │
                          │  (流程管控)   │
                          └──────┬───────┘
                    ┌────────────┼────────────┐
                    │            │            │
              ┌─────▼──┐  ┌─────▼──┐  ┌─────▼──┐
              │dispatch │  │ PDCA   │  │  ISC   │
              │ engine  │  │ engine │  │  core  │
              └────┬────┘  └────┬───┘  └────┬───┘
                   │            │           │
            ┌──────▼──────────▼─────────────▼──────┐
            │         子Agent执行层                  │
            │  researcher/coder/reviewer/analyst/   │
            │  writer/scout/worker                   │
            └──────────────────────────────────────┘
```

### 4.2 与ISC规则体系的对接

**核心思路：PM的流程卡点 = ISC规则的"执行者"**

当前ISC有188条规则，其中相当一部分涉及流程控制（如ISC-COMPLETION-HANDLER-001、ISC-AUTO-QA-001）。这些规则目前由main或脚本被动执行。PM引入后：

| 对接方式 | 说明 |
|----------|------|
| **PM作为规则执行者** | ISC规则中涉及流程控制的，handler改为"通知PM"而非"echo给main" |
| **PM作为规则提案者** | PM在流程中发现反复出现的问题，可以提议新的ISC规则 |
| **PM遵守ISC规则** | PM自身也受ISC规则约束，如角色分离（PM不能审查PM自己的决策） |

**需要新增/修改的ISC规则**：

| 规则ID | 内容 | 动作 |
|--------|------|------|
| ISC-PM-GATE-001 | 阶段转换必须经PM门禁检查 | 新增 |
| ISC-PM-DISPATCH-001 | 复合任务（≥3子任务）必须经PM编排后再dispatch | 新增 |
| ISC-PM-ESCALATION-001 | PM检测到高风险必须在5分钟内升级到main | 新增 |
| ISC-COMPLETION-HANDLER-001 | completion事件处理 | 修改：增加PM通知环节 |
| ISC-AUTO-QA-001 | 自动质量核查 | 修改：由PM触发而非echo提示 |

### 4.3 与task-board的集成

**核心思路：PM是task-board的"操盘手"，dispatch引擎是"执行引擎"**

```
当前：main → dispatch引擎 → 子Agent
引入PM后：main → PM → dispatch引擎 → 子Agent
```

| 能力 | 当前owner | PM引入后 |
|------|-----------|----------|
| 任务入队（enqueue） | main直接调用 | PM调用（main通过PM间接入队） |
| 状态更新（markDone等） | completion-handler脚本 | 不变（脚本执行后通知PM） |
| 队列优先级管理 | 无人管 | PM负责 |
| slot利用率监控 | completion-handler提示 | PM主动监控+自动回填 |
| 批次管理 | main手动分批 | PM自动分批 |
| 历史查询 | main按需查 | PM自动分析趋势 |

**关键变化**：
- dispatch引擎的API不变，PM通过API操作
- main不再直接enqueue，而是给PM一个"项目目标"，PM自己拆解后enqueue
- 简单的单任务请求可以bypass PM（main直接dispatch），复合任务必须经PM

### 4.4 与completion-handler的配合

**核心思路：completion事件是PM的"神经信号"**

```
当前流程：
  子Agent完成 → completion-handler.sh → 更新task-board + echo提示 → main看到提示

PM引入后：
  子Agent完成 → completion-handler.sh → 更新task-board → 通知PM
  PM收到通知 → 检查质量 → 决定下一步 → 自动派发/升级/汇报
```

**具体对接**：
1. completion-handler.sh执行完成后，发射事件 `pm.task.completed`
2. PM的事件监听器接收此事件
3. PM执行后续逻辑：
   - 更新project-plan中的任务状态
   - 检查是否需要质量核查（L1/L2/L3）
   - 检查是否有下游依赖任务可以启动
   - 检查当前milestone是否完成
   - 检查是否需要阶段转换
4. PM根据判断结果自动行动（派发新任务/升级/汇报）

**关键优势**：main不再需要处理"子任务完成了然后呢"的问题，这全部由PM接管。

### 4.5 与PDCA引擎的配合

**核心思路：PM管Plan+Do调度，PDCA Check由analyst独立执行，Act由PM驱动改进**

```
PDCA当前流程：
  init → plan → do → check → act → done
  (全部由main驱动)

PM引入后的分工：
  Plan: PM负责（任务拆解、WBS、资源分配）
  Do:   PM调度（通过dispatch引擎派发子Agent执行）
  Check: PM触发（但执行者必须是analyst，角色分离不变）
  Act:   PM负责改进闭环（收到check结果后决定重做/修改/通过）
```

| PDCA阶段 | 负责方 | PM角色 |
|----------|--------|--------|
| Plan准入门禁 | PDCA引擎 | PM确保任务来源合法、4要素齐全 |
| Plan准出门禁 | PDCA引擎 | PM确保计划有目标/时限/成本/验收标准 |
| Do准入门禁 | PDCA引擎 | PM确保executor已分配、evaluator已预分配 |
| Do准出门禁 | PDCA引擎 | PM确保产出物已交付 |
| Do→Check转换 | PDCA状态机 | PM触发转换（状态机自动检查角色分离） |
| Check执行 | analyst | PM不介入check具体执行（保持独立性） |
| Check→Act/Done | PDCA状态机 | PM根据check结果决定act还是done |
| Act执行 | PM | PM驱动改进（修改计划/重派任务/调整方案） |

**PDCA铁律不变**：
- ISC-EVAL-ROLE-SEPARATION-001 继续由状态机硬编码执行
- check阶段evaluator只能是analyst（铁令不变）
- PM不能充当check的evaluator（PM是流程角色，不是评审角色）

## 5. 决策权限矩阵

### 5.1 三色分级

| 等级 | 含义 | 响应方式 |
|------|------|----------|
| 🟢 绿色 | PM自主决策 | 直接执行，事后在project-log中记录 |
| 🟡 黄色 | PM决策+通知 | 执行后通知main，main可事后否决 |
| 🔴 红色 | 必须升级 | 暂停执行，等待main或用户决策 |

### 5.2 详细决策矩阵

#### 🟢 PM自主决策（不需任何人批准）

| 场景 | PM决策内容 | 自主依据 |
|------|-----------|----------|
| 任务分配 | 给某子任务选择执行Agent（如coder vs coder-02） | 按负载均衡和历史成功率 |
| 超时重试 | 子任务超时→标记失败→enqueue重试 | retry_count < 3 |
| 空闲回填 | 空slot→从queue取任务派发 | dispatch引擎标准逻辑 |
| L1/L2质量检查 | 格式检查、基本内容检查 | 标准checklist |
| 进度汇报 | 选择汇报时机和内容 | 阶段转换/里程碑完成时 |
| 优先级微调 | 非关键任务的优先级调整 | 不影响关键路径 |
| 项目目录创建 | 创建project-<id>/目录和标准文件结构 | 标准模板 |

#### 🟡 PM决策+通知主Agent（事后知会）

| 场景 | PM决策内容 | 通知原因 |
|------|-----------|----------|
| Agent角色替换 | 某Agent连续失败→换同角色另一实例 | main应知道人员变动 |
| 任务重排序 | 调整子任务执行顺序 | 可能影响交付时间线 |
| 方案微调 | 在不改变目标的前提下调整实施细节 | main应知道方案变化 |
| 批次拆分 | 将大批次拆为小批次以提高并行度 | 资源利用方式变化 |
| L3质量核查派发 | 派analyst做深度审查 | 占用一个Agent slot |
| 非关键任务取消 | 发现某任务已无必要→取消 | 减少了工作范围 |

#### 🔴 必须升级到主Agent或用户

| 场景 | 升级到谁 | 升级原因 |
|------|----------|----------|
| **方向性决策** | main或用户 | "应该用方案A还是方案B"涉及战略判断 |
| **需求不明确** | 用户 | 用户意图存在多种理解可能 |
| **资源严重不足** | main | 19 slot不够用，需要决定优先级 |
| **连续失败（>3次重试）** | main | 可能是方案层面的问题 |
| **质量检查连续不通过** | main | 可能需要重新设计方案 |
| **新增大范围任务** | main+用户 | 超出原始需求范围 |
| **用户满意度确认** | 用户 | 最终交付需要用户确认 |
| **项目取消/暂停** | main | 影响整体资源规划 |

### 5.3 升级协议

PM升级到main时，发送结构化消息：

```
🔴 [PM升级] 需要决策
━━━━━━━━━━━━━━━━━━━
📌 项目: proj_20260310_xxx
📌 阶段: 并行实施

📌 已尝试: 重试2次，换researcher-02 1次
📌 选项:
  A) 由main重新解释需求给researcher
  B) 降低需求文档的质量标准
  C) 跳过需求阶段，直接基于用户原始描述执行
📌 PM建议: 选项A（原始需求可能不够清晰）
━━━━━━━━━━━━━━━━━━━
请决策（回复A/B/C或自定义）

**关键设计决策**：

- PM禁止直接write/edit（PM是管理者，不是执行者；需要写文件时派子Agent）
- PM可以直接派发子Agent（不需要通过main中转）

### 6.2 PM的工作空间布局

```
workspace/
├── projects/                          # PM管理的项目空间
│   └── proj_<id>/
│       ├── project-plan.json         # 项目计划（PM维护）

### 6.3 专属技能（Skill）

PM Agent需要以下专属技能：

| 技能名 | 功能 | 优先级 |
|--------|------|--------|
| **project-planning** | 将用户意图拆解为项目计划（WBS、里程碑、子任务） | P0 必须 |
| **stage-gate** | 阶段门禁检查（五阶段的准入/准出条件检查） | P0 必须 |
| **progress-tracker** | 基于task-board数据生成进度报告 | P0 必须 |
| **risk-radar** | 自动扫描风险（超时、阻塞、失败率） | P1 重要 |
| **escalation-protocol** | 决策分流和升级协议（结构化升级消息） | P1 重要 |
| **deliverable-registry** | 产出物登记和检查 | P2 一般 |

**每个技能的核心逻辑**：

**project-planning** SKILL.md 核心指令：
```
你是PM Agent的任务拆解引擎。
输入：用户意图（自然语言）
输出：project-plan.json
规则：
1. 每个子任务必须有明确的agent角色、预估时间、验收标准
2. 依赖关系必须显式声明
3. 关键路径必须标注
4. 如果无法拆解（需求不明确），输出"需要升级"信号
```

**stage-gate** SKILL.md 核心指令：
```
你是PM Agent的阶段门禁。
输入：当前阶段、已有交付物、项目计划
输出：通过/不通过 + 具体缺陷列表
规则：
1. L1检查（格式）：PM自己秒级完成
2. L2检查（内容）：PM通过prompt自查
3. L3检查（深度）：派analyst执行
4. 不通过时，给出具体的修改建议
```

### 6.4 需要的ISC规则

| 规则ID | 名称 | 核心内容 |
|--------|------|----------|
| ISC-PM-GATE-001 | PM阶段门禁强制 | 项目阶段转换必须经PM门禁检查，不得跳过 |
| ISC-PM-DISPATCH-001 | 复合任务PM编排 | ≥3子任务的复合任务必须经PM拆解编排后dispatch |
| ISC-PM-ESCALATION-001 | PM升级时效 | PM检测到🔴级风险后5分钟内必须升级 |
| ISC-PM-PROGRESS-001 | PM主动汇报 | 阶段转换和里程碑完成时PM必须汇报 |
| ISC-PM-QUALITY-001 | PM质量核查触发 | coder/writer产出完成后PM必须触发质量核查（替代当前的echo提示） |
| ISC-PM-SEPARATION-001 | PM角色分离 | PM不能做check的evaluator，不能做具体编码/写作 |

### 6.5 启动/触发机制

**什么时候PM介入？**

```
触发条件一：main显式委托
  main收到用户请求 → 判断是复合任务 → 创建PM子Agent → 委托项目

触发条件二：completion事件驱动
  子Agent完成 → completion-handler → 事件通知PM → PM处理后续

触发条件三：定时巡检
  每5分钟（或可配置间隔）→ PM检查task-board → 风险扫描

触发条件四：阈值触发
  连续3次任务失败 → 自动唤醒PM介入
  queue堆积超过N → 自动唤醒PM优化调度
```

**PM的两种运行模式**：

| 模式 | 场景 | 生命周期 |
|------|------|----------|
| **项目模式** | main委托一个完整项目给PM | 项目完成后结束 |
| **守护模式** | PM作为常驻进程监控task-board | 持续运行，定时巡检 |

**建议先实现项目模式**，守护模式可以后续通过cron-worker + PM skill的方式模拟。

### 6.6 PM与main的交互协议

**main → PM（委托）**：
```
任务类型: project
用户原始意图: "..."
约束条件: [时间限制、质量要求、范围限制]
授权级别: standard（标准授权，PM可自主调度子Agent）
```

**PM → main（汇报/升级）**：
```
类型: progress | escalation | completion
项目ID: proj_xxx
内容: [进度摘要] | [升级请求+选项] | [交付结果]
```

**PM → main（交付）**：
```
类型: delivery
项目ID: proj_xxx
交付物清单: [文件路径列表]
质量报告: [所有check结果]
未解决风险: [如有]
建议: [交付给用户 / 需要补充]
```

### 6.7 渐进式实施路线

**Phase 1（立即可做）：PM作为skill，不是独立Agent**
- 在main的技能集中增加 `pm-planning` 和 `pm-gate` 技能
- main在处理复合任务时调用这些技能
- 不增加新Agent，降低系统复杂度

**Phase 2（验证后）：PM作为独立Agent**
- 新增pm Agent配置
- main将复合任务委托给PM
- PM获得子Agent派发权限

**Phase 3（成熟后）：PM守护模式**
- PM常驻监控task-board
- completion事件直接路由到PM
- main只在DCP时介入

**建议从Phase 1开始**，用2-3个真实项目验证PM的流程和决策逻辑，确认有效后再升级为独立Agent。

## 7. 风险与待确认

### 7.1 技术风险

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| **PM成为瓶颈** | 🟡 中 | PM的决策应该秒级完成；如果PM处理慢，直通main作为fallback |
| **PM决策质量** | 🟡 中 | PM的决策通过PDCA Check审计；积累badcase后迭代改进 |
| **PM与main的冲突** | 🟢 低 | 通过决策矩阵清晰划界；争议时main有最终决策权 |
| **增加系统复杂度** | 🟡 中 | 渐进式实施（Phase 1 → 2 → 3），每步验证后再进 |
| **completion事件路由变更** | 🟢 低 | completion-handler增加PM通知不影响现有逻辑 |

### 7.2 待确认事项

| 序号 | 待确认内容 | 影响 | 需要谁确认 |

| 2 | PM的model选择：thinking vs non-thinking？ | 响应速度 vs 决策质量 | 用户 |
| 3 | 简单任务（单子任务）是否bypass PM？ | 效率 vs 一致性 | main |
| 4 | PM失败/卡死时的fallback机制？ | 系统可靠性 | 架构决策 |
| 5 | PM的decision-log是否纳入CRAS认知层分析？ | 长期改进 | 架构决策 |
| 6 | Phase 1 用哪几个真实场景验证？ | 验证覆盖度 | 用户 |

## 你是谁
你是项目经理（PM），负责将战略意图转化为可执行的项目计划，并确保执行过程的质量和效率。
