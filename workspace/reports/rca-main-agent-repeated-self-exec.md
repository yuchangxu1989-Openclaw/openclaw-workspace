## 5. 程序化防护增强方案

### 5.1 方案概述：exec调用计数器 + 滑动窗口守卫

**核心思路**：不完全禁止exec（主Agent需要1-3行快速验证命令），而是对**连续exec调用**设置阈值告警和阻断。

### 5.2 具体方案

#### 层级一：exec调用计数器（软性守卫）

```
规则：在同一个Turn（用户消息→Agent回复）中：
- exec调用 ≤ 2次：✅ 允许（快速验证/状态检查）
- exec调用 = 3次：⚠️ 警告（在Agent上下文中注入提醒）
- exec调用 ≥ 4次：🛑 阻断（拒绝执行，提示"请委派子Agent"）
```

**实现方式**：
- 在OpenClaw框架层面（非MD规则），对主Agent的exec工具调用进行计数
- 达到阈值时，exec工具返回错误信息而非执行命令
- 错误信息格式：`[ISC-DELEGATION-GUARD] 本Turn已执行${count}次exec，超过阈值(3)。请spawn子Agent处理此任务。`

#### 层级二：exec命令模式匹配（语义守卫）

```
规则：以下exec命令模式视为"实现性工作"，即使单次也应拦截：
- sed -i / sed 's/.../.../g' （原地修改文件）
- awk '{...}' > file （生成/修改文件）
- python/node -c '...' > file （脚本生成文件）
- tee / cat > / echo > （写入文件）

以下exec命令模式视为"验证/查看"，允许：
- grep / rg / find （只读搜索）
- cat / head / tail / wc （只读查看）
- ls / stat / file / du （文件信息）
- git status / git log / git diff （状态查看）
```

#### 层级三：回顾性审计（事后守卫）

```
规则：每次主Agent回复用户后，审计本Turn的工具调用：
- 统计 exec 调用次数
- 检查是否有"修改型"exec命令（sed -i、tee、>重定向等）
- 生成审计记录写入日志
- 若违规，自动入库为badcase
```
