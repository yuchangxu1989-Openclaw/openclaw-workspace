# PB-010 子场景补债落地说明（2026-03-08）

## 目标

围绕 PB-010 直接补齐 4 个闭环子场景，覆盖：

- PB-010A：RCA
- PB-010B：gap
- PB-010C：fix
- PB-010D：dispatch success

并且做到：

- 能写进测试集的直接写进测试集
- 至少补最小 regression
- 给出 RCA / gap / fix / dispatch success 四段闭环的落地证据

---

## RCA

### 现状根因

仓库里原有 capability regression 只有 4 个基础样例：

- `p2e-ext-001` intent_expansion
- `p2e-ext-002` event_completion
- `p2e-ext-003` task_expansion
- `p2e-ext-004` directive skip

它们能证明基础能力存在，但**不能证明 PB-010 这类“从发现问题 → 补全缺口 → 修复 → 派发成功收口”的完整闭环子场景**已经被覆盖。缺口主要有 4 个：

1. **RCA 缺失**：原 intent_expansion 只有监控条件 + 通知动作，没有显式要求“根因定位动作”。
2. **gap 缺失**：原 event_completion 只要求补 `source_event` / `repo`，没有要求补 `gap_type` / `impacted_scope`。
3. **fix 闭环不足**：原 task_expansion 虽然有 inventory/remediation/verification，但没有围绕具体故障闭环表达“修复 + 回归”。
4. **dispatch success 未收口**：原 skip case 是普通 directive，不足以证明“已派发成功的指令不应再误触发 task expansion”。

### 对应 gap

PB-010 的债务本质上是：

- 评测集对“闭环型故障处理链路”覆盖不够
- regression 没有锁住 RCA / gap / fix / dispatch success 这 4 段收口行为
- pass runtime 也没有给出足够的 capability 输出样本，导致新 case 即使加进去也无法跑通

---

## Fix

### 1) 测试集新增 PB-010A/B/C/D 四个子场景

已写入：`principle-e2e-spec/08-capability-test-cases.json`

新增 case：

- `pb-010a`：RCA 场景，要求 `intent_expansion` 必须同时包含
  - `monitoring_condition`
  - `diagnosis_action`
  - `notification_action`
- `pb-010b`：gap 场景，要求 `event_completion` 必须补出
  - `source_event`
  - `gap_type`
  - `impacted_scope`
- `pb-010c`：fix 场景，要求 `task_expansion` 至少包含
  - `inventory`
  - `remediation`
  - `verification`
  且内容上要出现“修复”“回归”
- `pb-010d`：dispatch success 场景，要求已派发成功的 directive 维持 `SKIP`，不误触发扩列

### 2) 最小 regression 直接补进回归脚本

已更新：`principle-e2e-spec/scripts/test_capability_regression.py`

新增断言：

- `pb-010a` 必须出现 `diagnosis_action`
- `pb-010b` 必须补出 `gap_type=missed_regression_window`
- `pb-010b` 必须补出 `impacted_scope=payment-e2e@staging`
- `pb-010c` 的 task 内容必须同时包含“修复”和“回归”
- `pb-010d` 必须返回 `SKIP`

### 3) pass runtime 补齐样例输出，保证 regression 可执行

已更新：`principle-e2e-spec/examples/capability_runtime_pass.json`

补齐内容：

- `intent_expansion` 增加 `diagnosis_action`
- `event_completion` 增加 `gap_type` / `impacted_scope`
- `task_expansion` 的 verification 标题显式带“回归”
- 增加 `dataset_weighting.prioritized_dimensions`
  - `execution_chain_completeness`
  - `timely_task_fanout`

### 4) 兼容现有 runner 的高权重数据集校验

在落地过程中发现当前 `benchmark_runner.py` 已经不是早前 1.2.0 版本，而是带有 `DATASET-WEIGHTING-HARDENING` 的 1.3.0 版本。

这意味着只补 capability outputs 还不够，runtime 还必须声明：

- `dataset_weighting.prioritized_dimensions`

否则即便 capability stage 通过，整体 verdict 仍会因为高权重维度校验失败而 fail-closed。

因此这次补债同时把 pass runtime 一并补齐，避免新增 regression 落地后“样例本身跑不通”。

---

## Dispatch success 闭环说明

PB-010D 的意义不是再测一次普通 directive，而是锁住以下行为：

- 当 `dispatch_status=success` 时
- 指令已经完成派发收口
- 系统只需要确认/记录，不应再次把它扩成 inventory/remediation/verification 多任务链路

这保证了 PB-010 四段闭环的最后一段“dispatch success”有单独的 regression guard，而不是被普通 skip case 模糊覆盖。
