## 四、快路（Inline Hook）失效分析

### 现状
- 快路代码已完整实现（`intent-extractor-inline.js`，具备 `patchMessageHook()` 接口）
- 实际接线状态：**完全未接入消息处理管道**
- 证据：事件总线中仅有 **1条** `intent.inline.*` 事件，且来源于 git commit 事件，不是消息事件
- `patchMessageHook()` 函数从未在任何运行时路径中被调用

### 影响
快路原本设计用于实时捕获 QUERY / DIRECTIVE / FEEDBACK 意图（用户说话即触发）。快路缺失导致：
- **所有实时 QUERY 意图丢失**（用户问完即走，不留记忆痕迹）
- **DIRECTIVE 召回率减半**（只靠慢路从 memory 摘要中反向推断）
- **FEEDBACK 严重漏检**（批评/确认往往是短句，不会单独出现在 memory 日志）

## 五、主要问题汇总

### P0: QUERY 召回率 = 0%（根因：数据源结构性偏置）

慢路扫描器读取的是 **Agent 写的日志摘要**，不是用户原始消息。Memory 文件记录的是"发生了什么"，不是"用户问了什么"。查询意图在回答后即消解，几乎不留文字痕迹：
- ❌ memory 写法：`"API探测全healthy — 19 provider→3端点去重"` 
- ✅ 用户原始意图：`"帮我跑一下 API 探测，看看哪些还活着"`

这是数据源设计的根本缺陷。单靠慢路无法修复，必须快路覆盖。

### P1: Inline Hook 接线缺失（根因：代码存在但集成未完成）

`patchMessageHook()` 未在 gateway 启动或消息中间件中调用。实施路径清晰，修复成本低。

### P2: 慢路重复计算（根因：无跨轮次去重）

相同内容在不同 cron 轮次被多次扫描提取（同一段 memory 产出 3-4 条相同意图事件）。慢路有游标机制但缺少跨事件的 fingerprint 去重，导致约 **20%** 的事件是冗余重复。

### P3: Agent 行为被误识为用户意图（根因：扫描对象混入 Agent 输出）

Memory 文件同时包含 **用户意图** 和 **Agent 执行报告**（如"执行的变更: 1.删除cherryin..."）。LLM 无法区分两者，将 Agent 的行动笔记误识为用户 DIRECTIVE。

### P4: FEEDBACK → RULEIFY 过度合并（根因：Prompt 对纠偏场景的处理）

用户批评/纠正 AI 错误时，往往同时隐含规则化意图（"不应该这样 → 以后这样做"）。系统倾向只提取 RULEIFY，丢失了 FEEDBACK 层。3 条 FEEDBACK 中至少有 2 条对应内容能识别到更多 FEEDBACK 信号，但被同段 RULEIFY 掩盖。

## 六、改进点（按优先级）

### 🔴 P0-1: 立即接入 Inline Hook

**具体做法**：在 OpenClaw gateway 的消息处理中间件中调用 `patchMessageHook(messageHookInstance)`。

```js
// 在 gateway 启动时，消息 Hook 初始化之后加一行：
const { patchMessageHook } = require('./skills/cras/intent-extractor-inline');
patchMessageHook(messageHookInstance);
```

**预期效果**：QUERY 召回率从 0% 提升到 40-60%，FEEDBACK 改善明显。

---

### 🔴 P0-2: 慢路数据源扩展为"原始会话日志"

**具体做法**：除 memory/*.md 外，增加扫描 `infrastructure/logs/` 或 OpenClaw session JSONL 原始对话记录，提取用户发言的原文（非 Agent 摘要）。

**预期效果**：QUERY/DIRECTIVE 召回率全面提升，消除数据源偏置。

---

### 🟡 P1: 慢路跨轮次去重

**具体做法**：在 `intent-extractor.js` 的 `emit` 前，加 fingerprint 检查（类似快路的 `intentFingerprint()`，窗口调整为 24h）。

```js
// 在 emit 前增加：
const fp = intentFingerprint(intent.type, intent.target, intent.summary);
if (isDuplicate(fp)) continue;
markEmitted(fp);
```

**预期效果**：去除约 20% 重复事件，事件总线质量明显提升。

---

### 🟡 P2: Memory 文件分层扫描（用户 vs. Agent 内容）

**具体做法**：在扫描前按语义标记区分用户意图段落和 Agent 行动日志。可以用简单启发式：以"Agent："/"执行的变更"/"完成项"为 Agent 内容标记，仅对用户发言或双方对话段落提取意图。

**预期效果**：False Positive 减少（#30、#44 类误判消除），Precision 从 ~80% 提升到 90%+。

---

### 🟢 P3: FEEDBACK 独立提取门禁

**具体做法**：在 Prompt 中添加规则："若用户明确批评/纠正了某个行为，**优先**输出 FEEDBACK，即使同段话也包含规则化暗示"。

```
优先级规则：用户纠偏 > RULEIFY；只有当纠偏的改进方向被明确表述时，才同时输出 RULEIFY。
```

**预期效果**：FEEDBACK 召回率从 ~20% 提升到 50%+。

---

## 七、结论

| 维度 | 当前状态 | 评级 |
|------|---------|------|
| RULEIFY 提取质量 | Precision 90%，Recall 75% | ✅ 基本可用 |
| REFLECT 提取质量 | Precision 91%，Recall 80% | ✅ 基本可用 |
| DIRECTIVE 提取质量 | Precision 80%，Recall 40% | ⚠️ 需改善 |
| FEEDBACK 提取质量 | Precision 100%，Recall 20% | ❌ 严重偏低 |
| QUERY 提取质量 | Precision N/A，Recall 0% | ❌ 完全失效 |
| 快路 Inline Hook | 代码已有，运行时未接入 | ❌ 实质死路 |
| 慢路去重 | 无跨轮次去重，约20%冗余 | ⚠️ 待修复 |
| 数据源偏置 | Memory 摘要偏向RULEIFY/REFLECT | ❌ 结构性缺陷 |

**核心结论**：意图提取模块已 **工程就绪**（代码质量、分类逻辑均可），但 **线上效果接近失效**，因为它扫描的是 Agent 产出的摘要文本而非用户原始消息，且快路未接线。这不是 LLM 分类能力问题，而是数据接入和集成问题。

最关键的一步：**在 5 行代码内接入 Inline Hook**，即可让系统从"模块就绪但无效"升级到"真正线上运行"。
