# 超时根因分析报告（2026-03-09）

## 一句话结论
两个子Agent超时的核心共因是：**任务单位过大 + 默认超时配置偏短 + 外部LLM调用开销高（尤其GLM-5长prompt）**，导致在单次spawn窗口内无法稳定收敛。

---

## 任务1：e2e-eval-glm5-batch1（scout）

### 1) 事实证据
- 评测结果目录存在：`/root/.openclaw/workspace/reports/e2e-eval-results`
- 目录下有多份GLM-5相关结果，如：
  - `e2e-l1-glm-5-2026-03-09-1845.json`
  - `e2e-l1-glm-5-2026-03-09-1850.json`
  - `e2e-l1l2-glm-5-2026-03-09-1854.json`
- 说明评测链路可运行，但该批任务（10条）在给定窗口内未完成回传。
- 已知线索：单次API请求约57k tokens。

### 2) 执行脚本与模式判断
- 当前工作区可直接定位到结果文件，但未在本次扫描中锁定唯一“入口eval脚本”（可能由上层调度动态拼接命令或在其他目录/会话内执行）。
- 从现象与产物节奏判断，高概率是**按case串行执行**（至少不是高并发并行），并伴随模型调用/评审调用的串接。
- 在仓库中可见GLM调用代理脚本 `scripts/glm5-proxy.cjs`，其中明确有：
  - 重试机制：`maxRetries` + 指数退避
  - 退避延迟：初始2s（指数增长）
  - 这会在API不稳定或限流时放大总时长

### 3) 57k tokens/条是否合理
- 对“10条评测用例”场景而言，**57k/条明显偏重**，除非每条都携带超长上下文（完整规则集+历史对话+多段示例+长输出评审）。
- 风险：
  1. 单请求时延显著升高（首字节和总完成时间都拉长）
  2. 更易触发provider侧速率/吞吐限制
  3. 失败后重试成本极高
- 结论：存在明显冗余概率，建议做prompt预算与分层裁剪。

### 4) GLM-5 rate limit
- 本次仓内未发现明确“GLM-5官方限流阈值”文档常量（RPM/TPM固定值）。
- 但代理脚本已实现重试+指数退避，侧面说明调用端**预期会遭遇限流/瞬时失败**。
- 因此根因判断应按“存在限流/抖动风险”处理，而不是按“无限吞吐”假设。

### 5) 根因归纳（任务1）
1. **主因A：单case token 体量过大（57k）**，导致单次调用时延和失败成本过高。
2. **主因B：10条批任务在一个spawn窗口内执行，任务粒度过粗**。
3. **主因C：重试退避在高负载下叠加放大总耗时**。
4. **次因D：可能为串行链路（生成+评审）导致总时长线性累加**。

### 6) 修复建议（任务1）
- **拆批执行**：10条改为2~5条/批，批次并行，单批超时可控。
- **token预算硬阈值**：
  - 输入预算（如<=12k或<=16k）
  - 超预算自动压缩：规则摘要化、历史裁剪、示例去重
- **双阶段评测**：先低成本筛查，再对疑难case做重评。
- **失败快速降级**：遇到429/超时，先降上下文再重试，避免原样重放57k。
- **观测字段落盘**：每条case记录 `input_tokens/output_tokens/retry_count/duration_ms`。

## 任务2：delegation-fix-cli（coder）

### 1) 事实证据
- 相关脚本已存在：`/root/.openclaw/workspace/scripts/safe-interrupt.sh`
- 该任务描述是“排查根因 + 开发CLI（safe-interrupt）”，用户反馈“我塞了3件事”。
- 典型属于**分析+设计+实现**混合单任务。

### 2) 任务描述是否过大
- 结论：**是**。
- 单spawn里同时做：
  1. 根因排查
  2. 方案设计/接口定义
  3. CLI实现与验证
- 这会引发上下文膨胀、执行链拉长、返工概率升高，超时风险明显增加。

### 3) sessions_spawn 默认timeout
- 从工具签名可见 `sessions_spawn` 有 `timeoutSeconds` / `runTimeoutSeconds` 参数。
- 但本仓内文档未检索到一条“统一默认值（明确数值）”的权威写法。
- 因而可得结论：**若调用时未显式传timeout，易受平台默认值约束并触发超时；默认值需以运行时平台文档/配置为准**。

### 4) openclaw timeout配置方式（仓内可确认部分）
- 通过工具定义可确认：
  - 一次性运行可用 `runTimeoutSeconds`
  - 会话型可用 `timeoutSeconds`
- 实操建议：对重任务必须显式传递，不依赖默认值。

### 5) 根因归纳（任务2）
1. **主因A：任务打包过大（多目标混装）**。
2. **主因B：未显式设置足够timeout（或默认值不足）**。
3. **次因C：实现型任务如果包含环境探测/回归验证，时长不确定性更高**。

### 6) 修复建议（任务2）
- **任务拆分为3个spawn**：
  - T1: 仅根因排查（产出报告）
  - T2: 仅CLI实现（产出脚本+README）
  - T3: 仅验证与回归（产出测试记录）
- **显式timeout策略**：每个spawn按任务类型设置不同timeout（分析短、实现中、回归长）。
- **中间里程碑落盘**：每阶段写`reports/`，避免“全做完才回传”。

## 横向防护机制（两任务通用）

1. **Spawn前预算门禁（强制）**
   - 估算 token、子步骤数、外部API次数
   - 超阈值自动拆任务，不允许“巨任务直接发车”

2. **超时策略标准化**
   - 禁止依赖默认timeout
   - `sessions_spawn` 调用模板强制包含 `timeoutSeconds/runTimeoutSeconds`

3. **大prompt防爆机制**
   - 输入超过阈值自动摘要/裁剪
   - 重试前先降载，不允许原样高token重试

4. **可观测性最小集合**
   - 每任务必须记录：开始时间、结束时间、token、重试次数、失败码、阶段进度
   - 超时后可立即定位是“模型慢、限流、还是流程过长”

5. **任务原子化原则**
   - 一次spawn只承载一种主要产出类型（分析/实现/验证三选一）
   - 混合任务必须先拆分再调度
