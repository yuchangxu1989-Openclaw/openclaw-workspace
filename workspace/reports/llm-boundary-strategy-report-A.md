# LLM能力边界下的AI Agent趋势推演与打法策略

> 报告日期：2026-03-08 | 视角：系统架构师 / 战略分析  
> 核心命题：LLM"裸奔"可靠区间有限，如何在此约束下构建全球最强AI Agent系统？

---

## 执行摘要

**核心结论：LLM的能力边界不是bug，是物理定律级别的约束。赢家不是"模型最强的"，而是"外骨骼最好的"。**

1. 所有大厂（Apple/华为/三星/小米/Google）都撞上了同一面墙：LLM在多步骤、长记忆、高可靠场景下不可信赖。它们的解法趋同——用程序化/系统化补偿LLM的不可靠性。
2. 未来2-3年，端云协同 + 程序化外骨骼 + 垂直场景深耕将是消费电子AI Agent的主线。通用Agent仍是demo级别。
3. 开源记忆方案（mem0/Memori/Zep等）解决了"记住"问题，但没解决"可靠执行"问题——这正是最大的战略空白。
4. **我们的打法：不要试图让LLM变强，而是让外骨骼变强。LLM是肌肉，我们造骨架。**

---

## 一、大厂AI战局：它们撞上了同一面墙

### 1.1 各家现状与能力边界

| 厂商 | AI品牌 | 端侧模型 | 核心场景 | 遇到的边界问题 |
|------|--------|----------|---------|---------------|
| Apple | Apple Intelligence | ~3B参数端侧 + Private Cloud Compute | 文本改写、摘要、Siri增强、Image Playground | Siri仍无法可靠完成跨App多步操作；2024 WWDC承诺的"屏幕感知"延迟交付 |
| Google | Gemini Nano/Pro | Gemini Nano 端侧 + 云端Gemini | 通话摘要、智能回复、Circle to Search | Gemini在多步Agent任务中可靠性不足，Project Astra仍为受控demo |
| 三星 | Galaxy AI | 端侧+云端混合（与Google合作） | 实时翻译、笔记AI、图片编辑 | 功能点状分布，无统一Agent体验；跨场景记忆缺失 |
| 华为 | 小艺/盘古 | 端侧模型 + 云端盘古 | 语音助手、智能家居控制、出行规划 | 鸿蒙生态闭环内可控，但开放场景Agent能力有限 |
| 小米 | 小爱同学/MiLM | 端侧+云端 | 智能家居、日程管理 | 硬件生态强但AI深度不足，多轮交互可靠性差 |

**关键洞察：每家都在做"AI功能点"，没有一家做到了"可靠的多步Agent"。** 原因一致——LLM裸奔的可靠区间限制。

### 1.2 它们的边界问题和我们一样吗？

**完全一样，甚至更严重。** 理由：

- **我们的实测数据**：≤4步执行链 × 同一Turn × ≤3条并行 × ≤5条规则 → 可靠区间
- **大厂的处境**：手机Agent需要跨App操作（打车→发消息→设日历），轻松突破4步；用户期望跨Session连续记忆，远超单Turn
- **差异在于**：大厂有OS级别的API控制权（可以用确定性API替代LLM猜测），这是它们的"天然外骨骼"。但即便如此，Apple Siri的多步骤可靠性在2025年仍被用户广泛吐槽（Siri reliability meme长盛不衰）

**核心不等式**：
```
用户期望的Agent复杂度 >> LLM裸奔可靠区间
```
这个gap只能靠**程序化外骨骼**来填。

## 二、趋势推演：未来2-3年的三条主线

### 2.1 端侧 vs 云侧：走向"端云分工"而非"端云替代"

**结论：纯端侧不够用，纯云侧不够快。最终形态是分层架构。**

| 层级 | 部署位置 | 职责 | 模型规模 | 延迟要求 |
|------|---------|------|---------|---------|
| L0 感知层 | 端侧 | 意图识别、简单分类、隐私过滤 | 1-3B | <100ms |
| L1 执行层 | 端侧+边缘 | 单步操作执行、UI自动化 | 3-7B | <500ms |
| L2 规划层 | 云侧 | 多步规划、复杂推理、创意生成 | 70B+ | <3s |
| L3 记忆层 | 云侧（加密） | 长期用户画像、跨设备同步 | 向量DB+结构化 | 异步 |

**数据支撑**：
- Qualcomm Snapdragon 8 Gen 4（2025）支持端侧运行~13B模型，INT4量化下推理速度约30 tok/s
- Apple M4芯片Neural Engine 38 TOPS，可跑~3B模型流畅推理
- 但端侧模型在复杂推理任务上准确率比云端70B+模型低40-60%（Stanford HELM Benchmark, 2024）

### 2.2 记忆 vs 程序化：两者不是对立，而是互补

**结论：记忆解决"知道什么"，程序化解决"做到什么"。大多数团队过度投资记忆，忽视程序化。**

```
                    记忆系统                    程序化系统
用途              存储用户偏好/历史            强制执行规则/流程
解决的问题        "你上次说喜欢川菜"          "先查库存→再下单→最后通知"
LLM依赖度         中（检索+理解）              低（LLM只在决策点介入）
可靠性            中（检索可能miss）            高（确定性流程）
用户感知          "它懂我"                     "它靠谱"
```

**战略推论**：用户对"靠谱"的感知价值 > "懂我"。一个记住你名字但订错票的Agent不如一个不记得你名字但每次都订对票的Agent。

### 2.3 通用 vs 垂直：2-3年内垂直碾压通用

**结论：通用Agent是终极愿景，但2-3年内，垂直Agent在用户体验上会碾压通用Agent。**

理由：
- **可靠性数学**：4步可靠区间内，垂直场景（如"帮我订机票"）的步骤是固定的、可程序化的；通用场景（如"帮我规划旅行"）步骤数不确定
- **Apple的教训**：Shortcuts（垂直、程序化）比Siri（通用、LLM驱动）可靠10倍以上
- **华为的经验**：鸿蒙"服务卡片"（垂直场景预定义流程）用户满意度远高于小艺开放对话

**预测时间线**：
- **2025-2026**：垂直Agent + 程序化流程为主，"通用Agent"仍是噱头
- **2026-2027**：模型能力提升（可靠区间从4步→6-8步），半通用Agent开始可用
- **2027-2028**：端云协同成熟，通用Agent在受控生态内（如Apple全家桶）初步可用

## 三、开源记忆方案对比：解决了什么，没解决什么

### 3.1 主要方案概览

| 方案 | 核心机制 | 解决的问题 | 未解决的问题 |
|------|---------|-----------|-------------|
| **mem0** | 自动提取+存储用户偏好/事实到向量DB；支持用户/Session/Agent三级记忆 | 跨Session记忆持久化；自动去重更新 | ①无执行追踪——记住了偏好但不保证按偏好执行 ②检索质量依赖embedding，复杂意图易miss ③无冲突解决机制（新旧偏好矛盾时无确定性策略） |
| **Zep** | 知识图谱+事实提取+时间衰减 | 结构化记忆；时间感知（最近的记忆权重更高） | ①图谱构建依赖LLM提取，复杂关系易出错 ②无程序化执行层 ③规模化后图谱维护成本高 |
| **LangChain Memory** | ConversationBuffer/Summary/KG多种模式 | 灵活、易集成 | ①本质是Prompt拼接，受Context Window限制 ②无持久化（需自己接DB） ③无可靠性保证 |
| **MemGPT/Letta** | 虚拟内存分页（主内存+归档内存）；LLM自主管理内存 | 突破Context Window限制；LLM可"主动回忆" | ①LLM管理内存本身就不可靠（该记的没记、该忘的没忘） ②内存操作消耗Token，成本高 ③无外部校验机制 |
| **Cognee** | 知识图谱+语义分块 | 结构化知识管理 | ①侧重知识而非行为记忆 ②无执行层 |

### 3.2 核心空白：所有方案都缺"执行外骨骼"

```
记忆方案解决的：  LLM知道什么 → 记住什么 → 回忆什么
记忆方案没解决的：LLM计划什么 → 执行什么 → 验证什么 → 纠错什么
```

**这正是我们的实测数据指向的核心问题**：LLM的不可靠性不在于"忘了"，而在于"做错了"。记忆系统让LLM"不健忘"，但程序化外骨骼让LLM"不犯错"。

**类比**：记忆系统 = 给失忆症患者一本笔记本；程序化外骨骼 = 给帕金森患者一个机械臂。两者解决的是完全不同的问题。

## 四、打法策略：构建"全球最强AI Agent系统"

### 4.1 核心战略：做外骨骼，不做肌肉

**模型会被OpenAI/Anthropic/Google卷，但外骨骼是我们的壁垒。**

```
我们的定位：
┌──────────────────────────────────────┐
│         LLM（任意模型，可替换）         │  ← 别人的肌肉
├──────────────────────────────────────┤
│    程序化外骨骼（我们的核心价值）        │  ← 我们的骨架
│  ┌──────────┬──────────┬──────────┐  │
│  │ 执行追踪  │ 规则引擎  │ 状态机   │  │
│  │ 记忆管理  │ 自检验证  │ 回滚恢复 │  │
│  └──────────┴──────────┴──────────┘  │
├──────────────────────────────────────┤
│         设备/服务/数据接口层            │  ← 连接万物
└──────────────────────────────────────┘
```

### 4.2 五大关键取舍

| 取舍项 | 选择 | 放弃 | 理由 |
|--------|------|------|------|
| **模型 vs 系统** | 投资系统（外骨骼） | 自研大模型 | 模型是红海，系统是蓝海；模型能力提升也利好我们 |
| **通用 vs 垂直** | 先垂直后通用 | 一步到位做通用Agent | 垂直场景可程序化→可靠→可商用；通用需等模型能力跨过阈值 |
| **端侧 vs 云侧** | 云侧为主，端侧做感知 | 重投端侧大模型 | 外骨骼逻辑天然适合云侧；端侧做轻量感知+隐私过滤 |
| **记忆 vs 程序化** | 程序化优先，记忆辅助 | 只做记忆 | 可靠执行 > 用户画像；记忆是锦上添花，执行是生死线 |
| **开放 vs 封闭** | 开放生态（支持多模型/多设备） | 绑定单一模型或生态 | Apple的壁垒是硬件生态，我们的壁垒应该是系统能力 |

### 4.3 具体架构建议

**分层可靠性架构**：

```
Layer 4: 意图理解（LLM驱动，允许模糊）
    ↓ 意图 → 结构化任务描述
Layer 3: 任务规划（LLM辅助 + 模板库，半确定性）
    ↓ 计划 → 步骤序列
Layer 2: 步骤执行（程序化为主，LLM仅在异常时介入）
    ↓ 每步结果 → 状态追踪
Layer 1: 验证与回滚（纯程序化，零LLM依赖）
    ↓ 通过/失败 → 重试或人工介入
Layer 0: 持久化（记忆+日志+状态，纯确定性）
```

**关键设计原则**：
1. **LLM只在Layer 3-4介入**，Layer 0-2是确定性代码
2. **每个LLM调用都有超时+回退**，不依赖LLM的"自检"（我们的数据证明自检不可靠）
3. **状态外置**——所有中间状态存在外部存储，不在LLM context中
4. **步骤原子化**——每步≤2个LLM调用，保持在可靠区间内

### 4.4 与大厂的差异化

| 维度 | Apple/华为等 | 我们的定位 |
|------|------------|-----------|
| 核心资产 | OS + 硬件生态 | 外骨骼系统 + 跨平台 |
| 模型策略 | 自研端侧 + 合作云端 | 模型无关，接入任意LLM |
| 记忆策略 | 设备本地为主 | 云端加密 + 跨设备同步 |
| 场景策略 | OS原生功能增强 | 跨OS、跨设备的Agent服务 |
| 壁垒来源 | 生态锁定 | 系统可靠性 + 开放生态 |

## 五、风险与不确定性

1. **模型能力跳变风险**：如果GPT-6/Claude-5的可靠区间从4步跳到20步，外骨骼的价值会被压缩（但不会归零——100步的任务仍需要外骨骼）
2. **大厂下场风险**：Apple/Google可能直接在OS层做外骨骼（Shortcuts就是雏形），但它们的外骨骼是封闭的
3. **开源竞争**：LangGraph/CrewAI等框架在向外骨骼方向演进，但目前仍偏"编排"而非"可靠性保证"
4. **隐私监管**：云端记忆面临GDPR/数据本地化挑战，需要端侧处理+加密上传的混合方案

---

## 六、行动建议（优先级排序）

1. **P0 - 构建执行外骨骼核心**：状态机 + 步骤追踪 + 自动回滚，这是护城河
2. **P0 - 选3个垂直场景做到极致可靠**：如日程管理、信息检索整理、设备控制
3. **P1 - 记忆系统作为模块集成**：不重复造轮子，在mem0/Zep基础上增加"执行相关记忆"
4. **P1 - 多模型适配层**：确保外骨骼与模型解耦，支持快速切换
5. **P2 - 端侧感知SDK**：轻量化感知模块，做隐私过滤和意图预分类
6. **P2 - 开发者生态**：让第三方可以在外骨骼上定义新的垂直场景流程
