# 故障恢复树

## 1. timeout
- **首选动作**：
  - 记录超时发生的操作、目标对象、超时阈值与上下文。
  - 将同一操作立即重试 1 次，并适度增加超时时间。
  - 若操作依赖外部资源，优先检查网络、服务状态、页面/接口加载状态。
- **二次尝试条件**：
  - 首次重试后仍超时，但可确认目标服务仍可达。
  - 超时原因可能来自瞬时拥塞、负载抖动、冷启动、页面尚未稳定。
  - 输入参数与前置状态已确认无误。
- **最大重试次数**：
  - 总计最多 3 次（首次执行 + 2 次重试）。
  - 每次重试应采用渐进式退避，并在必要时提高超时阈值。
- **何时停止并上报**：
  - 连续达到最大重试次数仍失败。
  - 发现服务不可达、依赖已中断、前置条件无法满足。
  - 超时已影响后续步骤正确性，继续重试无明确收益时。

## 2. permission
- **首选动作**：
  - 立即识别是文件权限、API 权限、账户角色权限还是运行环境限制。
  - 检查当前凭据、作用域、目录可写性或调用通道是否具备所需授权。
  - 若存在等价的低权限可行路径，切换到不需要额外权限的方案。
- **二次尝试条件**：
  - 已明确权限缺口仅为临时态，例如会话失效、令牌过期、挂载目录切换错误。
  - 可通过刷新认证、切换正确账户、改用已有权限路径来恢复。
  - 不涉及越权操作，也不会违反安全边界。
- **最大重试次数**：
  - 最多 2 次（首次执行 + 1 次针对权限修正后的重试）。
  - 不允许在权限未变化的情况下盲目重复尝试。
- **何时停止并上报**：
  - 缺少必要授权且当前上下文无法自行补齐。
  - 需要用户、管理员或外部系统授予权限。
  - 涉及安全策略、沙箱限制、访问控制拒绝，继续尝试无意义时。

## 3. not-found
- **首选动作**：
  - 核对目标标识符是否正确，包括路径、文件名、URL、记录 ID、页面元素引用等。
  - 检查资源是否被移动、删除、重命名，或是否位于错误环境/目录/空间。
  - 若是依赖前序步骤生成的对象，先验证前序步骤是否真正成功。
- **二次尝试条件**：
  - 目标可能因延迟创建、索引未刷新、页面尚未渲染而暂时不可见。
  - 存在可靠的替代定位方式，例如按父目录、标题、模糊匹配或最新生成结果重新查找。
  - 已排除拼写错误与环境切换错误。
- **最大重试次数**：
  - 最多 3 次（首次执行 + 2 次重新定位/重新查询）。
  - 每次重试必须改变定位策略，不能重复同一无效查询。
- **何时停止并上报**：
  - 确认资源不存在、已删除、未创建成功或无权访问其所在位置。
  - 替代定位策略均失败，且无更多可验证线索。
  - 缺失目标会阻断后续任务时，应停止并说明缺失对象及已尝试路径。

## 4. flaky
- **首选动作**：
  - 将问题视为非确定性失败，先记录失败特征：频率、触发步骤、环境、日志片段。
  - 重新执行该步骤，并尽量固定变量，如等待稳定状态、减少并发、显式校验前置条件。
  - 若是 UI/网络类操作，增加稳定性检查，例如元素可见、接口响应完成、状态文本出现。
- **二次尝试条件**：
  - 失败表现不一致，且历史上同操作存在成功案例。
  - 已能识别潜在抖动源，如异步渲染、竞态、短暂网络波动、缓存未就绪。
  - 可通过增加等待、串行化、重新获取引用、重新打开上下文来降低不稳定性。
- **最大重试次数**：
  - 最多 3 次（首次执行 + 2 次有针对性的稳定化重试）。
  - 每次重试需附带一种稳定化措施，避免无差别重复。
- **何时停止并上报**：
  - 达到最大重试次数后仍无法稳定复现成功路径。
  - 故障表现相互矛盾，无法判断真实状态。
  - 继续执行会放大副作用、污染数据或造成重复提交时，应立即停止并上报。
