# 工具超预算后的降级策略（Playbook）

适用目标：当工具调用接近或超过时间、次数、上下文、令牌或结果体积预算时，统一采用可预测、可审计的降级路径，避免无边界继续扩查。

## 总则

1. **先缩小范围**：优先减少输入规模、结果数量、时间窗口、页面深度、文件范围或命令作用域。
2. **再停止扩查**：如果缩小后仍无法在预算内得到稳定结论，立即停止继续探索，不做“再试一次也许会成功”的无界尝试。
3. **输出不确定性**：明确说明已执行到哪一步、哪些证据不足、结论的置信度和可能遗漏。
4. **请求人工介入**：当问题依赖外部判断、授权、交互、环境修复或更大预算时，明确给出需要人工提供的最小补充信息或操作。

## 统一触发信号

出现以下任一情况，可视为“超预算”并启动降级：

- 工具调用次数已达到预设上限。
- 单次调用结果过大，导致有效信息提取效率显著下降。
- 连续两轮缩小范围后，仍未获得足够证据。
- 外部环境不稳定：超时、限流、页面频繁变化、命令卡住、权限不足。
- 继续扩查的新增信息价值低，重复结果明显增多。

## 一、read 类工具降级策略

适用场景：读取大文件、日志、配置、代码、文本输出等。

### 推荐顺序

#### 1. 缩小范围
按以下顺序逐步收缩：

- 只读目标文件，不连带读取相关文件。
- 限制行数或字节数，优先读取头部、尾部或用户指定区段。
- 基于已知关键词定位后，仅读取命中附近少量上下文。
- 只读最可能相关的 1–3 个文件，不扩展到全项目扫描。
- 对日志类文件，优先读取最新时间窗口，而非全量历史。

#### 2. 停止扩查
满足任一条件即停止：

- 已完成 2 次以上范围收缩，仍找不到明确证据。
- 读取内容持续重复、噪声高，新增片段未提供新线索。
- 文件过大且无有效定位锚点，继续分页读取只会线性消耗预算。

#### 3. 输出不确定性
输出中应包含：

- 已读取的文件路径、区段、关键词或时间范围。
- 未读取的部分及原因（过大、无定位、预算不足）。
- 当前只能得出的保守判断，例如“在已检查区段内未发现…，但不能排除未读部分存在…”。

#### 4. 请求人工介入
在以下情况下请求人工：

- 需要用户指出更具体的文件、函数、关键词、时间段。
- 需要提供压缩后的复现日志或最小相关片段。
- 需要授权读取额外目录、私有文件或更大日志窗口。

### read 类停止条件

- 读取 2–3 个关键区段后仍无新信息。
- 已覆盖最可能相关位置（头/尾/命中上下文）但无法形成结论。
- 继续读取会明显超过消息体积或处理预算。

## 二、search 类工具降级策略

适用场景：站内/站外搜索、代码搜索、网页搜索、知识库搜索。

### 推荐顺序

#### 1. 缩小范围
按以下顺序逐步收缩：

- 收紧查询词，加入唯一标识词、错误码、文件名、模块名。
- 降低结果数，只看前 3–5 条高相关结果。
- 限定来源：单一域名、单一仓库、单一目录、单一知识库。
- 限定时间范围，仅查最近或指定时间段内容。
- 去除泛化问题，转为验证式查询（例如“是否存在 X”而不是“告诉我一切关于 X”）。

#### 2. 停止扩查
满足任一条件即停止：

- 前两轮查询结果高度重复。
- 新搜索词只是同义改写，未引入新的约束信息。
- 高相关结果已读完，剩余结果相关性明显下降。
- 搜索命中彼此冲突，且无额外证据源可快速裁决。

#### 3. 输出不确定性
输出中应包含：

- 已尝试的查询词、限定条件和结果来源。
- 结论基于哪些高相关结果。
- 未覆盖的来源范围，以及因此带来的不确定性。
- 若存在冲突信息，需明确写出“不同来源结论不一致，当前无法在预算内完成交叉验证”。

#### 4. 请求人工介入
在以下情况下请求人工：

- 需要用户提供更精准关键词、专有名词、内部术语。
- 需要访问受限站点、私有知识库或登录态内容。
- 需要人工决定优先信任哪个来源或哪一版本文档。

### search 类停止条件

- 连续 2 轮搜索未引入新证据。
- 前 3–5 条高相关结果已检查完毕且结论仍不稳定。
- 继续搜索主要是在扩大覆盖面，而非提升结论质量。

## 三、exec 类工具降级策略

适用场景：执行 shell 命令、脚本、构建、测试、诊断命令。

### 推荐顺序

#### 1. 缩小范围
按以下顺序逐步收缩：

- 先运行只读、低成本、无副作用的诊断命令。
- 对目录、文件、测试目标做最小化：单文件、单模块、单测试用例。
- 为命令设置明确超时，禁止无限等待。
- 限制输出规模：加过滤、分页、tail、head、grep、错误级别筛选。
- 避免整仓构建/全量测试，优先最小复现路径。

#### 2. 停止扩查
满足任一条件即停止：

- 同类命令连续 2 次超时或卡住。
- 输出过大但没有收敛到具体错误点。
- 环境问题明显（缺依赖、权限、网络、锁、交互阻塞）且无法在当前权限下修复。
- 再执行一次只会重复已有失败模式。

#### 3. 输出不确定性
输出中应包含：

- 已执行的命令、工作目录、超时设置、是否只读。
- 观察到的关键现象（报错、退出码、卡住位置）。
- 未执行的高成本命令及跳过原因。
- 对结论的限制，例如“当前更像环境故障而非代码逻辑故障，但未完成全量验证”。

#### 4. 请求人工介入
在以下情况下请求人工：

- 需要安装依赖、提升权限、提供密钥、解锁资源或修复环境。
- 需要用户确认是否允许执行高成本或有副作用命令。
- 需要用户提供稳定复现步骤、样例输入或预期输出。

### exec 类停止条件

- 2 次超时/阻塞后仍未定位到更具体问题。
- 已尝试最小复现路径，仍被环境问题拦截。
- 继续执行会产生明显副作用或资源消耗，且结论收益不足。

## 四、browser 类工具降级策略

适用场景：网页浏览、页面抓取、UI 自动化、表单填写、点击导航。

### 推荐顺序

#### 1. 缩小范围
按以下顺序逐步收缩：

- 优先只做 snapshot / 读取页面结构，不直接做多步自动化。
- 限制在单标签页、单页面、单弹窗内操作，不并行扩展多个页面。
- 缩短交互链路：从“完整流程”降为“验证关键一步是否存在/可点击/可见”。
- 限制等待条件，只等待关键元素或关键文本，不做长时间泛等待。
- 对信息获取类任务，优先抓取当前可见内容，不滚动到底、不深挖分页。

#### 2. 停止扩查
满足任一条件即停止：

- 页面结构频繁变化，元素引用连续失效。
- 登录、验证码、权限弹窗、扩展接管等前置条件缺失。
- 同一路径尝试 2 次后仍卡在相同步骤。
- 页面加载缓慢或不稳定，继续自动化只会增加失败噪声。

#### 3. 输出不确定性
输出中应包含：

- 已访问的页面、已验证的关键元素或步骤。
- 卡住的具体环节（登录、验证码、按钮不可见、权限拦截等）。
- 当前只能确认的事实，与无法确认的后续流程。
- 如页面动态性强，需说明“当前快照级证据有限，后续状态可能变化”。

#### 4. 请求人工介入
在以下情况下请求人工：

- 需要用户手动登录、完成验证码、点击扩展按钮、授权权限。
- 需要用户指定目标标签页、具体页面或操作路径。
- 需要人工确认页面上语义模糊的选项、按钮或业务判断。

### browser 类停止条件

- 同一步骤失败 2 次且失败原因未变化。
- 缺少登录态/权限/人工确认，无法继续。
- 页面不稳定导致自动化结果不再可靠。

## 结果输出模板

当进入降级状态时，建议统一使用以下结构输出：

1. **已做范围收缩**：列出缩小了哪些范围。
2. **停止原因**：说明为何不再继续扩查。
3. **当前可确认结论**：只给保守结论。
4. **不确定性**：明确未覆盖部分和风险。
5. **需要人工提供**：列出最小必要输入/操作。

## 最低执行要求

- 不因“也许下一次会成功”而无限续试。
- 每类工具最多做有限轮次收缩；收缩后仍无新增证据即停止。
- 输出必须显式包含不确定性，不能把“未发现”表述成“确定不存在”。
- 请求人工介入时，要给出具体可执行的最小动作，而不是笼统说“请提供更多信息”。
