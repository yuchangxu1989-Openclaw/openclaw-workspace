### 1.2 问题清单

**P0 — 4个分类（222条，占52.5%）无明确映射规则：**

方案将纠偏类(65)、交付质量类(64)、全局未对齐类(64)、连锁跷跷板类(29)标注为"需逐条判断"，但未给出判断依据。这222条是工作量的主体，不能留白。

建议补充二级映射规则，例如：
- 纠偏类：若涉及"用户催促才行动" → 自主闭环率；若涉及"规则创建后未生效" → 言出法随达成率；若涉及"修复未做根因分析" → 根因分析覆盖率
- 交付质量类：若涉及"只看文档不看代码" → 认知层真实代码覆盖率；若涉及"自审通过但有缺陷" → 独立QA覆盖率
- 全局未对齐类/连锁跷跷板类：同理需按case内容特征给出映射决策树

**P1 — 多标签规则未定义：**

方案提到"标注对应的1-2个北极星指标"，但V4 batch参考实现（`v4-gate-cases-batch1.json`）中每条case只标注1个指标（字符串类型，非数组）。需明确：
- 字段类型是字符串还是数组？
- 若允许多标签，主指标如何确定？
- 建议与V4 batch保持一致，使用单标签字符串

**P1 — 无法映射的case处理策略缺失：**

若某条case确实不直接对应任何北极星指标（如纯过程指标场景），`north_star_indicator`填什么？null？还是强制映射到最近的指标？方案未说明。

### 1.3 结论

映射规则仅覆盖3/5个指标、201/423条case（47.5%）。剩余52.5%的case缺乏可操作的映射指引。**指标1（言出法随）和指标4（独立QA）在现有423条中确实无直接对应case，方案正确识别了这一缺口，但应在Phase 1中也给出"这些case是否可能间接关联"的判断规则，而非完全留到Phase 2。**

## 二、scoring_rubric 三档标准审查

### 2.1 格式一致性问题（P0）

V4标准要求的格式（参照 `v4-gate-cases-batch1.json`）：

```json
{
  "pass": "完全正确行为的具体描述",
  "partial": "部分正确但有遗漏的具体描述",
  "badcase": "完全错误行为的具体描述"
}
```

但现有 `goodcases-from-badcases.json` 中已有的 `scoring_rubric` 是**纯字符串格式**：

```json
"scoring_rubric": "系统应执行: 任务总结命中badcase关键词 then..."
```

**方案完全未提及这一格式不一致问题。** 这意味着：
1. 已有scoring_rubric的case也需要格式转换（从字符串→三档对象）
2. 工作量被低估（不仅是423条从零生成，还有已有case的格式迁移）

### 2.2 三档区分原则未定义（P0）

方案说"为每条生成 pass/partial/badcase 三级评分标准"，但未定义：
- pass与partial的边界在哪里？（V4 batch中的模式：partial通常是"执行了但遗漏1-2个子步骤"）
- partial与badcase的边界在哪里？（V4 batch中的模式：badcase通常是"完全跳过关键步骤"或"方向性错误"）
- 每档描述的粒度要求？（V4 batch中每档都精确到具体行为，非模糊描述）

**建议补充rubric生成模板：**

```
pass: [完整执行了expected_output中的全部关键步骤，且结果正确]
partial: [执行了主要步骤但遗漏了{具体哪些子步骤}，或{具体哪个环节}不够精确]
badcase: [完全未执行{核心步骤}，或{方向性错误的具体描述}]
```

### 2.3 LLM生成质量风险（P1）

方案提出"LLM批量生成初稿"，但未评估以下风险：
- LLM容易生成模糊的、不可操作的rubric（如"基本完成了任务"而非"执行了4层根因分析中的3层"）
- LLM可能不理解AEO领域特定概念（如ISC规则全链路、Pre-Gate校验项）
- 无few-shot示例策略：应从V4 batch中提取3-5条高质量rubric作为few-shot prompt

### 2.4 结论

scoring_rubric是整个补全方案中工作量最大、质量风险最高的部分。方案缺少格式规范、区分原则、生成模板和质量基线。**必须在执行前补充rubric生成规范和few-shot prompt设计。**

## 三、gate 标签策略审查

### 3.1 与V4文档的一致性

V4定义的gate值域和含义：

| gate值 | 含义 | V4执行阶段 |
|--------|------|-----------|
| `pre-gate` | Pre-Gate基础完整性门禁（规则id/trigger/handler/映射） | 第一道关卡 |
| `gate-a` | Gate-A审计工具可信门（审计脚本/评测工具的可信性） | 第二道关卡 |
| `gate-b` | Gate-B标准-脚本绑定门（标准版本与脚本同步） | 第四道关卡 |
| `null` | 不涉及门禁，属于北极星指标评测阶段 | 第三道（主评测） |

方案说"大部分case为null，少数涉及规则创建/验证的可标注"。

### 3.2 问题清单

**P1 — 标注规则不够具体：**

"少数涉及规则创建/验证的可标注"过于模糊。建议补充具体判断规则：
- 若case涉及ISC规则字段完整性校验 → `pre-gate`
- 若case涉及审计脚本/评测工具的可信性验证 → `gate-a`
- 若case涉及标准文档版本与评测脚本的同步性 → `gate-b`
- 其余 → `null`

**P2 — 423条中可能存在隐性gate case：**

方案断言"423条中无任何Gate相关case"，但未逐条验证。例如：
- 交付质量类中可能有涉及"评测脚本产出不可信"的case → 可标gate-a
- 全局未对齐类中可能有涉及"标准与执行不一致"的case → 可标gate-b

建议在Phase 1标注时增加一轮gate候选筛查。

### 3.3 结论

gate标签策略方向正确（大部分为null），但缺少可操作的判断规则。风险可控，补充判断规则后可执行。

## 五、批量补全质量保障机制审查

### 5.1 方案现有机制

方案提到的质量保障仅两句话：
1. "LLM批量生成初稿 + 人工审核"（scoring_rubric）
2. "基于分类规则批量映射 + 人工校验"（north_star_indicator）

### 5.2 缺失的关键机制（按优先级排序）

**P0 — 无验收标准：**

什么样的补全结果算"合格"？需要定义：
- scoring_rubric：三档描述是否可操作（能否据此明确判定一个实际输出属于pass/partial/badcase）
- north_star_indicator：标注是否与case内容一致（抽样交叉验证通过率≥95%）
- gate：标注是否与V4门禁定义一致

**P0 — 无抽样审核策略：**

423条全量人工审核不现实（尤其scoring_rubric）。需要定义：
- 分层抽样方案：每个分类至少抽检N条（建议每类≥10条，共≥80条）
- 抽检通过标准：抽样合格率≥90%则该批次通过，否则全批次返工
- 边界case重点审核：分类映射模糊的case（纠偏类/交付质量类等）应全量审核north_star_indicator

**P1 — 无自动化一致性校验：**

补全后应运行自动化检查脚本，至少包括：
- 格式校验：scoring_rubric是否为`{pass, partial, badcase}`三键对象
- 非空校验：三档描述是否均非空且长度≥20字符
- 交叉一致性：scoring_rubric中提到的行为是否与expected_output一致
- north_star_indicator值域校验：是否为5个合法值之一
- gate值域校验：是否为`pre-gate`/`gate-a`/`gate-b`/`null`之一

**P1 — 无与V4 batch的对标验证：**

应将补全后的case与`v4-gate-cases-batch1.json`中的15条case进行格式和质量对标，确保：
- 字段结构完全一致
- rubric粒度和风格一致
- 可以混合加载到同一评测框架中运行

**P2 — 无回退机制：**

如果LLM生成质量不达标（如抽检合格率<80%），方案未说明：
- 是否切换生成策略（如改用更强模型、增加few-shot数量）
- 是否回退到全量人工编写
- 时间和资源的应急预案

**P2 — 无版本管理策略：**

补全过程中的中间版本如何管理？建议：
- 每个Phase完成后git tag标记
- 补全前后的diff可追溯
- 保留LLM生成的原始版本和人工修正后的版本

### 5.3 结论

质量保障机制严重不足，仅有"生成+审核"的粗略描述，缺少验收标准、抽样策略、自动化校验、对标验证、回退机制五个关键环节。**这是整个方案最大的风险点。**
