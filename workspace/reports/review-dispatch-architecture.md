## 一、完整性 — 评分: 7/10

### 优点
- 三个根因（感知-负载、感知-权限、规划-拆解）的分类逻辑清晰，维度正交
- 现状诊断准确：经核实，main agent的tools.deny确实缺少exec和read，systemPrompt确实为NULL，agent数量确实是19个
- 框架能力边界分析（2.1/2.2节）诚实标注了"不支持"的能力，没有虚构框架特性

### 问题
1. **根因归类不够精确**：根因2"exec未封→主Agent自己干活"本质上不是"感知-权限"问题，而是"执行边界"问题。主Agent不是"感知不到"自己有权限，而是"没有约束"不去用。归类为"感知层"会误导方案设计方向
2. **MECE检验不通过**：三个根因之间存在交叉——
   - 根因2（主Agent自己干活）和根因3（不拆解）经常同时发生：主Agent不拆解的原因之一就是它能自己exec干活。封掉exec后，根因3可能自然缓解
   - 这意味着方案可能过度设计——封exec一招可能同时解决根因2和部分根因3
3. **缺少数据支撑**：报告没有给出"主Agent自己干活"的频率统计、"盲派导致堆积"的实际案例数据。诊断基于定性判断而非定量分析

## 二、可行性 — 评分: 5/10

### 优点
- tools.deny机制已验证生效（实际配置确认）
- dispatch-guard.js v1已存在且功能可用
- Agent池化方案（预注册）在框架内确实可行

### 严重问题

#### 2.1 dispatch-guard v2代码与v1完全不兼容
报告中提出的v2代码是全新实现，但**实际部署的v1**（已验证存在于`/root/.openclaw/workspace/scripts/dispatch-guard.js`）结构完全不同：
- v1: 硬编码ROLE_MAP、validate/pickIdle接口、CLI参数为`(id, label)`
- v2: 动态发现、globalSnapshot/pickBest/allocateBatch接口、CLI参数为`(snapshot|pick|batch)`

报告没有提及迁移方案。如果直接替换v2，**所有依赖v1接口的现有调用都会断掉**。

#### 2.2 tools.allow未验证就纳入方案
报告2.1节标注`tools.allow`为"⚠️需验证"，但后续方案没有依赖它，这点OK。但如果未来需要白名单模式，必须先验证。

#### 2.3 deny列表出现3个不同版本——自相矛盾
| 位置 | exec | read | feishu_doc | canvas |
|------|------|------|------------|--------|
| 3.2.1节（首版） | ✅封 | ✅封 | 分action讨论 | 未提及 |
| 3.2.3节（务实版） | ❌保留 | ✅封 | ✅封 | ✅封 |
| 4.1节（最终配置） | ❌保留 | ✅封 | ✅封 | ✅封 |

报告内部对"是否封exec"反复摇摆，最终选择保留exec但用systemPrompt限制。这个决策过程暴露了方案的核心矛盾（见风险部分）。

#### 2.4 systemPrompt强制调用dispatch-guard的可靠性存疑
报告自己承认"systemPrompt约束不是硬限制"。在实际运行中，LLM在长对话、复杂任务、上下文压力下经常忽略systemPrompt中的流程指令。**把关键调度逻辑建立在"LLM会遵守指令"的假设上，可靠性不足。**

#### 2.5 封read后主Agent如何获取子Agent结果？
报告提出封掉read，但主Agent需要读取子Agent返回的结果来做整合汇报。虽然子Agent完成后会自动announce，但如果主Agent需要回溯历史结果（如用sessions_history），返回的内容可能不够。这个交互链路没有验证。

## 三、风险 — 评分: 4/10

### 报告已识别的风险（5项）
报告第五节列出了5个风险，但缓解措施普遍偏弱。

### 报告未识别的严重风险

#### 3.1 🔴 exec保留 = write/edit/read的deny全部可绕过
这是最严重的安全漏洞。保留exec意味着主Agent可以：
```bash
exec: echo "恶意内容" > /path/to/file          # 绕过write deny
exec: cat /path/to/file                          # 绕过read deny
exec: sed -i 's/old/new/' /path/to/file          # 绕过edit deny
exec: curl -X POST ... | bash                    # 任意远程执行
exec: node -e "require('fs').writeFileSync(...)"  # 绕过所有文件操作deny
```
报告在风险表中提到了`echo > file`，但缓解措施仅为"systemPrompt严格限制 + 定期审计"。**这不是缓解，这是祈祷。** 保留exec等于deny列表形同虚设。

#### 3.2 🔴 main-elevated是权限提升后门
方案提出的"提权代理"设计：
```
主Agent → spawn(agentId="main-elevated", task="用户授权执行: xxx")
```
问题：
- "用户授权"的判断完全由主Agent的LLM决定，没有任何硬验证机制
- 主Agent可以在任何时候以"用户授权"为由spawn main-elevated执行任意操作
- 这等于给主Agent开了一个不受deny约束的后门
- 正确做法：临时授权应由**用户直接操作**（如修改配置、发送特定指令给独立的elevated agent），而不是由被限制的agent自己决定何时提权

#### 3.3 🟡 MECE强制拆解的副作用未评估
- 简单任务（如"查看某文件内容"）如果也走MECE流程，会引入不必要的延迟（spawn scout → 等待返回 → 汇报，至少多30秒）
- 报告的Step 0"规模判断"标准模糊（"不需要读文件、不需要执行命令"），LLM对边界case的判断不稳定
- 过度拆解的overhead：每个子Agent启动有固定成本（~5-10秒），如果把一个10秒能完成的任务拆成5个子任务，总耗时反而更长

#### 3.4 🟡 dispatch-guard.js的sessions.json读取是竞态操作
多个并发的dispatch-guard调用同时读sessions.json，可能读到不一致的状态。报告说"可接受"，但在高并发场景下（正是方案要追求的），竞态概率显著增加。

## 四、遗漏 — 评分: 3/10

### 严重遗漏

#### 4.1 子Agent超时/失败后的重试策略 — 完全缺失
当前系统没有任何机制处理：
- 子Agent执行超时（卡死）
- 子Agent执行失败（报错退出）
- 子Agent执行完但结果质量不合格

这在高并发场景下是必然会发生的。没有重试/降级策略，一个子Agent失败就会导致整个任务链断裂。

#### 4.2 多个子Agent结果冲突时的仲裁 — 完全缺失
MECE拆解后多个子Agent并行执行，如果：
- 两个子Agent修改了同一个文件（git冲突）
- 两个子Agent给出了矛盾的结论
- 整合时发现子任务之间有未预见的依赖

报告没有任何仲裁机制。

#### 4.3 看板推送自动化 — 仅一句话带过
报告4.4节说"dispatch-guard.js的globalSnapshot()输出可直接用于看板推送"，但：
- 没有具体的集成方案
- 当前看板推送依赖cron定时（之前修复过），与dispatch-guard无关
- 这是用户已知的痛点，不应一笔带过

#### 4.4 systemPrompt的token消耗未评估
报告提出的"调度协议v1"systemPrompt约有800-1000 tokens。主Agent的每次对话都会携带这些tokens，在长对话中会显著压缩可用上下文窗口。对于200K上下文的模型影响不大，但应该明确评估。

#### 4.5 现有dispatch-guard.js v1的集成状态未核实
报告1.1节说dispatch-guard.js"未集成到任何自动化流程中"。但实际上v1已经存在且有CLI接口，是否已经被某些脚本或流程调用？报告没有做grep排查。

#### 4.6 子Agent之间的协调机制缺失
高并发场景下，多个子Agent可能需要：
- 共享中间结果（如一个agent发现的信息对另一个有用）
- 避免重复工作（两个agent不小心做了同一件事）
- 顺序协调（某些子任务有隐含依赖）

报告完全没有涉及。

## 五、与用户需求的对齐 — 评分: 6/10

### 对齐的部分
- "并发是北极星指标"在报告中被反复强调，方向正确
- dispatch-guard的负载均衡、MECE拆解、Agent池化都直接服务于并发提升
- 并发度量（utilization%）的引入让并发可观测

### 未对齐的部分

#### 5.1 没有量化预期收益
报告没有回答：实施后并发利用率预期从多少提升到多少？
- 当前baseline是多少？（应该统计过去N天的平均并发数）
- 实施P0后预期提升多少？
- 实施P1后预期提升多少？
没有数字，就无法验证方案是否有效。

#### 5.2 并发的真正瓶颈可能不是调度
即使调度完美，并发上限取决于：
- 任务是否真的可以并行（很多用户任务本质上是串行的）
- 子Agent的质量（并发10个但5个失败 = 有效并发5个）
- 用户的任务到达率（用户一次只提一个需求，调度再好也只能拆成N个子任务）

报告假设"只要调度好，并发就能上去"，但没有分析任务本身的并行度。

#### 5.3 验证方案过于理想化
第七节的4个验证场景都是happy path，没有覆盖：
- 子Agent失败时的行为
- 高并发下的资源争抢
- systemPrompt被忽略时的降级

## 七、改进建议

### P0 — 必须修复后才能实施

1. **统一deny列表**：报告中3个版本必须收敛为1个。建议：**封掉exec**，dispatch-guard的调用改为派scout子Agent执行，虽然多一跳延迟但安全性质变
2. **删除main-elevated方案**：用"用户直接对elevated agent发消息"替代"主Agent自己决定提权"。或者至少加一个硬验证（如特定文件token）
3. **补充失败处理机制**：至少定义子Agent超时阈值（如5分钟）、超时后的重试策略（换agent重试1次）、最终失败的上报机制

### P1 — 实施前应补充

4. **补充v1→v2迁移方案**：明确v1的调用方、接口变更影响、灰度切换步骤
5. **补充并发baseline数据**：统计过去7天的平均并发数、峰值并发数，作为方案效果的对比基准
6. **补充结果仲裁机制**：至少定义"多个子Agent修改同一文件时的git merge策略"
7. **评估systemPrompt的token成本**：计算调度协议v1的token数，确认对上下文窗口的影响可接受

### P2 — 持续优化

8. **引入任务并行度分析**：不是所有任务都适合拆解，应该有"并行度评估"步骤，避免过度拆解
9. **dispatch-guard加锁或用原子操作**：解决sessions.json的竞态读取问题
10. **看板推送与dispatch-guard深度集成**：不是"可以用"，而是给出具体的集成代码和触发时机
