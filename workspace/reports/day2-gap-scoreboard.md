# Day2 缺口总表（Agent 运行效率与质量）

> 范围限定：仅评估 **Agent 运行效率与质量**。不涉及项目管理、排期、协作流程。  
> 评级含义：`done` = 已收口；`partial` = 已有方案但未闭环；`p0` = 当前会直接影响主链路效率/质量；`p1` = 影响次主链路或放大运维成本；`observe` = 暂不构成阻断，但需持续观测。

## 总表

| 维度 | 当前状态 | 结论 | 硬判断 |
|---|---|---|---|
| 意图（Intent） | partial | 已具备基础能力，但意图分流、歧义消解、失败回退后的二次判定还未完全收口 | **不是 P0，但仍未闭环** |
| 事件（Event） | partial | 事件触发链路可用，但事件语义标准化、去重、优先级和质量标记不够硬 | **当前是质量放大器，不是最终收口态** |
| 执行链（Execution Chain） | p0 | 多步执行成功率、链路可解释性、异常中断恢复仍是主风险源 | **P0，直接决定 Agent 可用性** |
| Cron 模型治理 | p1 | 定时任务/周期性调用可运行，但模型选择、预算控制、任务分级治理未完全固化 | **P1，会持续侵蚀效率与成本** |
| Failover | p0 | 降级与切换机制若未做强约束，会导致执行中断或质量骤降不可控 | **P0，必须明确主备与触发条件** |
| 业务级 E2E | p0 | 缺少业务结果口径下的端到端闭环验证，单点能力通过不代表真实可用 | **P0，没有 E2E 就没有真实质量结论** |
| 静默策略 | partial | 静默/少打扰策略有必要，但过强会掩盖失败，过弱会打断主流程 | **需平衡，当前未完全收口** |
| 稳定性 | p0 | 抖动、超时、重试风暴、状态漂移仍是系统级质量核心问题 | **P0，是 Day2 质量底盘** |

## 分项判断

| 维度 | 状态 | 已收口 | 未收口缺口 | 对效率/质量的直接影响 |
|---|---|---|---|---|
| 意图 | partial | 基础意图识别、常见任务进入执行链的能力通常已具备 | 多意图并存、模糊输入、上下文切换、执行失败后的重判定未完全标准化 | 会造成误判、错链、重复追问，拉低首轮成功率 |
| 事件 | partial | 基础事件驱动和状态回传通常能跑通 | 事件分类、幂等、去重、优先级、事件质量标签、事件到动作的映射边界不够清晰 | 容易触发重复执行、漏执行、错误升级，带来噪声和抖动 |
| 执行链 | p0 | 局部链路可执行 | 缺少稳定的链路编排、步骤级回滚/重试、上下文续接、失败定位与可解释性 | 直接决定任务是否能稳定完成，是当前主矛盾 |
| Cron 模型治理 | p1 | 定时任务可触发，部分任务可自动运行 | 不同 cron 任务的模型分层、成本/时延约束、低价值任务限流、失败后再调度策略未完全治理 | 会持续消耗额度、放大长尾失败，但不一定立即阻断主链路 |
| Failover | p0 | 可能已有简单重试或兜底 | 主模型失败时的备选模型、工具失败时的降级路径、超时切换条件、结果质量下限未被硬编码 | 一旦故障发生，系统容易从“可用”直接跌到“不可用” |
| 业务级 E2E | p0 | 单组件、单工具、局部场景可能已验证 | 缺少围绕真实业务目标的全链路验收：触发→理解→执行→反馈→结果校验 | 会出现“组件都通，但业务不成”的假阳性质量结论 |
| 静默策略 | partial | 已意识到减少打扰的重要性 | 缺少按任务风险、失败等级、用户在线态、重试阶段分层的静默/提醒策略 | 过度静默会吞错，过度提醒会打断体验和效率 |
| 稳定性 | p0 | 基础链路可能已能连续运行 | 超时、并发竞争、状态不一致、重试风暴、长尾抖动、恢复后的状态校准尚未充分收口 | 会持续拉低完成率，并制造难以复现的问题 |

## 重点结论（硬）

1. **当前 P0 不是点状能力缺失，而是执行链 + failover + 业务级 E2E + 稳定性四项未完全收口。** 这四项共同决定 Agent 是否“真的能跑、持续能跑、故障时还能跑”。
2. **意图与事件不是当前最硬的阻断，但也不能视为 done。** 它们更像上游质量放大器：一旦定义不硬，会持续向下游注入噪声。
3. **Cron 模型治理当前应定为 P1，而不是 P0。** 它会明显侵蚀成本、时延和长尾质量，但相较主执行链故障，对即时可用性的破坏级别略低。
4. **静默策略当前只能算 partial。** 如果没有和失败等级、用户感知、重试阶段绑定，它不是优化项，而是隐藏故障的放大器。
5. **稳定性必须按 P0 看。** 没有稳定性，任何单次 demo 成功都不能证明 Agent 达到 Day2 可运营质量。

## 最终评分板

| 状态 | 项目 |
|---|---|
| done | 暂无可硬性判定为已完全收口项 |
| partial | 意图、事件、静默策略 |
| p0 | 执行链、Failover、业务级 E2E、稳定性 |
| p1 | Cron 模型治理 |
| observe | 暂无；当前各项都已进入需要明确结论的阶段 |

## 一句话总结

**Day2 视角下，Agent 运行效率与质量的核心结论很硬：主风险不在“有没有能力”，而在“执行链能否稳定闭环、故障能否有质量地下切、以及业务结果是否被 E2E 证明”。当前真正的 P0 是执行链、failover、业务级 E2E、稳定性。**
