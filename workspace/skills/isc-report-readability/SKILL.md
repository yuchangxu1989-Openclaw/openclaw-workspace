# ISC-REPORT-READABILITY-001

## 适用范围

当任务产出属于以下任一类型时，默认启用本技能：

- 重要报告
- 分析报告
- 决策材料
- 方案汇报
- 项目复盘
- 阶段总结

## 核心约束

### 1. 更适合中文阅读

- 先结论，后展开
- 少用翻译腔和过度欧化表达
- 一段尽量只承载一个中心意思
- 术语优先中文，英文只作补充

### 2. 多讲思路

报告不能只有答案，还要有：

- 判断路径
- 分析框架
- 关键假设
- 取舍依据
- 为什么不是别的结论

### 3. 少提代码

- 非技术实现型报告，正文不以代码为主
- 命令、接口、配置、脚本细节尽量降级
- 超过 5 行的代码块默认移到附录或不展示
- 能用自然语言解释，就不用代码证明

### 4. 结构清晰

默认结构：

1. 一句话结论 / 执行摘要
2. 背景与目标
3. 分析思路
4. 核心发现
5. 建议 / 决策项
6. 风险与待确认
7. 下一步

### 5. 主次分明

- 先写最重要的结论和动作
- 核心发现按优先级排序，不平铺罗列
- 重要信息尽量做成短句或要点
- 支撑材料放在结论之后，不反客为主

### 6. 不啰嗦

- 删除重复表达
- 删除空泛套话
- 删除“为了更好地/进一步地/需要指出的是”等低信息密度连接词堆砌
- 能三句说完，不写十句

## 推荐写法模板

```md
# 标题

## 一句话结论
用 1~3 句话说明最重要结论。

## 背景与目标
说明这份报告为什么写、要解决什么问题。

## 分析思路
说明从哪些维度判断、关键假设是什么。

## 核心发现
### 发现 1（最重要）
先写结论，再写依据。

### 发现 2
先写结论，再写依据。

## 建议 / 决策项
给出可执行动作，而不是只给观察。

## 风险与待确认
列出仍不确定的部分。

## 下一步
说明接下来做什么、谁来做、先后顺序是什么。
```

## 交付前自检

- 中文读者是否能快速抓住主线？
- 是否把思路讲清，而不只是把结果摆出来？
- 正文是否被代码、命令、接口细节抢走重心？
- 标题与段落是否形成清楚的层级？
- 还有哪些句子删除后不影响信息量？

## 反例信号

出现以下任一情况，说明违反本钢印：

- 大段技术细节压过业务结论
- 一上来铺背景，读到后面才知道结论
- 只有结论，没有判断依据与推理链
- 章节很多，但看不出轻重缓急
- 用很长的话重复同一个意思
