# EvoMap发布器 - 架构纠正说明

## 纠正背景

### 旧设计的问题
- **过度复杂**：9状态流水线（INITIAL→DEVELOP→TEST→REVIEW→RELEASE→SYNC→ONLINE→FAILED→ARCHIVED）
- **职责重叠**：包含了开发、测试、审核等本应由SEEF负责的环节
- **文档臃肿**：62KB+的复杂架构文档
- **维护困难**：状态流转复杂，容易出错

### 新设计的原则
- **极简专注**：只做一件事 - 发布
- **职责清晰**：SEEF负责进化，DTO负责调度，EP负责发布
- **状态精简**：3个状态（IDLE→PUBLISHING→PUBLISHED/FAILED）
- **易于维护**：代码简洁，逻辑清晰

## 架构对比

### 状态机对比

| 方面 | 旧设计（复杂版） | 新设计（极简版） |
|:-----|:---------------|:---------------|
| 状态数量 | 9个 | 3个 |
| 状态流转 | 12+条规则 | 2条主路径 |
| 自动流转 | 复杂条件判断 | 线性执行 |
| 异常处理 | 多层级 | 统一重试3次 |

#### 旧状态机（9状态）
```
INITIAL → DEVELOP → TEST → REVIEW → RELEASE → SYNC → ONLINE
             ↑      ↓       ↓                  ↓
             └──────┴───────┴──────────────────┴──► FAILED → ARCHIVED
```

#### 新状态机（3状态）
```
IDLE ──publish()──▶ PUBLISHING ──success──▶ PUBLISHED
                          │
                          └─fail(3x)──▶ FAILED
```

### 职责边界对比

| 功能 | 旧设计 | 新设计 |
|:-----|:-------|:-------|
| 技能开发 | SEP内嵌 | **SEEF** |
| 质量测试 | SEP内嵌 | **SEEF** |
| ISC验证 | SEP内嵌 | **SEEF** |
| 版本管理 | SEP内嵌 | **本地任务编排/SEEF** |
| 任务调度 | SEP部分 | **本地任务编排** |
| 网络发布 | SEP | **EP（本组件）** |

### 输入输出对比

#### 旧设计输入
- 文件系统监听
- Git webhook
- 定时扫描
- 人工触发
- 状态变更事件

#### 新设计输入
- 本地任务编排.publish() 指令
- SEEF验证结果引用

#### 旧设计输出
- EvoMap Gene/Capsule
- 多状态通知
- 审计日志
- 监控指标

#### 新设计输出
- EvoMap Gene/Capsule
- DTO回调（成功/失败）

## 集成关系

### 旧设计
```
┌──────────────────────────────────────────────────────┐
│                    SEP (大杂烩)                       │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐        │
│  │ 开发   │ │ 测试   │ │ 审核   │ │ 发布   │        │
│  └────────┘ └────────┘ └────────┘ └────────┘        │
│         ↑        ↑         ↑         ↑               │
│         └────────┴─────────┴─────────┘               │
│                    SEEF                              │
└──────────────────────────────────────────────────────┘
```

### 新设计
```
┌─────────────────┐
│     SEEF        │ ──▶ 开发、测试、验证
│   (进化工厂)     │
└────────┬────────┘
         │ 验证通过
         ▼
┌─────────────────┐
│      本地任务编排        │ ──▶ 调度、编排
│   (调度中心)     │
└────────┬────────┘
         │ publish()
         ▼
┌─────────────────┐
│      EP         │ ──▶ 网络发布
│  (极简发布器)    │
└────────┬────────┘
         │ WebSocket
         ▼
┌─────────────────┐
│   EvoMap Hub    │
└─────────────────┘
```

## 错误处理对比

### 旧设计
- 每状态独立重试
- 复杂降级策略
- 人工介入点
- 状态回滚机制

### 新设计
- 统一重试3次
- 指数退避（1s, 2s, 4s）
- 失败回调DTO
- 无状态回滚（简化）

---

## 总结

| 维度 | 旧设计 | 新设计 |
|:-----|:-------|:-------|
| 复杂度 | 高 | 低 |
| 职责清晰度 | 模糊 | 明确 |
| 维护成本 | 高 | 低 |
| 可测试性 | 低 | 高 |
| 与SEEF/DTO的耦合 | 紧耦合 | 松耦合 |
| 专注度 | 多功能 | 单一职责 |

**核心改进**：从"大而全"的流水线，转变为"小而精"的发布器。SEEF负责进化，DTO负责调度，EP只负责发布。
