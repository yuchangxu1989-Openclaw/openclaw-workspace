## 1. 架构概述

### 1.1 设计目标

EvoMap 进化流水线是一个自动化的技能进化系统，旨在：

- **自动化评估**: 持续评估技能质量和演化潜力
- **智能决策**: 基于评估结果自动决策优化策略
- **安全执行**: 确保每次进化都是可验证、可回滚的
- **持续集成**: 与 Git/GitHub 无缝集成，实现 DevOps 闭环

### 1.2 架构原则

| 原则 | 说明 |
|------|------|
| 可观察性 | 每个状态转换、每次执行都有完整日志 |
| 幂等性 | 相同输入多次执行结果一致 |
| 可回滚 | 任何失败都能安全回滚到上一稳定状态 |
| 模块化 | 组件间松耦合，可独立升级替换 |

### 2.2 状态详细说明

| 状态 | 标识 | 说明 | 停留时限 |
|------|------|------|----------|
| IDLE | `idle` | 空闲状态，技能已注册但未开始进化 | 无限制 |
| DRAFT | `draft` | 草稿状态，技能有未评估的变更 | 7天 |
| EVALUATE | `evaluate` | 评估中，正在进行质量评估 | 30分钟 |
| REJECTED | `rejected` | 已拒绝，评估未通过 | 无限制 |
| OPTIMIZE_PENDING | `optimize_pending` | 待优化，等待优化执行 | 3天 |
| OPTIMIZING | `optimizing` | 优化中，正在进行自动优化 | 2小时 |
| TEST | `test` | 测试中，正在验证优化结果 | 1小时 |
| RELEASE_PENDING | `release_pending` | 待发布，等待最终发布 | 7天 |
| DEPLOYED | `deployed` | 已发布，技能已发布到生产环境 | 无限制 |
| ARCHIVED | `archived` | 已归档，技能已退役 | 永久 |
| ERROR | `error` | 错误状态，需要人工干预 | 无限制 |

### 2.3 状态转换触发器

| 触发器 | 源状态 | 目标状态 | 执行者 |
|--------|--------|----------|--------|
| `skill.draft.created` | IDLE | DRAFT | 系统 |
| `skill.draft.submitted` | DRAFT | EVALUATE | 系统/人工 |
| `skill.evaluate.passed` | EVALUATE | OPTIMIZE_PENDING | 评估引擎 |
| `skill.evaluate.failed` | EVALUATE | REJECTED | 评估引擎 |
| `skill.optimize.started` | OPTIMIZE_PENDING | OPTIMIZING | 决策引擎 |
| `skill.optimize.completed` | OPTIMIZING | TEST | 执行引擎 |
| `skill.optimize.failed` | OPTIMIZING | OPTIMIZE_PENDING | 执行引擎 |
| `skill.test.passed` | TEST | RELEASE_PENDING | 测试引擎 |
| `skill.test.failed` | TEST | OPTIMIZE_PENDING | 测试引擎 |
| `skill.release.approved` | RELEASE_PENDING | DEPLOYED | 发布引擎 |
| `skill.release.rejected` | RELEASE_PENDING | REJECTED | 人工 |
| `skill.deploy.archived` | DEPLOYED | ARCHIVED | 系统/人工 |
| `skill.draft.cancelled` | DRAFT | IDLE | 人工 |
| `skill.error.occurred` | *任意* | ERROR | 系统 |
| `skill.error.resolved` | ERROR | 上一状态 | 人工 |
| `skill.rollback.requested` | *任意* | 指定状态 | 系统/人工 |

### 3.2 输入事件格式

```typescript
// 基础事件接口
interface EvolutionEvent {
  eventId: string;           // 事件唯一ID: evo_{timestamp}_{uuid}
  eventType: EventType;      // 事件类型枚举
  timestamp: number;         // Unix时间戳(ms)
  source: EventSource;       // 事件来源
  
  payload: {
    skillId: string;         // 技能唯一标识
    skillPath: string;       // 技能路径
    version: string;         // 当前版本
    changes: ChangeSet[];    // 变更集合
    metadata: SkillMetadata; // 元数据
  };
  
  context: {
    gitCommit?: string;      // Git提交哈希
    gitBranch?: string;      // Git分支
    triggeredBy: string;     // 触发者
    correlationId: string;   // 关联ID(链路追踪)
  };
}

// 事件类型枚举
enum EventType {
  GIT_PUSH = 'git.push',
  GIT_PR = 'git.pull_request',
  GIT_TAG = 'git.tag',
  MANUAL_TRIGGER = 'manual.trigger',
  SCHEDULED_SCAN = 'scheduled.scan',
  DEPENDENCY_UPDATE = 'dependency.update',
  QUALITY_ALERT = 'quality.alert'
}

// 事件来源
enum EventSource {
  GITHUB = 'github',
  GITLAB = 'gitlab',
  CLI = 'cli',
  API = 'api',
  SCHEDULER = 'scheduler',
  WEBHOOK = 'webhook'
}
```

### 4.2 技能评估器 (SkillEvaluator)

详见 `interfaces.json` 中的接口定义。

**职责:**
- 对技能进行全面的静态分析
- 计算多维度的质量评分
- 识别优化机会
- 生成结构化的评估报告

**评估维度:**

| 维度 | 权重 | 评估内容 |
|------|------|----------|
| 代码规范 | 20% | ESLint/Prettier/Black合规性 |
| 代码复杂度 | 15% | 圈复杂度、认知复杂度 |
| 可维护性 | 15% | 注释覆盖率、文档完整性 |
| 安全性 | 20% | 漏洞扫描、敏感信息检测 |
| 性能 | 15% | 执行效率、资源占用 |
| 兼容性 | 15% | 依赖版本、API兼容性 |

**评分标准:**
- 90-100: 优秀 (A)
- 80-89: 良好 (B) - 可通过，建议优化
- 70-79: 一般 (C) - 需优化后通过
- 60-69: 较差 (D) - 强烈建议优化
- <60: 不合格 (F) - 拒绝通过

### 4.3 决策引擎 (DecisionEngine)

**职责:**
- 解析评估报告
- 应用决策规则
- 生成执行计划
- 风险评估与分级

**决策规则示例:**

```yaml
rules:
  - name: "auto_pass"
    condition: "score >= 90 AND critical_issues == 0"
    action: "SKIP_OPTIMIZE"
    
  - name: "auto_optimize"
    condition: "score >= 70 AND score < 90 AND auto_fixable_issues > 0"
    action: "EXECUTE_OPTIMIZE"
    
  - name: "manual_review"
    condition: "score < 70 OR critical_issues > 0"
    action: "REQUEST_REVIEW"
```

### 4.4 执行器 (Executor)

**职责:**
- 执行优化操作
- 管理执行环境
- 错误处理与回滚
- 生成执行报告

**执行策略:**
- **安全优先**: 每个操作前创建检查点
- **原子性**: 操作要么全部成功，要么全部回滚
- **并行化**: 无依赖的操作可并行执行
- **超时控制**: 每个操作有超时限制

## 5. 错误处理与回滚

### 5.1 错误分类

| 错误级别 | 代码前缀 | 处理策略 |
|----------|----------|----------|
| 警告 (Warning) | WARN | 记录日志，继续执行 |
| 错误 (Error) | ERR | 当前操作失败，尝试重试 |
| 严重错误 (Fatal) | FATAL | 立即停止，执行回滚 |

### 5.2 回滚机制

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           回滚机制流程                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 检查点创建 (Checkpoint)                                                │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ 每个阶段开始前创建检查点:                                               │  │
│  │ • Git: 创建临时分支 git branch checkpoint/{evolution_id}             │  │
│  │ • DB: 记录当前状态快照                                                 │  │
│  │ • FS: 复制关键文件到 .evo/backup/{timestamp}/                          │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  2. 错误检测 (Error Detection)                                             │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ 监控点:                                                                 │  │
│  │ • 每个操作返回码检查                                                    │  │
│  │ • 超时检测 (Operation Timeout)                                         │  │
│  │ • 异常捕获 (try-catch)                                                 │  │
│  │ • 验证失败 (Validation Failure)                                        │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  3. 回滚执行 (Rollback Execution)                                          │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ 回滚步骤:                                                               │  │
│  │ 1. 停止当前执行流程                                                     │  │
│  │ 2. 恢复Git状态: git reset --hard checkpoint/{evolution_id}             │  │
│  │ 3. 恢复DB状态: 从事务日志回滚                                           │  │
│  │ 4. 恢复文件系统: 从备份目录复制                                          │  │
│  │ 5. 清理临时资源                                                          │  │
│  │ 6. 更新状态为 ROLLED_BACK                                               │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  4. 通知与报告 (Notification)                                              │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ • 发送回滚通知 (Email/Webhook/Slack)                                   │  │
│  │ • 生成回滚报告                                                          │  │
│  │ • 更新技能状态为 ERROR                                                  │  │
│  │ • 创建人工干预工单 (可选)                                                │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.3 状态恢复

当系统崩溃或重启后，从持久化存储恢复状态：

```typescript
async function recoverState(): Promise<void> {
  // 1. 读取未完成的事务
  const pendingEvolutions = await db.query(
    "SELECT * FROM evolutions WHERE status IN ('running', 'evaluating', 'optimizing')"
  );
  
  // 2. 根据最后状态决定恢复策略
  for (const evo of pendingEvolutions) {
    const lastCheckpoint = await db.getCheckpoint(evo.id);
    
    if (lastCheckpoint) {
      // 恢复到检查点
      await rollbackToCheckpoint(evo.id, lastCheckpoint.id);
    } else {
      // 无检查点，标记为失败
      await markFailed(evo.id, 'SYSTEM_CRASH');
    }
  }
}
```
