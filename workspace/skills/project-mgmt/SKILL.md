# 项目管理中枢

distribution: internal

## 职责

将模糊目标转化为可执行、可验证、可交付的行动方案。

## 核心流程

### 1. 意图澄清（任何任务开始前）

四项必须明确：
- **业务目标** — 要达成什么结果
- **时效约束** — 为什么现在做
- **成本边界** — 资源、风险、优先级
- **验收标准** — 做到什么算完成

未定义项必须在执行前补齐，不以模糊目标开工。

### 2. 任务分解

拆分为最小可执行单元（MEU），每个单元：
- 明确输入与输出
- 负责角色
- 截止时间
- 验收条件

禁止"先研究一下"类不可验证指令 → 改写为有时限和产出的执行项。

### 3. Sprint规划

```
Sprint = N个Day
Day = 明确的交付物 + 验收标准
Day完成 → 凌霄阁裁决 → 解决遗留 → 下一Day
```

**规则**：
- 能并行就并行，能spawn就spawn
- AEO功能质量测试 + Agent数据效果评测是重点
- 每个Day研发完成后必须经凌霄阁裁决

### 4. 架构评审流水线

```
架构师出方案 → 工程师+质量分析师并行复审 → 不通过自动打回 → 通过后凌霄阁终审 → 用户最终裁决
```

- 不等用户说就自动派复审
- 不等用户踢就自动打回
- thinking=high 默认开
- 半成品不进凌霄阁

### 5. 凌霄阁-7人裁决神殿

7席认知维度：
| 席位 | 维度 | 职责 |
|------|------|------|
| 道 | 第一性原理 | 追问本质 |
| 战 | 战略取舍 | 做什么不做什么 |
| 工 | 工程实现 | 可落地性 |
| 盾 | 风险安全 | 防御视角 |
| 眼 | 用户市场 | 需求洞察 |
| 远 | 未来进化 | 扩展性 |
| 衡 | 综合仲裁 | 最终建议 |

**用户是最终裁决者，凌霄阁只给建议。**

适用场景：重大架构决策、拿不定主意的设计、影响系统进化方向的决策。

### 6. 并行调度

```javascript
// 情报先行
sessions_spawn({ task: "验证技术可行性A方案，2小时内返回POC" })
sessions_spawn({ task: "调研竞品实现路径，输出对比矩阵" })

// 主攻跟进
sessions_spawn({ task: "基于验证方案实现核心模块" })
```

每个任务指令4要素：**动作、产出、时限、验收**。

### 7. 结果整合

- 统一各角色产出，处理冲突与重复
- 保留单一主线结论
- 对外只有三类状态：**已完成 / 阻塞中 / 待决策**

## 交付军规

- 先执行后汇报
- 三轮迭代制：V1骨架 → 用户拍砖 → V2血肉 → 极限挑战 → V3终版
- 可直接执行的代码或即插即用Prompt，禁止纯理论空谈

## 角色池（按需动态编组）

- 系统架构师、开发工程师、质量仲裁官
- 洞察分析师、创作大师、情报专家
- 可按任务规模动态扩缩、并行调度、角色互换

## 检查点

每次启动项目管理流程时自检：
1. 四项意图是否明确？
2. MEU拆分是否到位？
3. 并行机会是否识别？
4. 验收标准是否可验证？
5. 凌霄阁是否在流程中？
