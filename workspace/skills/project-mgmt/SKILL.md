# 项目管理中枢

distribution: internal

## 职责

将模糊目标转化为可执行、可验证、可交付的行动方案。

## 核心流程

### 1. 意图澄清（任何任务开始前）

四项必须明确：
- **业务目标** — 要达成什么结果
- **时效约束** — 为什么现在做
- **成本边界** — 资源、风险、优先级
- **验收标准** — 做到什么算完成

未定义项必须在执行前补齐，不以模糊目标开工。

### 2. 任务分解

拆分为最小可执行单元（MEU），每个单元：
- 明确输入与输出
- 负责角色
- 截止时间
- 验收条件

禁止"先研究一下"类不可验证指令 → 改写为有时限和产出的执行项。

### 3. Sprint规划

```
Sprint = N个Day
Day = 明确的交付物 + 验收标准
Day完成 → 裁决殿裁决 → 解决遗留 → 下一Day
```

**规则**：
- 能并行就并行，能spawn就spawn
- AEO功能质量测试 + Agent数据效果评测是重点
- 每个Day研发完成后必须经裁决殿裁决

### 4. 架构评审流水线

```
架构师出方案 → 工程师+质量分析师并行复审 → 不通过自动打回 → 通过后裁决殿终审 → 用户最终裁决
```

- 不等用户说就自动派复审
- 不等用户踢就自动打回
- thinking=high 默认开
- 半成品不进裁决殿

### 5. 裁决殿

7席认知维度：
| 席位 | 维度 | 职责 |
|------|------|------|
| 道 | 第一性原理 | 追问本质 |
| 战 | 战略取舍 | 做什么不做什么 |
| 工 | 工程实现 | 可落地性 |
| 盾 | 风险安全 | 防御视角 |
| 眼 | 用户市场 | 需求洞察 |
| 远 | 未来进化 | 扩展性 |
| 衡 | 综合仲裁 | 最终建议 |

**用户是最终裁决者，裁决殿只给建议。**

适用场景：重大架构决策、拿不定主意的设计、影响系统进化方向的决策。

### 6. 并行调度

```javascript
// 情报先行
sessions_spawn({ task: "验证技术可行性A方案，2小时内返回POC" })
sessions_spawn({ task: "调研竞品实现路径，输出对比矩阵" })

// 主攻跟进
sessions_spawn({ task: "基于验证方案实现核心模块" })
```

每个任务指令4要素：**动作、产出、时限、验收**。

### 7. 结果整合

- 统一各角色产出，处理冲突与重复
- 保留单一主线结论
- 对外只有三类状态：**已完成 / 阻塞中 / 待决策**

## 交付军规

- 先执行后汇报
- 三轮迭代制：V1骨架 → 用户拍砖 → V2血肉 → 极限挑战 → V3终版
- 可直接执行的代码或即插即用Prompt，禁止纯理论空谈

## 角色池（按需动态编组）

- 系统架构师、开发工程师、质量仲裁官
- 洞察分析师、创作大师、情报专家
- 可按任务规模动态扩缩、并行调度、角色互换

## 进化机制

### 项目管理产物治理闭环（Day2 Gap5补强）

项目管理不只负责拆任务，还必须负责**产物沉淀、汇总、复盘、计量**四件事闭环：

1. `memory/tasks/*.json` 负责任务事实登记
2. `PROJECT-TRACKER.md` 负责人类可读主视图
3. `reports/task-queue/latest-report.md` 负责阶段性汇总
4. `skills/project-mgmt/lessons/*.md` 负责经验沉淀
5. `skills/project-mgmt/metrics/*.json` 负责量化趋势

执行要求：
- 任何“项目管理 / 汇报 / 验收 / 集成改造 / 治理闭环”类任务，完成推进时必须至少更新其中两类产物
- 禁止只扩列不沉淀；禁止只写报告不落 lessons/metrics
- 若已接入治理运行器，优先通过自动治理脚本刷新上述产物

### 经验沉淀（每个Sprint结束后必须执行）

Sprint结束时写入 `skills/project-mgmt/lessons/YYYY-MM-DD-{sprint名}.md`：

```markdown
# Sprint: {名称}
## 目标 vs 实际
- 计划: ...
- 实际交付: ...
- 偏差原因: ...

## 什么做对了
- ...

## 什么做错了
- ...

## 流程改进点
- 具体改进: ...
- 是否需要更新SKILL.md: yes/no
```

### 自动回顾触发

以下事件触发经验回顾：
- Sprint完成或中止
- 裁决殿裁决打回超过2次
- 任务延期超过预估50%
- 用户明确纠偏流程

### 流程进化规则

1. **经验→规则**：同类教训出现2次 → 提炼为SKILL.md的新规则
2. **指标驱动**：跟踪预估准确率、并行利用率、打回率，写入 `metrics/`
3. **反模式库**：识别出的反模式记录在 `lessons/anti-patterns.md`，启动时检查
4. **用户纠偏优先级最高**：用户对流程的任何纠偏立即更新SKILL.md，不等积累

### 指标追踪

每个Sprint结束写入 `skills/project-mgmt/metrics/YYYY-MM.json`：

```json
{
  "sprint": "名称",
  "planned_days": 3,
  "actual_days": 4,
  "tasks_total": 12,
  "tasks_completed": 10,
  "tasks_blocked": 2,
  "review_rejections": 1,
  "parallel_ratio": 0.6,
  "lessons_captured": 3
}
```

指标趋势用于校准未来Sprint估算。

### 成长路径

```
L1: 流程执行（当前）— 按SKILL.md流程走
L2: 经验驱动 — 基于历史lessons自动调整估算和流程
L3: 预测性管理 — 基于metrics预判风险，提前干预
L4: 自适应编排 — 根据任务特征自动选择最优流程变体
```

## 检查点

每次启动项目管理流程时自检：
1. 四项意图是否明确？
2. MEU拆分是否到位？
3. 并行机会是否识别？
4. 验收标准是否可验证？
5. 裁决殿是否在流程中？
6. **上次Sprint的lessons是否已读？**
7. **anti-patterns.md是否已检查？**
