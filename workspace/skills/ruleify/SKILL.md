# ruleify — 规则化 = 程序化闭环

**核心原则：没有代码的规则 = 不存在。**

规则化的交付物不是JSON声明文件，而是一条可自主运行的自动化链条：
系统能自己感知→自己判断→自己执行→自己验证，无需人类记忆或手动触发。

## 展开流程

### 1. 意图澄清（30秒内完成）
- 规则化什么？（一句话描述）
- 触发点在哪？（什么状态变化是信号）
- 做到什么算成功？（验收标准）

### 2. 三层交付（每层必须有可执行代码）

#### 感知层 — 探针代码
谁在监听？用什么机制捕获信号？
- git hook（pre-commit / post-commit）
- 文件系统watcher
- cron定时扫描
- 事件总线监听
**产出：可运行的探针脚本，放入对应hook或cron**

#### 认知层 — 判断逻辑
捕获信号后如何判断？
- 条件检查函数
- 规则匹配逻辑
- LLM辅助判断（复杂场景）
**产出：判断函数/脚本，输入=信号，输出=决策（通过/阻断/告警）**

#### 执行层 — 动作代码
判断结果出来后做什么？
- 自动修复（移动文件、修改配置）
- 阻断操作（拒绝提交、拒绝部署）
- 通知告警（飞书/日志）
**产出：执行脚本，被判断逻辑调用**

### 3. 注册与集成
- ISC规则JSON → `skills/isc-core/rules/`（元数据声明，不是核心交付物）
- 探针代码 → 注册到对应触发机制（hook/cron/watcher）
- 验证：手动触发一次，确认全链条跑通

### 4. 自检清单
- [ ] 感知层有代码？能自动触发？
- [ ] 认知层有代码？能自动判断？
- [ ] 执行层有代码？能自动执行？
- [ ] 三层串联？信号→判断→动作全自动？
- [ ] 无需人类记忆或手动干预？

## 反模式
- ❌ 只写JSON规则不写代码（= 黄灯规则，等于不存在）
- ❌ 代码写了但没注册到触发机制（= 死代码）
- ❌ 依赖Agent"记住"去执行（= 必定遗忘）
