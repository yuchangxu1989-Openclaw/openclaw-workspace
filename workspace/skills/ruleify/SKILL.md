# ruleify — 规则化引擎

当用户说"规则化"时，对目标概念执行以下展开：

## 展开流程（必须完整执行）

### 1. 意图澄清
- 这个规则要解决什么问题？
- 触发场景是什么？（什么事件/条件发生时应该启动）
- 执行结果是什么？（做到什么算完成）

### 2. 三层分解（感知→认知→执行）

**感知层（事件定义）**
- 事件名称：`<domain>.<object>.<action>`
- 事件源：谁/什么探针在监听？
- 触发条件：什么状态变化触发此事件？

**认知层（ISC规则）**
- 规则文件：`rule.<domain>-<name>-NNN.json`
- 匹配条件：什么事件+什么上下文→触发此规则
- 决策逻辑：匹配后判断什么、做什么决策

**执行层（DTO任务/技能调用）**
- 执行动作：具体调用什么技能/脚本
- 验收标准：怎么判断执行成功
- 失败处理：失败后怎么办

### 3. 产出物清单

每次规则化必须产出：
1. ISC规则JSON文件 → `skills/isc-core/rules/`
2. 事件定义（写入规则的trigger.events）
3. DTO任务定义（如需定时/调度）→ `skills/dto-core/tasks/`
4. 更新CAPABILITY-ANCHOR.md（如涉及新技能）

### 4. 自检

- [ ] 规则是否MECE？（与现有规则不重叠）
- [ ] 三层是否都有明确归属？（感知谁做、认知谁做、执行谁做）
- [ ] 是否可泛化？（解决一类问题而非一个问题）
- [ ] 10倍规模下是否成立？

## 反模式

- ❌ 只写规则不定义事件（无法触发）
- ❌ 只定义事件不写执行（无法闭环）
- ❌ 规则与现有规则语义重叠（熵增）
- ❌ 硬编码具体值（不可泛化）
