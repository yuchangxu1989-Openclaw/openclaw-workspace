## 🔴 关键教训：OpenClaw apiKey vs apiKeys（重大Bug修复）

**时间**: 2026-03-01 晚间  
**严重程度**: P0 — 导致所有子Agent全部失败

### 根因分析
1. 用户提供了7个 penguinsaichat API keys
2. 我将 provider config 中的 `apiKey`(string) 改为 `apiKeys`(array of 7)
3. **OpenClaw config schema 只支持 `apiKey`（单个 string）**，不存在 `apiKeys` 字段
4. Gateway 静默忽略了未知字段 `apiKeys`，provider 实际无可用 key
5. 所有子Agent spawn后拿不到 API key → 全部空壳（empty history）

### 修复操作
- 恢复 `apiKey` 为单个 string（第一个key: `sk-103s...`）
- 删除无效的 `apiKeys` 数组
- `kill -SIGUSR1` 热重载 Gateway config

### 铁律（刻入记忆）
```
OpenClaw provider config 字段:
✅ apiKey: string (单个)
✅ baseUrl: string
✅ auth: object
✅ api: string  
✅ headers: object
✅ authHeader: string
✅ models: object
❌ apiKeys: 不存在！Gateway不认！

多key轮换方案:
- 方案A: 注册多个provider (penguinsaichat-1, penguinsaichat-2, ...)
- 方案B: auth.profiles + auth.order 机制
- 方案C: 外部代理/负载均衡
- ❌ 禁止: 直接写 apiKeys 数组
```

### 影响范围
- 3个正在运行的子Agent（全量审计、DTO-RCA、文档自动校验）全部失败
- 1个待spawn的子Agent（auto-vectorization）未启动
- 1个待重跑的子Agent（全量端到端验证）未启动
- **需要全部重新spawn**

### 防重犯检查点
每次修改 Gateway provider config 前，必须：
1. `gateway(action="config.schema")` 确认字段合法性
2. 只使用 schema 中明确定义的字段
3. 修改后验证 config 加载无报错
