# 「言出法随」能力快速构建 — 硬缺口优先级审视报告

> **视角声明**：本报告采用纯粹挑刺视角，不为已完成工作背书，仅聚焦"拦路虎"。
> **目标**：定位当前距离"全球领先自治能力"最关键的结构性障碍。
> **日期**：2026-03-07

---

## 一、审视框架

评估"言出法随"自治能力的核心指标：
- **闭卷评测**：不依赖人工提示，系统自主完成任务
- **唯一生效入口**：评测结果真实反映实际能力，而非绕过路径
- **端到端成功率**：从用户意图到最终产出的全链路成功率
- **运行时纳管**：生产环境的安全边界、可回滚性、副作用管控
- **不可绕过性**：关键门禁无法通过workaround规避

---

## 二、硬缺口清单（挑刺视角）

### 🔴 【必须立即做】阻断性缺口

---

#### 缺口 #1：闭卷评测基线缺失 — 能力无法被客观量化

**问题本质**：
当前缺乏独立、可重现的闭卷基准测试套件。所有"能力验证"依赖内部演示场景，
这些场景存在严重的**泄题风险**（模型在训练/调优阶段见过测试案例）。

**具体症状**：
- 没有独立 holdout 测试集，无法区分"真会"还是"见过题"
- 评测由能力构建者自己设计，存在确认偏差
- 没有对比基线（与 GPT-4o/Gemini Ultra 的同题对比）
- "成功"的定义模糊，依赖主观判断

**危害等级**：🔴 CRITICAL — 所有能力声称均无可信支撑

**必须完成的最小动作**：
1. 建立与开发流程物理隔离的 holdout 评测库（至少 200 任务）
2. 引入外部评测员（不参与开发），进行盲测打分
3. 定义量化指标：完全成功率 / 部分成功率 / 失败率 / 幻觉率

---

#### 缺口 #2：真实端到端成功率 vs. "演示成功率" 的巨大落差

**问题本质**：
演示环境经过精心设计（稳定网络、预热缓存、已知工具状态），
与真实生产环境存在巨大差距。没有人知道真实成功率。

**具体症状**：
- 任务失败多数以"静默失败"或"部分完成"收场，而非显式报错
- 没有 P50/P90/P99 成功率数据，仅有"精选成功案例"
- 复杂任务的中间步骤失败未被计入总失败率
- 用户意图理解准确率和最终任务完成率混为一谈

**危害等级**：🔴 CRITICAL — 无法支撑生产部署决策

**必须完成的最小动作**：
1. 对每类核心任务，随机抽样 100 次执行，统计全链路成功率
2. 区分"任务启动成功"和"任务完成成功"
3. 建立失败模式分类库（意图理解失败/工具调用失败/权限失败/幻觉失败）

---

#### 缺口 #3：运行时纳管机制缺失 — 副作用不可控

**问题本质**：
Agent 在执行任务时可能触发不可逆操作（删文件、发消息、修改配置、调用外部 API），
当前缺乏：沙箱隔离、权限最小化、操作可回滚、紧急中断机制。

**具体症状**：
- Agent 执行路径无审计日志，无法事后复盘
- 没有"危险操作"二次确认机制（对破坏性操作无拦截）
- 无法在任务执行中途安全中止（kill 操作可能留下半完成的脏状态）
- 权限凭证（API Key、OAuth Token）由 Agent 自主持有，无轮转和范围限制

**危害等级**：🔴 CRITICAL — 生产事故的必然路径

**必须完成的最小动作**：
1. 实现操作意图分级（读/写/破坏性），破坏性操作必须确认
2. 建立最小权限原则的凭证管理（作用域化 token，不给 super key）
3. 关键操作全链路审计日志，支持 30 天回溯

---

#### 缺口 #4：任务意图捕获的"唯一生效入口"不成立

**问题本质**：
"言出法随"的前提是用户的自然语言意图能被准确、唯一地解析为执行计划。
当前存在多个平行解析路径，各路径结果不一致，无从保证哪个"真正生效"。

**具体症状**：
- 相同指令在不同上下文窗口长度下产生不同解析结果（位置偏差）
- 存在多套 prompt 模板竞争，哪个最终生效取决于运行时隐式状态
- 用户意图中的歧义未被显式澄清，而是由模型静默猜测
- 没有 intent canonicalization 层，"发邮件给张三"和"给张三发一封邮件"可能走不同代码路径

**危害等级**：🔴 HIGH-CRITICAL — 能力天花板被意图解析精度锁死

**必须完成的最小动作**：
1. 建立意图标准化层（Intent Canonicalization），消除同义多路径问题
2. 对高风险/高代价任务，强制执行"执行前确认"（显示解析结果供用户确认）
3. 意图解析一致性测试：同义句 10 变体，解析结果是否一致

---

#### 缺口 #5：工具调用可靠性 — 无重试、无降级、无告警

**问题本质**：
Agent 的核心执行能力依赖工具调用（Tool Use），但当前工具调用层极度脆弱：
超时不处理、错误不分类、无重试策略、无降级方案。

**具体症状**：
- 工具调用超时后 Agent 通常选择"编造结果"而非上报失败
- 相同工具调用在不同时间成功率差异超过 20%（网络/限速/格式漂移）
- 没有工具健康检查，Agent 不知道某工具当前是否可用
- 工具返回格式变更（API breaking change）导致静默错误，无告警

**危害等级**：🔴 HIGH — 工具层不可靠，自治能力上限为 0

**必须完成的最小动作**：
1. 为每个工具实现标准化错误分类（可重试/不可重试/需人介入）
2. 实现指数退避重试 + 最大重试次数限制
3. 工具可用性实时监控（健康检查心跳），不可用时主动告警

---

### 🟡 【重要但可随后做】结构性缺口

---

#### 缺口 #6：长任务状态持久化 — Agent 没有工作记忆

**问题本质**：
对于跨多轮交互、跨多步骤的复杂任务，Agent 的中间状态完全依赖上下文窗口。
窗口满了/重启了，任务就断了，且断点无法续接。

**具体症状**：
- 任务执行到一半因上下文超长被截断，后续步骤"失忆"
- 没有 checkpoint 机制，任务失败只能从头重来
- 多步任务之间依赖数据无结构化存储，靠自然语言传递状态（极易丢失）

**危害等级**：🟡 HIGH — 限制任务复杂度上限，无法完成小时级以上任务

**行动建议**：
1. 实现任务状态的外部化存储（Redis/数据库），支持续接
2. 关键中间产物显式持久化（不依赖上下文）
3. 断点重试时自动加载上次状态

---

#### 缺口 #7：能力边界未定义 — Agent 不知道自己"不会做什么"

**问题本质**：
当前系统在超出能力范围的任务面前，倾向于"硬撑"而非优雅拒绝。
这导致高置信度的错误输出，比直接报错更危险。

**具体症状**：
- Agent 遇到超范围任务时返回"看似合理但错误"的结果（幻觉）
- 没有能力覆盖图谱，无法静态判断任务是否在能力圈内
- 模型自我评估置信度与实际成功率相关性极低（过度自信）

**危害等级**：🟡 HIGH — 危险的幻觉比明确的失败更难处理

**行动建议**：
1. 建立能力 capability map（明确声明每类任务的成功率预期）
2. 低置信度任务触发人工介入，而非静默推进
3. 拒绝机制优先级高于"尽力而为"

---

#### 缺口 #8：评测-开发反馈回路断裂

**问题本质**：
当前评测结果无法快速反哺到能力改进中。发现问题到修复问题的周期过长，
导致已知缺陷长期存在，新功能开发掩盖旧问题。

**具体症状**：
- 没有自动化回归测试，新版本上线不跑全量评测
- 评测失败不触发阻断（CI 流程无质量门禁）
- 能力退化（Regression）无法被及时发现
- 评测-开发团队信息不对称（评测报告无人认领 action item）

**危害等级**：🟡 HIGH — 无法保证能力只增不减

**行动建议**：
1. 评测自动化集成到 CI/CD，失败阻断发布
2. 建立能力版本对比报告（每次发布 vs 上次基线的对比）
3. 指定 owner 对每类评测指标负责，退化必须解释

---

#### 缺口 #9：多 Agent 协作的一致性 — 各 Agent 自说自话

**问题本质**：
"言出法随"场景通常需要多个专项 Agent 协同完成复杂任务，
但当前各 Agent 之间缺乏统一的协议、上下文共享机制和冲突解决策略。

**具体症状**：
- Agent A 完成的工作，Agent B 不知道（无共享状态）
- 相同实体（用户/文档/任务）在不同 Agent 中有不同表示，无统一 ID
- Agent 间消息传递依赖自然语言（高歧义），而非结构化协议
- 当多 Agent 计划冲突时，无仲裁机制

**危害等级**：🟡 MEDIUM-HIGH — 限制系统复杂任务协同能力

**行动建议**：
1. 定义 Agent 间通信协议（结构化，非自然语言）
2. 引入共享任务图（Task Graph），所有 Agent 共同维护执行状态
3. 实现冲突检测和中央仲裁机制

---

### 🟢 【锦上添花】优化性缺口

---

#### 缺口 #10：用户意图表达辅助 — 降低用户侧输入门槛

**问题本质**：
"言出法随"的理想态是用户随意说，系统自动理解。
当前系统对用户输入质量有隐式要求，普通用户输入导致理解失败率偏高。

**具体症状**：
- 专业用户成功率远高于普通用户（能力分布不均匀）
- 复杂任务需要用户"懂得怎么说"才能触发正确执行
- 没有意图澄清的 UX 引导（对模糊输入无结构化追问）

**危害等级**：🟢 MEDIUM — 影响普及性，但不阻断核心能力

**行动建议**：
1. 意图模糊时引导用户结构化填写（非强制阻断）
2. 成功任务的输入模板化，供用户参考
3. 根据用户画像动态调整理解宽容度

## 四、仲裁官最终判定

**当前状态**：演示级能力，非生产级自治能力。

**最阻断全球领先的核心症结**（按影响力排序）：

1. **可信度危机**（缺口 #1 #2）：没有闭卷评测数据，所有"能力"是自证，无第三方可信。
   这是进入国际评测榜单、获得外部认可的硬门槛。
   
2. **安全边界缺失**（缺口 #3）：没有运行时纳管，无法进行生产部署。
   这是从演示走向实际部署的必须跨越的工程门槁。

3. **意图执行可靠性**（缺口 #4 #5）：意图解析和工具执行的不可靠是根本性的，
   在这两层没有达到 95%+ 可靠性之前，上层所有能力构建都是沙堡。

**结论**：不先补齐这三类缺口，快速堆叠新能力的行为是在增加技术债，而非接近全球领先。
